The house rule is: **if it can change while the process runs, it's state, not config.** The one declared exception is a re-readable source you opted into with `freshness: "live"` — and on Kubernetes there is exactly one re-readable source you get for free: a volume-mounted file. This page walks the whole path end to end: ConfigMap/Secret mount, live file tesserae, per-request pinning, pool rebuilds keyed on `epoch()`, what happens when a rotation goes bad — current truth wins, with `keepLastGood` as the per-tessera opt-in — and a `health()` report wired to probes that can't turn one bad push into a crash loop.

## Why env vars cannot do this

The tempting route is the one that cannot work:

```yaml
# deployment.yaml — the env route
env:
  - name: LOG_LEVEL
    valueFrom:
      configMapKeyRef:
        name: app-config
        key: LOG_LEVEL
```

Edit the ConfigMap and nothing happens to a running pod. A process's OS environment is captured once, at `exec` — the kernel gives you no way to change it afterwards. Kubernetes knows this: changing a Deployment's `env` triggers a rollout, i.e. new pods. "Live env vars" on Node are not a missing feature; they are physically impossible.

That's why declaring it is a mistake the library points out:

```ts
const logLevel = tessera({
  key: "LOG_LEVEL",
  schema: z.enum(["debug", "info", "warn", "error"]),
  exposure: "public",
  freshness: "live",
  sources: [{ env: "LOG_LEVEL" }],
  // dev-mode warning: "live" on an env source is a no-op on Node —
  // a running process's environment never changes. Use a file source.
});
```

(The same warning fires on Cloudflare Workers, where `env` is fixed per deployment — `live` is a no-op there too.)

## Mount config as files

ConfigMap and Secret volume mounts *do* update in place: the kubelet writes the new generation into a `..data` directory and swaps a symlink, atomically, for the whole mount at once. Updates typically land within the kubelet sync period (about a minute by default).

```yaml
# deployment.yaml — directory mounts, updated in place by the kubelet
volumes:
  - name: app-config
    configMap:
      name: app-config
  - name: db-creds
    secret:
      secretName: db-creds
containers:
  - name: api
    volumeMounts:
      - name: app-config
        mountPath: /etc/app-config
        readOnly: true
      - name: db-creds
        mountPath: /etc/db-creds
        readOnly: true
```

One trap to know before anything else:

```yaml
# TRAP — a subPath mount is copied ONCE at container start and NEVER updated.
# The ..data symlink swap happens in the volume; a subPath copy is outside it.
volumeMounts:
  - name: app-config
    mountPath: /etc/app-config/LOG_LEVEL
    subPath: LOG_LEVEL   # this file is frozen for the life of the container
```

Mount the directory, not individual keys via `subPath`.

## Declare the live tesserae

```ts
import { z } from "zod";
import { tessera, defineTesserae } from "tessellum";

export const logLevel = tessera({
  key: "LOG_LEVEL",
  schema: z.enum(["debug", "info", "warn", "error"]),
  doc: "Runtime log verbosity; tunable without a rollout",
  exposure: "public",
  sources: [{ file: "/etc/app-config/LOG_LEVEL" }],
  freshness: "live",
});

export const dbHost = tessera({
  key: "DB_HOST",
  schema: z.string().min(1),
  doc: "Postgres host; the platform team rotates it together with DB_PASSWORD",
  exposure: "public",
  sources: [{ file: "/etc/db-creds/DB_HOST" }],
  freshness: "live",
});

export const dbPassword = tessera({
  key: "DB_PASSWORD",
  schema: z.string().min(16),
  doc: "Postgres password; rotated together with DB_HOST",
  // exposure defaults to "secret" — fail closed
  sources: [{ file: "/etc/db-creds/DB_PASSWORD" }],
  freshness: "live",
});

export const dbTesserae = defineTesserae([dbHost, dbPassword]);
```

Descriptors stay inert: declaring `freshness: "live"` performs zero I/O and starts no watcher. Nothing happens until you bind.

## Bind once, poll on read

```ts
import { bind } from "tessellum/node";

export const live = bind([logLevel, dbHost, dbPassword], {
  maxStalenessMs: 2_000, // default is ~1s for file refs
  onInvalid(k, _raw, issues) {
    // Observation plane: fires once per (tessera, distinct invalid raw),
    // not once per read. By the time it fires the tessera is poisoned and
    // reads of it throw; the hook and those errors share an incidentId,
    // so this page and the request failures join in your logs.
    // For secret tesserae, _raw is { rawDigest, length } — never the value.
    alerting.page("config-invalid", { key: k.key, issueCount: issues.length });
  },
});
// Coloring: these entries hold live tesserae and nothing async, so bind's
// result type is LiveBinding — a type view deliberately NOT assignable
// where a Snapshot is expected.
```

Note what the bind call does *not* contain: a source list. The entries are **bare** — each tessera already declares its file ref, and under the `tessellum/node` entry point bare entries resolve against the runtime's natural defaults (`process.env` and the filesystem). The declared paths *are* the mount paths, so there is nothing to rewire. When a deployment does need different wiring, that's a per-entry combinator — `file(dbHost, "/etc/other-mount/DB_HOST")` patches the declared file ref in place — never a global source list. (Under the core `"tessellum"` bind, which supplies no natural defaults, these bare file refs would refuse to bind with `NoWiringError` — that refusal is the browser bundle's structural guarantee, and the reason this page imports `bind` from `tessellum/node`.)

Binding is still eager over the bound subset: if the volume isn't mounted, boot dies right here with one aggregated `MissingConfigError` naming every missing tessera — not three crashes spread over the first hour of traffic.

After that, freshness is **poll-on-read**. `live.unwrap(logLevel)` re-reads the raw file only if the last read is older than `maxStalenessMs`, and re-runs the validator only when the raw bytes actually changed (memoize-by-raw — zod runs once per rotation, not once per request). There are no background loops, timers, or `fs.watch` handles in core, ever: a tessera nobody reads is never re-read, and an idle process does exactly nothing.

## Pin a Snapshot per request

Handlers should not watch config drift mid-request. Pin once at the edge:

```ts
app.use((req, _res, next) => {
  req.cfg = live.snapshot(); // pinned, immutable Snapshot for this request
  next();
});

app.get("/orders", (req, res) => {
  const level = req.cfg.unwrap(logLevel); // same answer for the whole request
  // ...
});
```

Everything downstream of the middleware sees one coherent world. And because cross-package APIs are typed against `Snapshot` and `Scope` — never `LiveBinding` — the compiler stops you from handing a factory something that can shift under it: pass `req.cfg` or `req.cfg.pick(dbTesserae)`, not `live`.

## Rotating pairs without tearing

The platform team rotates `DB_HOST` and `DB_PASSWORD` together, in one update to the `db-creds` Secret. Here's the failure mode with naive independent reads:

```ts
// TORN PAIR — two independent reads can land on opposite sides of a re-read:
const host = live.unwrap(dbHost);          // old host
// ... an await, or just the staleness window elapsing ...
const password = live.unwrap(dbPassword);  // new password
// Result: new password presented to the old host. Auth failures until the
// next unlucky alignment resolves itself.
```

The fix is the pin you already have. `live.snapshot()` performs a **single coherent pass per source** — one read of the `db-creds` mount serves both tesserae:

```ts
const pinned = live.snapshot();          // one pass over /etc/db-creds
const host = pinned.unwrap(dbHost);         // both values come from that pass:
const password = pinned.unwrap(dbPassword); // old+old or new+new, never mixed
```

Kubernetes cooperates: the `..data` symlink swap replaces the whole directory in one step, so a single pass through the mount observes exactly one generation of the pair.

## Rebuild the pool on epoch()

First, the trap. Live freshness cannot help a value that is consumed once:

```ts
// TRAP — the value is read at construction and captured forever.
// freshness:"live" changes nothing about what this Pool dials tomorrow.
const pool = new Pool({
  host: live.unwrap(dbHost),
  password: live.unwrap(dbPassword),
});
```

Live values only help code that re-reads them. Long-lived objects need a rebuild trigger, and that trigger is `epoch()` — a monotonic per-tessera counter that bumps whenever the tessera's observable truth changes: a newly validated value, and equally a transition into (or out of) invalidity. One counter, one clock — validity changes arrive through the same number you key rebuilds on:

```ts
import { Pool } from "pg";

let pool: Pool | undefined;
let poolEpochs = "";

export function getPool(): Pool {
  const epochs = `${live.epoch(dbHost)}:${live.epoch(dbPassword)}`;
  if (!pool || epochs !== poolEpochs) {
    const pinned = live.snapshot(); // coherent host+password pair
    void pool?.end();
    pool = new Pool({
      host: pinned.unwrap(dbHost),
      password: pinned.unwrap(dbPassword),
    });
    poolEpochs = epochs;
  }
  return pool;
}
```

A rewrite of identical bytes doesn't bump the epoch, so the pool never churns on noise. A rotation that fails validation, though, *does* bump it — validity changes ride the same clock — and as written, `pinned.unwrap(dbHost)` would throw mid-rebuild on that bump. That throw is the next section's subject; the hardened rebuild there reads through `safe` and degrades instead. Note what this is *not*: there is no `watch()`, no `subscribe()`, no change callback, by design. You check the epoch on a read path you already execute. If your process has no such path, it also has no live behavior — which is the honest outcome for a core that runs no timers.

## When the file goes bad: current truth wins

After boot, an invalid rotation is not papered over. There are no magically inherited values: if someone ships an empty `DB_HOST`, the tessera becomes **poisoned**. Pinning still works — `live.snapshot()` never throws wholesale — but the poisoned tessera occupies an *error slot* in the snapshot. Every other tessera keeps answering; reading the poisoned one throws:

```ts
const pinned = live.snapshot();  // never throws wholesale
pinned.unwrap(logLevel);            // unaffected — every other tessera keeps working

pinned.unwrap(dbHost);
// InvalidLiveValueError: DB_HOST is invalid after previously holding a
// valid value
//   issues       — the validator's findings
//   lastValidAt  — when the source last produced a value that validated
//   invalidSince — when it stopped
//   incidentId   — the same id the onInvalid hook saw for this raw
//   ref          — read provenance: the ref that supplied the raw
//                  (here: file /etc/db-creds/DB_HOST)
// (the offending raw value is attached only for non-secret tesserae)
```

Blast radius is proportional to actual usage: an invalid tessera nobody reads must never 500 the site, and an invalid tessera on the hot path fails exactly the reads that depend on it. The trade, stated plainly: the throw default converts a bad config push from silent staleness into a loud partial outage that lasts until the push is fixed — and for tesserae where availability beats freshness, you opt into `keepLastGood` per tessera (below).

Poisoning is invalid-*after*-valid. A tessera that has *never* validated in this process is a different failure with its own error:

```ts
// A half-lazy binding presence-checks eagerly but defers validation to
// first read. The file exists — but a botched rotation left garbage in it:
const lazy = bind.lazy([dbHost, dbPassword]);

lazy.unwrap(dbPassword);
// NoValidValueError: DB_PASSWORD has never produced a valid value
```

`NoValidValueError` stays reserved for the lazy and unchecked doors; the eager `bind` this page uses still dies at bind time, so a running handler never sees it.

### safe, and the hardened pool rebuild

`safe` is the non-throwing consumption form — it returns `{ ok: true, value }` or `{ ok: false, error }` — and it is the first thing epoch-consumer code should reach for. The epoch bumps on the invalid transition too, so the rebuild pattern is mechanical: *epoch changed → `safe` → ok? rebuild with the value : degrade with the error.*

```ts
export function getPool(): Pool {
  const epochs = `${live.epoch(dbHost)}:${live.epoch(dbPassword)}`;
  if (!pool || epochs !== poolEpochs) {
    const pinned = live.snapshot(); // coherent host+password pair
    const host = pinned.safe(dbHost);
    const password = pinned.safe(dbPassword);
    if (host.ok && password.ok) {
      void pool?.end();
      pool = new Pool({ host: host.value, password: password.value });
    } else if (pool) {
      // Degrade: the rotation is bad, but the pool built from the last
      // valid pair still dials fine. Keep it — onInvalid already paged.
    } else {
      // No pool yet and no valid pair to build one from.
      throw host.ok ? password.error : host.error;
    }
    poolEpochs = epochs;
  }
  return pool;
}
```

Recovery rides the same clock: when the platform team fixes the Secret, the transition back to valid bumps the epoch again and the next `getPool()` call rebuilds. No callback, no watcher — the same read path you already execute.

### Opting into keep-last-good

Keep-last-good still exists — as a bind-site opt-in, for tesserae where availability beats freshness. Log verbosity is the canonical case: nobody should lose requests because the verbosity file has a typo.

```ts
import { keepLastGood } from "tessellum";
import { bind } from "tessellum/node";

export const live = bind([logLevel, dbHost, dbPassword], {
  maxStalenessMs: 2_000,
  invalid: [keepLastGood(logLevel, { maxCoastMs: 300_000 })],
  onInvalid(k, _raw, issues) {
    /* unchanged from above — it still fires when logLevel goes bad */
  },
});
```

`maxCoastMs` is required — there is no unbounded coasting. For up to five minutes, reads of a poisoned `logLevel` keep returning the last valid value while `health()` reports the tessera as `"coasting"`. Past the budget the tessera throws `CoastBudgetExceededError` and the status hardens to `"invalid-was-valid"`: you bought time to fix the push, not permanence.

Note where the opt-in lives: at the bind site, never on the descriptor. Staleness tolerance is a consumer preference, and descriptors are identity. As an identity-class field it would make two teams' differing staleness preferences a descriptor mismatch for the *same* tessera; as a cosmetic-class field it would hide real behavior from identity. Both are wrong, so it is neither.

### Invalid is not unavailable

An unparseable value and an unreachable source are different failures. When the source goes down — the mount vanishes, reads start erroring — the binding keeps serving the cached value *within* `maxStalenessMs`. That is not inherited state; it is the staleness budget you signed at bind: `bind(entries, { maxStalenessMs: 2_000 })` declares that any read may be up to two seconds stale, outage or not. Beyond the budget, reads throw `SourceUnavailableError` — it carries `cause` and `staleSince` — and the `onSourceError` hook fires. An unavailable source is never routed through `onInvalid`: a down mount and a bad rotation deserve different pages.

### Recovery is one read

Poisoning changes the read cadence. Normally a read inside `maxStalenessMs` is served from memory; a poisoned tessera bypasses that memoization, so the next read re-reads the source immediately. Fix the file and recovery costs one read, not one poll interval.

The bypass also collapses the file-tearing window. On Kubernetes it barely exists to begin with: the `..data` symlink swap is atomic for a single-file, single-pass read, and multi-file sources are read in one resolved-directory pass. The genuine risk is plain files outside Kubernetes, written non-atomically — a half-written file validates as garbage and poisons the tessera, and the poisoned bypass means the very next read sees the completed write.

The full taxonomy, one line each — and every one names the ref that produced it (read provenance; on this page, always a mounted file path):

- `InvalidLiveValueError` — invalid after previously valid; carries `issues`, `lastValidAt`, `invalidSince`, `incidentId` (the raw value only for non-secret tesserae).
- `SourceUnavailableError` — source unreachable past the staleness budget; carries `cause` and `staleSince`.
- `CoastBudgetExceededError` — a `keepLastGood` tessera coasted past its `maxCoastMs`.
- `NoValidValueError` — never valid in this process; lazy/unchecked doors only.

## Wire health() to the right probe

The throw default keeps failure loud at the point of use; `health()` gives the operator the map. It reports one entry per tessera — secret values never appear in the report:

- `status` — `"valid"`, `"invalid-was-valid"`, `"never-valid"`, or `"coasting"`
- `lastValidAt` / `lastInvalidAt` — the source-observation side: what the binding has seen
- `lastThrownAt` / `throwCount` — the consumer-impact side: what requests have felt

The two sides stay distinct because they answer different questions: `"invalid-was-valid"` with a `throwCount` of zero is *invalid but dormant* — a bad value nobody currently reads; the same status with a climbing `throwCount` is *actively failing requests*. That is the operator triage split.

```ts
app.get("/livez", (_req, res) => {
  // Liveness asks "is the process wedged?" — config validity is never
  // part of that answer.
  res.status(200).send("ok");
});

app.get("/readyz", (_req, res) => {
  const report = live.health();
  const neverValid = Object.values(report).some(
    (entry) => entry.status === "never-valid",
  );
  res.status(neverValid ? 503 : 200).json(report);
});
```

```yaml
livenessProbe:
  httpGet:
    path: /livez
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 15
readinessProbe:
  httpGet:
    path: /readyz
    port: 8080
  periodSeconds: 15
```

The probe contract is strict: **liveness probes never consult tessera validity, and readiness reacts to `"never-valid"` at most.** Restarting a pod does not fix a bad ConfigMap — the replacement reads the same bytes, and with eager `bind` it dies at boot — so a liveness probe wired to validity turns one bad push into a fleet-wide crash loop. (With the eager bind on this page, `"never-valid"` can only appear through the lazy doors, which makes the readiness gate cheap insurance.) Route `"invalid-was-valid"` and `"coasting"` to dashboards and alerting, not to the kubelet.

## What live does not give you

Stated plainly, because every guarantee has a boundary:

- **No push, no subscriptions.** Freshness is poll-on-read only. There is no `watch()`, `onChange`, or event stream, and there never will be a background loop in core. For provider-backed values (SSM- or Vault-style impls wired with `provider()` under the async-capable `tessellum/async` bind), `refresh(live)` is the explicit re-pull: it returns a *new* pinned `Snapshot`, all-or-nothing on partial source failure (opt-in `partial: "keep-old"`), single-flight coalesced.
- **No construction-time magic.** Objects that consume a value once (pools, SDK clients) capture it forever; the `epoch()`-keyed rebuild above is the pattern, not an optional nicety.
- **No live env vars.** OS process environments don't change; on Kubernetes that path is a pod restart. `live` plus an env source is a dev-mode warning, not a capability.
- **No live on workerd.** The Workers `env` bag is fixed per deployment; `live` is a no-op there (dev warning).
- **No help against `subPath`.** The library reads what the kubelet writes. A `subPath` mount is frozen at container start, and no amount of polling revives it.
- **Not a feature-flag system.** No targeting, no percentage rollouts, no per-user evaluation — that's a flag service's job, and a declared non-goal here.
