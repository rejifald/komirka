## The shape of the app

Three rules do all the work on a Node server:

1. **Declare zernos next to the code that reads them.** The database module owns the database zernos, the billing module owns Stripe's. There is no central config module.
2. **Bind once, at the composition root.** `main.ts` is the only file that touches wiring. `bind` is eager, so a broken environment kills the process before it can accept a single request.
3. **Hand each factory a picked scope.** A factory's signature declares exactly which zernos it reads; everything else is invisible to it — at compile time and at runtime.

## Declare zernos where they're used

Zerno construction is inert — zero I/O, zero validation, zero side effects — so these modules are free to import from anywhere, including tests and build tooling.

```ts
// src/db/zernos.ts
import { zerno, defineZernos } from "komirka";
import { z } from "zod";

export const dbUrl = zerno({
  key: "DATABASE_URL",
  schema: z.string().url(),
  doc: "Primary Postgres connection string",
  // exposure defaults to "secret" — fail closed. Safe requires no annotation.
});

export const poolSize = zerno({
  key: "DB_POOL_SIZE",
  schema: z.coerce.number().int().min(1).max(64), // Zerno<number> — T is the schema OUTPUT
  doc: "Max connections in the pg pool",
  exposure: "public",
  default: 10, // validated at bind like any sourced value — defaults get no privileged path
});

export const dbZernos = defineZernos([dbUrl, poolSize]);
```

```ts
// src/billing/zernos.ts
import { zerno, defineZernos } from "komirka";
import { z } from "zod";

export const stripeKey = zerno({
  key: "STRIPE_SECRET_KEY",
  schema: z.string().startsWith("sk_"),
  doc: "Stripe API secret key",
});

export const webhookSecret = zerno({
  key: "STRIPE_WEBHOOK_SECRET",
  schema: z.string().startsWith("whsec_"),
  doc: "Signing secret for verifying Stripe webhook payloads",
});

export const billingZernos = defineZernos([stripeKey, webhookSecret]);
```

## Factories take a scope, not the world

The signature is the contract: `createRepo` can read the database zernos and nothing else. The root snapshot is deliberately **not** assignable to `Scope` — only `pick()` produces one — so you cannot accidentally pass the whole config through a narrow door.

```ts
// src/db/repo.ts
import { Pool } from "pg";
import type { Scope, ConfigOf } from "komirka";
import { dbZernos, dbUrl, poolSize } from "./zernos";

export function createRepo(cfg: Scope<ConfigOf<typeof dbZernos>>) {
  const pool = new Pool({
    connectionString: cfg.unwrap(dbUrl),
    max: cfg.unwrap(poolSize),
  });

  return {
    async findUser(id: string) {
      const { rows } = await pool.query("select * from users where id = $1", [id]);
      return rows[0] ?? null;
    },
    close: () => pool.end(),
  };
}
```

```ts
// src/billing/client.ts
import type { Scope, ConfigOf } from "komirka";
import { billingZernos } from "./zernos";

export function createBilling(cfg: Scope<ConfigOf<typeof billingZernos>>) {
  /* Stripe client + webhook verification, reading cfg.unwrap(stripeKey) etc. */
}
```

Calling `createRepo` without a bound, picked scope is a compile error — a dependency cannot conjure config out of thin air, and neither can your own modules.

## The composition root

```ts
// src/main.ts
import express from "express";
import { zerno } from "komirka";
import { bind } from "komirka/node";
import { z } from "zod";
import { dbZernos, dbUrl, poolSize } from "./db/zernos";
import { billingZernos, stripeKey, webhookSecret } from "./billing/zernos";
import { createRepo } from "./db/repo";
import { createBilling } from "./billing/client";

// main.ts is the call site for the port, so its zerno lives here — near its use.
const port = zerno({
  key: "PORT",
  schema: z.coerce.number().int().min(1).max(65535),
  exposure: "public",
  default: 3000,
});

// Bare entries: each zerno's declared refs resolve against the node entry
// point's natural defaults — process.env and the filesystem.
const root = bind({ dbUrl, poolSize, stripeKey, webhookSecret, port });

const repo = createRepo(root.pick(dbZernos));
const billing = createBilling(root.pick(billingZernos));

const app = express();

app.get("/users/:id", async (req, res) => {
  const user = await repo.findUser(req.params.id);
  if (!user) return res.status(404).end();
  res.json(user);
});

app.post("/webhooks/stripe", express.raw({ type: "application/json" }), (req, res) => {
  const ok = billing.verifyWebhook(req.body, req.header("stripe-signature") ?? "");
  res.status(ok ? 204 : 400).end();
});

app.listen(root.unwrap(port));
```

Bare entries are the whole story until a deployment disagrees with a name. When the platform injects the Postgres URL under its own variable, rename just that ref — per entry, in place:

```ts
import { bind, env } from "komirka/node";

const root = bind({
  dbUrl: env(dbUrl, "PG_CONNECTION_STRING"), // patches the env ref in place — other declared fallbacks survive
  poolSize,
  stripeKey,
  webhookSecret,
  port,
});
```

The rename touches only the matching declared ref; asking to rename a ref kind the zerno never declared is an `UnknownWiringError` at bind. And wiring never forks identity: `dbUrl` is the same zerno under any name — `pick`, scopes, and the lockfile all agree, and errors speak the effective name first.

`bind` is eager over the bound subset: every listed zerno is resolved and validated on that one line. A broken environment fails once, with the complete picture — one aggregated `MissingConfigError`, not one crash per variable per redeploy:

```text
MissingConfigError: 3 of 5 zernos failed to bind

  DATABASE_URL        missing   no ref produced a value  (tried: env DATABASE_URL)
  STRIPE_SECRET_KEY   missing   no ref produced a value  (tried: env STRIPE_SECRET_KEY)
  DB_POOL_SIZE        invalid   1 issue at value         (from: env DB_POOL_SIZE)
```

For secret zernos the report carries issue counts and paths only — never validator messages, because BYO validators echo the values they receive. When a value binds but you're not sure where it came from, `root.explain(dbUrl)` names the ref that supplied the value and the effective name it read, with secret values redacted.

## Async secrets at boot: one await

Production often keeps secrets in a manager, not in env. Which manager, and under which key, is deployment wiring — so it lives at the composition root, not on the zerno. `chain` swaps a zerno's declared refs for an ordered list of links, so one binding covers prod and local dev:

```ts
// src/config/ssm.ts — providers are plain objects you own; komirka ships no cloud SDK clients.
import { SSMClient, GetParameterCommand } from "@aws-sdk/client-ssm";

const client = new SSMClient({});

export const ssm = {
  id: "ssm", // the id is what the lockfile records — anonymous impls are a bind-time error on bake-covered targets
  color: "async", // declared, not inferred: an untyped color is a compile error
  async resolve(key: string) {
    const out = await client.send(new GetParameterCommand({ Name: key, WithDecryption: true }));
    return out.Parameter?.Value;
  },
};
```

```ts
// src/main.ts
import { chain } from "komirka";
import { bind, env, provider } from "komirka/async";
import { ssm } from "./config/ssm";

const root = await bind({
  dbUrl,
  poolSize,
  port,
  webhookSecret,
  stripeKey: chain(stripeKey, provider(ssm, "/prod/billing/stripe-key"), env("STRIPE_SECRET_KEY")),
}).snapshot();
// One await, at boot. The returned Snapshot is pinned:
// every read after this line is synchronous, including deep inside request handlers.
```

`chain` is the only full-replacement spelling: it discards the zerno's declared refs in favor of the ordered links you give it, and resolution errors list the refs it dropped — a fallback that quietly stopped being consulted still shows up in the failure.

Fetching from SSM is async, and the color is in the types: every link carries its impl's color, and a single async link recolors the whole binding to an `AsyncBinding` — a type only `komirka/async`'s `bind` produces. `await bind(entries).snapshot()` is the async door, and it is async exactly **once**: the pin resolves everything up front and hands back a plain sync-read `Snapshot`.

Handing an async-colored provider to `komirka/node`'s `bind` is not a footgun that fires later — it doesn't compile. Likewise a schema that turns out to be async — async-ness can be input-dependent, so every validation site checks — throws `AsyncValidatorInSyncBindingError` rather than silently returning a pending promise.

## .env files

Core never parses files — there is no `.env` reader in `komirka`. The recommended route is to let Node populate the process environment itself:

```bash
node --env-file=.env dist/main.js
```

Bare entries then see those values like any other environment variable — `komirka/node`'s natural defaults read the process environment, not the file. If you need more than `--env-file` offers, dotenv is a userland provider you wire per entry — a provider impl, not a core concern. And because every zerno carries a `doc` field, `komirka bake` keeps a `.env.example` generated for you, so the file new teammates copy is never stale.

## Boot errors vs runtime errors

Boot errors mean the **environment or the wiring** is wrong: loud, aggregated, thrown before `listen`. Runtime errors mean a **read** went somewhere it was never given: named, precise, and never a silent fallback.

| Error | Thrown when | Phase |
| --- | --- | --- |
| `MissingConfigError` | a bound zerno is missing or invalid — aggregated across the whole set | boot (`bind`, or the one `await` on an async binding) |
| `UnknownWiringError` | a per-entry rename targets a ref kind the zerno never declared — `env(zerno, …)` on a zerno with no env ref | boot (`bind`) |
| `DuplicateWiringError` | one zerno is reachable through two entries of the same `bind`, groups flattened | boot (`bind`) |
| `AsyncValidatorInSyncBindingError` | a schema returns a thenable at any validation site | boot, or first read under `bind.lazy` |
| `UnboundZernoError` | `unwrap` is called with a zerno the binding never listed | runtime |
| `NotInScopeError` | `unwrap` goes through a `Scope` that did not pick that zerno | runtime |
| `DescriptorMismatchError` | the same key resolves through two different descriptors (dual-package installs); carries a field-level diff and a paste-able `unify` snippet | runtime |

There is no fallback path: a read that cannot be answered throws one of these names, never `undefined`. For a genuinely optional read, say so in code:

```ts
const size = root.safe(poolSize);
if (size.ok) tunePool(size.value);
```

## Anti-patterns

### One god-module of every zerno

```ts
// ✗ src/config/index.ts — "the config file", imported by everything
export const allZernos = defineZernos([
  dbUrl, poolSize, stripeKey, webhookSecret, port, /* …and forty more */
]);
```

Groups are declared near call sites, never as one central module imported at runtime. A god-module makes every importer depend on every config domain, erases the least-privilege story `pick()` gives you (one scope containing everything is no scope at all), and turns every zerno change into an app-wide rebuild. The composition root listing every zerno in a single `bind` entry record is different in kind: `main.ts` is the *only* module that sees everything, and it immediately picks it apart into per-domain scopes.

### Reading outside the composition root

```ts
// ✗ src/db/repo.ts — module-scope binding inside a leaf module
import { bind } from "komirka/node";
import { dbUrl } from "./zernos";

const cfg = bind({ dbUrl });                  // hidden import-time work, and a
export const pool = makePool(cfg.unwrap(dbUrl)); // second binding to drift from main.ts
```

Values exist only inside explicit bindings — there is no ambient global to fall back on, so the temptation is to mint a private binding per module. Don't: it re-reads the environment at import time, dodges the aggregated boot check, and can disagree with the root about wiring. Leaf modules export zernos and factories; only the root binds.

The same discipline holds inside handlers. A scope will not let code wander past what it was given:

```ts
const dbScope = root.pick(dbZernos);

// ✗ a handler reaching past its scope
app.get("/debug/billing", (_req, res) => {
  res.json({ key: dbScope.unwrap(stripeKey) });
  // NotInScopeError: STRIPE_SECRET_KEY is not in this scope
});
```

To be honest about the boundary: `pick()` is least-privilege for mistakes and for auditability, not a sandbox — hostile in-process code can always read `process.env` directly. What it guarantees is that *your* dependency graph states, in types and at runtime, who reads what.

## Testing the factories

Because factories accept a scope rather than reading the environment, tests never touch `process.env` — a test binding is just `bind` with every entry a `literal`. Literals are input-typed, validation runs at bind, and exposure redaction still applies. All-literal entries need no natural sources, so the core `komirka` import is enough, and the result is a plain `Snapshot`:

```ts
// test/repo.spec.ts
import { bind, literal } from "komirka";
import { dbZernos, dbUrl, poolSize } from "../src/db/zernos";
import { createRepo } from "../src/db/repo";

const cfg = bind({
  dbUrl: literal(dbUrl, "postgres://localhost/test"),
  poolSize: literal(poolSize, 2),
});
const repo = createRepo(cfg.pick(dbZernos)); // still validated — a bad literal fails here, not in CI at 2am
```

Construct a fresh snapshot per test and pass it in. No mutation, no restore-after-each, no test-order coupling.
