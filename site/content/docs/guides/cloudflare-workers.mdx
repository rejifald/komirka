Cloudflare Workers is the runtime that makes the case for atomic configuration all by itself. On workerd, environment values are not ambient: they arrive as the `env` argument to your `fetch()` handler, per request. Any library that validates configuration at import time is not validating too early — it is validating in a scope where the configuration *does not exist*.

## Why import-time config cannot see a Worker's env

The incumbent pattern — parse the environment at module scope, export the result — is structurally blind on workerd:

```ts
// config.ts — the import-time pattern shared by t3-env, envalid, convict, node-config
import { z } from "zod";

export const config = z
  .object({ DATABASE_URL: z.string().url() })
  .parse(process.env);

// On workerd this module evaluates at isolate startup, before any request exists:
//   ReferenceError: process is not defined
// The real environment arrives later — as an argument to fetch().
```

A Node compatibility flag can mirror plain-text string bindings onto a `process.env` shim, but that mirror is per-isolate, excludes non-string bindings, and evaporates the moment the bag varies per request (Workers for Platforms dispatch, per-tenant remapping). The honest model is the platform's own model: **env is a function argument**, and configuration must be bound where the argument is in hand.

Knobs are built for exactly this split. A knob is an inert descriptor — zero I/O, zero validation, zero side effects at construction — so declaring it at module scope is safe on every runtime, workerd included. Values exist only inside an explicit binding, and you create that binding inside the handler.

## Declare at module scope, bind per request

```ts
// knobs.ts — inert descriptors; importing this file does nothing observable
import { knob, defineKnobs } from "knobs";
import { z } from "zod";

export const dbUrl = knob({
  key: "DATABASE_URL",
  schema: z.string().url(),
  doc: "Primary Postgres connection string",
  // exposure defaults to "secret" — fail closed
});

export const poolSize = knob({
  key: "DB_POOL_SIZE",
  schema: z.coerce.number().int().min(1),
  exposure: "public",
  default: 10,
  doc: "Upper bound on pooled connections per isolate",
});

export const appKnobs = defineKnobs([dbUrl, poolSize]);
```

```ts
// worker.ts
import { bind } from "knobs/workers";
import { dbUrl, poolSize } from "./knobs";

export default {
  async fetch(request: Request, env: Env, ctx: ExecutionContext): Promise<Response> {
    // Eager bind: every listed knob is resolved and validated NOW.
    // Anything missing or invalid throws ONE aggregated MissingConfigError
    // for this request — before your handler touches a single value.
    const cfg = bind({ dbUrl, poolSize }, { env });

    const db = connect(cfg.get(dbUrl), { max: cfg.get(poolSize) });
    return handle(request, db);
  },
};
```

The bag is a **required** option, at the type level. `bind` from `knobs/workers` supplies no natural defaults — there is no ambient environment on workerd for a bare entry to fall back to — so the entries resolve against the bag you pass, and omitting `{ env }` is a compile error, not a runtime surprise.

The same pattern applies to `scheduled()` and `queue()` handlers — each receives its own `env`, and each gets its own bind.

There is no ambient fallback to reach for. A knob is not a value; reading one outside an explicit binding is a named `UnboundKnobError`, never a silent `undefined`. That is what makes the per-request discipline stick: the library gives you no way to accidentally do it wrong at module scope.

## The stash anti-pattern: caching the first request's env

The classic workaround — capture `env` on the first request so module-scope code can see it — is the bug this page exists to prevent:

```ts
// WRONG: the first request's bag wins forever
import type { Snapshot } from "knobs";

let cached: Snapshot | undefined;

export default {
  async fetch(request: Request, env: Env) {
    cached ??= bind({ dbUrl, poolSize }, { env });
    // ...
  },
};
```

In any deployment where the bag can differ between requests — dispatch namespaces, per-tenant remapping, preview sessions — this serves request 1's tenant to everyone after it. Tenant A's `DATABASE_URL` answers tenant B's traffic: cross-tenant bleed, with secrets. It also quietly converts eager validation into "validated once, for possibly the wrong bag". And it is invisible in single-tenant dev, which is why it survives review.

The fix is not cleverer caching. It is binding per request and letting the library make that cheap. And note that the surface area for this bug is exactly the one line above: a per-request bag can never be captured inside wiring data — combinators like `env(knob, name)` are pure data, and capability enters only at `bind`.

## Per-request eager bind is close to free

Eagerly validating every knob on every request sounds expensive. It isn't, because core keeps a per-process (on workerd: per-isolate) validation cache keyed on the pair **(knob identity hash, raw value)**. Validators are contractually pure, so memoizing by raw input is sound.

```ts
import { identityOf } from "knobs";

identityOf(dbUrl); // "knobv1:xxh128:…" — one half of the cache key; the raw string is the other
```

Request N re-runs a validator only for knobs whose raw value differs from something already proven. On a stable deployment the env bag never changes between requests, so every bind after the first is hash lookups and string comparisons — zero schema executions, no zod on the hot path. You get import-time economics with per-request correctness.

Because the key is the content-addressed identity hash, the cache is also robust in the ways that matter: duplicated knob instances from dual-package installs share entries, and different raw values (different tenants) occupy separate entries instead of thrashing one slot.

## live on workerd env is a no-op

`freshness: "live"` exists for sources that can be re-read while the process runs — files, providers. A Worker's env is not one of them: on workerd the bag is fixed per deployment, and changing a variable or secret creates a *new* deployment whose isolates replace the old ones. There is nothing to poll; the "update" is a restart.

```ts
const logLevel = knob({
  key: "LOG_LEVEL",
  schema: z.enum(["debug", "info", "warn", "error"]),
  exposure: "public",
  default: "info",
  freshness: "live", // meaningless for a workerd env source — emits a dev-mode warning
});

const live = bind([logLevel], { env });
// Typed as LiveBinding, but epoch(logLevel) never advances and every
// snapshot() pins the same values — live over workerd env is pure overhead.
```

Declare env-sourced knobs with the default `freshness: "snapshot"` on Workers. Reserve `live` for the one place it is meaningful here: provider-backed values.

## Provider-backed values: refresh() rides ctx.waitUntil

Some values genuinely change between deployments — a pricing table in KV, a remote-config document. Providers are re-readable, so `freshness: "live"` applies. But two constraints shape the pattern:

1. Core never runs background loops, timers, or watchers — and workerd would not give it execution time between requests anyway.
2. Provider impls declare a **color** — `"sync"` or `"async"`; leaving it untyped is a compile error. Color propagates through the entries: one async entry makes the whole binding an `AsyncBinding`, and only the async-capable `bind` from `knobs/async` produces one. There is no separate load step — the async read happens where you `await` the binding's `snapshot()`, and every read after that pin is sync.

So freshness rides the request lifecycle, via `ctx.waitUntil`:

```ts
import { knob, refresh } from "knobs";
import { bind, provider } from "knobs/async";
import type { AsyncBinding } from "knobs";

const pricing = knob({
  key: "PRICING_TABLE",
  schema: PricingSchema,
  exposure: "public",
  sources: [{ provider: "pricing-kv", key: "pricing/v2" }],
  freshness: "live", // providers are re-readable — live is meaningful here
});

// A provider impl is a plain object you write; core ships no KV or
// secret-manager clients. The id is what the lockfile records; the
// declared color is what colors the binding.
function pricingKvProvider(kv: KVNamespace) {
  return {
    id: "pricing-kv",
    color: "async" as const,
    async read(key: string) {
      return kv.get(key, "json");
    },
  };
}

let live: AsyncBinding | undefined;

export default {
  async fetch(request: Request, env: Env, ctx: ExecutionContext) {
    // provider(entry, impl) supplies the impl for the knob's declared
    // provider ref; the impl's async color makes this an AsyncBinding.
    live ??= bind({ pricing: provider(pricing, pricingKvProvider(env.PRICING_KV)) });

    const cfg = await live.snapshot(); // pin: this request sees one coherent view, no torn reads
    ctx.waitUntil(refresh(live));      // fetch a newer table for LATER requests

    return quote(request, cfg.get(pricing));
  },
};
```

The current request is always served from the last pinned snapshot — the first `snapshot()` is where the provider read actually happens; `refresh()` produces a new pinned snapshot for requests that come after. Refresh is single-flight coalesced in core, so a burst of concurrent requests each calling `waitUntil(refresh(live))` triggers one provider read, not a stampede. On failure it is all-or-nothing by default (opt in to `partial: "keep-old"` if you prefer), and a refresh that returns invalid data throws — the caller keeps the previous pinned snapshot, so a bad deploy of the pricing document fails the refresh loudly (with `onInvalid` fired once for the bad raw) instead of crashing the edge; requests keep serving the old snapshot they already hold. `live.health()` reports per-knob staleness and the last validation error if you expose a liveness probe.

One boundary, stated plainly: caching `live` at module scope is safe here *only* because nothing inside it is per-request — the KV namespace handle the impl closed over is fixed for the isolate's deployment. The per-request thing, the env bag's values under dispatch, must never be captured this way. Provider handles: cacheable. Tenant env: never.

## Multi-tenant: one contract, many wirings

The env bag is a bind option and names are per-entry wiring data. That makes per-tenant remapping a pair of `env()` renames, and the config contract identical across tenants:

```ts
// One physical Worker fronting many tenants; per-tenant vars live under
// prefixed names in the same deployment:
//   TENANT_ACME_DATABASE_URL,   TENANT_ACME_DB_POOL_SIZE,
//   TENANT_ZENITH_DATABASE_URL, TENANT_ZENITH_DB_POOL_SIZE, …
import { bind, env } from "knobs/workers";
import { dbUrl, poolSize } from "./knobs";

function tenantEntries(tenant: string) {
  const prefix = `TENANT_${tenant.toUpperCase()}_`;
  // env() here is the wiring combinator — pure name-ref data, no bag captured
  return {
    dbUrl: env(dbUrl, `${prefix}DATABASE_URL`),
    poolSize: env(poolSize, `${prefix}DB_POOL_SIZE`),
  };
}

export default {
  async fetch(request: Request, env: Env, ctx: ExecutionContext) {
    const tenant = tenantFromHost(new URL(request.url).hostname);

    // Same descriptors, different names. Wiring never forks identity:
    // the contract does not change per tenant.
    const cfg = bind(tenantEntries(tenant), { env });

    // A misconfigured tenant fails HERE — one MissingConfigError naming every
    // missing or invalid knob — and only that tenant's requests fail.
    return handle(request, cfg);
  },
};
```

`env(dbUrl, name)` is a per-kind ref rename: it patches the knob's declared env ref in place and preserves any other declared fallbacks. It is deployment wiring, not a new knob — identity is untouched, so the contract, the lockfile entry, and the validation cache are shared across every tenant.

Because binding is per request, the blast radius is per request: tenant ZENITH missing its pool size returns errors for ZENITH and nobody else — and the error speaks the effective name it looked for, `TENANT_ZENITH_DB_POOL_SIZE`, first. The validation cache keys on (identity, raw), so each tenant's values occupy their own entries — tenants do not evict each other, and the steady state stays validator-free even with hundreds of tenants. When you need to see what actually resolved, `cfg.explain(dbUrl)` reports which ref supplied the value and the effective name it read; the secret value itself is redacted.

On Workers for Platforms, the renames disappear entirely: each dispatched user Worker receives its own `env`, and `bind({ dbUrl, poolSize }, { env })` inside it is already tenant-correct.

## Testing handlers without a Workers runtime

Factor handler logic to accept a `Snapshot` (or a `Scope` via `pick()`), keep the env bag confined to the `fetch()` entry point, and unit tests need neither miniflare nor a single `process.env` mutation — a test snapshot is an all-literal bind:

```ts
import { bind, literal } from "knobs";
import { dbUrl, poolSize } from "./knobs";

const cfg = bind({
  dbUrl: literal(dbUrl, "postgres://user:pw@localhost:5432/test"),
  poolSize: literal(poolSize, 2),
});
// Still validated, at bind: a non-URL literal here fails exactly like
// production would. Literal entries need no bag and no natural defaults,
// so this is core bind — no workers entry point in your tests at all.

const res = await handle(new Request("https://example.test/health"), cfg);
```

Construct one per test, pass it in, and the per-request binding discipline you use in production is the same shape your tests exercise.
