## The setup

Your configuration lives in a database table. An admin panel edits it. A fleet of replicas reads it. This guide covers that shape end to end.

The tool is the **async binding**: live semantics with async transport plus operator-plane writes. The mental model carries over unchanged ‚ÄĒ pin with `snapshot()`, compare `epoch()`, inspect `health()` ‚ÄĒ with two additions: reads that may touch the network are awaited, and `set()` writes back through the binding. You get one from the `tessellum/async` entry point: its `bind` accepts async-colored entries, and the coloring rule does the typing ‚ÄĒ any async entry makes the whole binding an `AsyncBinding`. Only the shared types and error classes sit in core, which stays synchronous.

## The provider contract

```ts
import type { Json } from "tessellum";

type ReadSlot =
  | { raw: Json; version?: string }   // present
  | { absent: true }                       // definitively absent
  | { error: unknown };                    // per-key store failure (final ‚ÄĒ provider retries transients internally)

interface AsyncProvider {
  id: string;                // stable identity ‚ÄĒ recorded in the lockfile
  color: "async";
  limits?: { maxReadBatch?: number; maxWriteBatch?: number; maxConcurrentBatches?: number };
  read(keys: readonly string[]): Promise<readonly ReadSlot[]>;
  // INPUT-ALIGNED: result[i] answers keys[i]; a length mismatch is ProviderContractError.
  write?(entries: ReadonlyArray<{ key: string; raw: Json; ifVersion?: string }>):
    Promise<ReadonlyArray<{ version?: string; storedRaw?: Json }>>;
  capabilities?: {
    versionTokens?: "unique" | "coarse" | "none";   // timestamps are NOT versions
    strongRead?: boolean; echoWrite?: boolean; normalizes?: boolean;
    atomicWrites?: boolean;      // batch write = one transaction, all-or-nothing incl. CAS abort
    consistentReads?: boolean;   // one read() call within maxReadBatch = one consistent cut
    holdsSecrets?: boolean;      // this general store legitimately holds secrets ‚ÄĒ full secret obligations apply
  };
}
```

The contract is batch-shaped all the way down ‚ÄĒ there is deliberately no single-key method. `unwrap()` and `set()` ride the same path as single-element batches that core unwraps, so the binding surface never changes and a provider has exactly one read path and one write path to get right: the batch path is the tested path. Results are **input-aligned** ‚ÄĒ `result[i]` answers `keys[i]` ‚ÄĒ and a length mismatch is a `ProviderContractError`, surfaced loudly instead of values silently attributed to the wrong tessera.

The `id` is not decoration: it is what the lockfile records for bake-covered targets, so an anonymous impl is a bind-time error there. And `color` is load-bearing type information ‚ÄĒ leaving it untyped is a compile error, because coloring is how `bind` computes its result type.

Providers also carry an optional `class` field: `class?: "config"` (the default) marks an ordinary configuration store, while `class: "secret"` marks the dedicated `SecretProvider`, covered on the Secrets page. A general (config-class) provider whose store legitimately holds secrets must declare `holdsSecrets: true` to be wireable to secret-exposure tesserae. The pairing rule cuts both ways: a secret-class provider serves **only** secret tesserae, and a provider-sourced secret tessera requires either a secret-class provider or a `holdsSecrets` general provider (env- and file-sourced secrets are unaffected). `holdsSecrets` inherits the **full** secret obligation set ‚ÄĒ error quarantine, result reshaping, the secret conformance profile, echo-verify disabled for those tesserae ‚ÄĒ the opt-in buys pairing legality, never reduced hygiene, and it lands in a flagged section of the lockfile.

The envelope is versioned from day one: a present slot may carry a `version` (a SQL row version, a DynamoDB condition attribute, an etcd revision), and each `write` entry may carry `ifVersion` for compare-and-set. Providers without versions still work ‚ÄĒ they degrade to **documented** last-write-wins. That is an acceptable default and an unacceptable ceiling, so the seam exists even if your first provider ignores it.

The heart of the contract is a two-sided treaty. Each side's list is short, and neither side may assume the other's job.

**Core guarantees, on every call it makes:**

- Keys in a read batch are unique, and batches are never empty.
- A batch never exceeds the declared `limits` ‚ÄĒ chunking is exclusively core's job. A provider must **never** split a batch internally: internal chunking silently voids `consistentReads` and resurrects the torn-pair bug (a discriminant in one chunk, its branch values in another). Read chunks fan out in parallel, bounded by `maxConcurrentBatches` (default ~4) ‚ÄĒ which is what makes a `maxReadBatch: 1` store like Vault KV v2 usable at `ready()` with 20 secrets ‚ÄĒ while write chunks stay sequential and dependency-ordered.
- Batches are **segregated by exposure**: secret and public tesserae never share one wire batch, even on the same provider ‚ÄĒ a batch-level error message on the public side must never be able to enumerate secret rows.
- Duplicate keys in a write batch are rejected before any I/O happens.
- Multi-entry writes are issued **only** to providers declaring `atomicWrites: true`. For everyone else, core sequences single-entry writes itself, in dependency order ‚ÄĒ branch values first, discriminant last ‚ÄĒ stopping at the first failure. Dependency ordering stays core's knowledge; it is never an unstated obligation on a provider's loop order.

**A provider guarantees, in every answer it gives:**

- Every position gets a slot: `{ absent: true }` for definitively missing (a set-difference on a SQL `IN()` result or a prefix scan), `{ error }` for a per-key store failure (say, DynamoDB `UnprocessedKeys` still unprocessed after the provider's own retries). An `{ error }` slot is final ‚ÄĒ transient retries live inside the provider, not in core.
- A whole-call throw means connection-level unavailability for **all** requested keys ‚ÄĒ never "some keys failed".
- `atomicWrites` means nothing persists on a mid-batch CAS miss, and the resulting `WriteConflictError` names every conflicting key.
- `echoWrite` must hold on the batch path. A transactional path that cannot echo declares `echoWrite: false` and relies on `strongRead`.

Claims this load-bearing do not rest on the honor system: a published `verifyProvider(impl)` conformance kit runs against your real store and mechanically probes each one ‚ÄĒ input alignment, absent-vs-error slots, empty-batch and duplicate-key rejection, the CAS conflict shape, atomic abort (a poisoned second entry must leave the first unwritten), echo on the batch path. Passing it is the documented bar for declaring `atomicWrites` or `consistentReads`.

A Postgres provider is one query per method:

```ts
import type { AsyncProvider } from "tessellum/async";
import { WriteConflictError } from "tessellum";
import { sql } from "./db";

export const pgProvider: AsyncProvider = {
  id: "pg-config",
  color: "async",
  limits: { maxReadBatch: 1000 },
  capabilities: {
    versionTokens: "unique",
    echoWrite: true,
    atomicWrites: true,
    consistentReads: true,
  },
  async read(keys) {
    const rows = await sql`SELECT key, raw, version FROM config WHERE key = ANY(${keys})`;
    const found = new Map(rows.map((r) => [r.key, { raw: r.raw, version: String(r.version) }]));
    return keys.map((k) => found.get(k) ?? { absent: true }); // set-difference: input-aligned, absent is definitive
  },
  async write(entries) {
    return sql.begin(async (tx) => {
      const out = [];
      for (const { key, raw, ifVersion } of entries) {
        const [row] = ifVersion
          ? await tx`UPDATE config SET raw = ${raw}, version = version + 1
                     WHERE key = ${key} AND version = ${ifVersion} RETURNING raw, version`
          : await tx`INSERT INTO config (key, raw, version) VALUES (${key}, ${raw}, 1)
                     ON CONFLICT (key) DO UPDATE SET raw = ${raw}, version = config.version + 1
                     RETURNING raw, version`;
        if (!row) throw new WriteConflictError([key]); // aborts the transaction ‚ÄĒ nothing persists
        out.push({ version: String(row.version), storedRaw: row.raw });
      }
      return out; // input-aligned; storedRaw is the echo
    });
  },
};
```

If this Postgres store also holds secrets, declare `holdsSecrets: true` in its capabilities and accept the full secret obligation set; otherwise keep secrets in a dedicated secret-class store.

Wrapping a single-key store ‚ÄĒ a Redis client, a REST endpoint ‚ÄĒ is a three-line **recipe**, not a second contract form and not an exported helper:

```ts
async read(keys) {
  return Promise.all(keys.map(async (k) => {
    const hit = await store.get(k);
    return hit === undefined ? { absent: true } : { raw: hit.value, version: hit.version };
  }));
}
```

A wrapper like this must not declare `consistentReads` ‚ÄĒ each key is its own round trip, so a batch is never one cut.

## Wiring it up

```ts
import { tessera } from "tessellum";
import { bind, provider } from "tessellum/async";
import { z } from "zod";

export const rateLimit = tessera({
  key: "api.rateLimit",
  schema: z.coerce.number().int().positive(),
  doc: "Requests per minute, per API key.",
  exposure: "public",
  default: "100",
  freshness: "live",
});

export const corsOrigins = tessera({
  key: "http.corsOrigins",
  schema: z.string().transform((s) => s.split(",").map((o) => o.trim())),
  doc: "Comma-separated allowed origins.",
  exposure: "public",
  default: "https://app.example.com",
  freshness: "live",
});

const cfg = bind(
  {
    rateLimit: provider(rateLimit, pgProvider),
    corsOrigins: provider(corsOrigins, pgProvider),
  },
  { maxStalenessMs: 5_000 },
);
await cfg.ready();
```

`bind` is inert ‚ÄĒ it never performs I/O. Each `provider()` entry is async-colored, so the result types as an `AsyncBinding` automatically; there is no separate async constructor. `await cfg.ready()` is the eager startup line: one batched `read` (chunked against `limits.maxReadBatch` when needed), then resolve and validate every bound tessera, failing fast with a single aggregated `MissingConfigError` that names everything wrong at once, not just the first thing.

Wiring is per entry, so one binding mixes transports freely. A tessera that should read Parameter Store but fall back to an env var spells the whole chain out ‚ÄĒ `chain` is the only full-replacement wiring, and if the tessera declared fallback refs of its own, errors will list exactly what the chain dropped:

```ts
import { chain } from "tessellum";
import { env } from "tessellum/async"; // node combinators are re-exported here

bind({
  rateLimit: provider(rateLimit, pgProvider),
  sendgridKey: chain(sendgridKey, provider(ssmProvider, "/prod/sendgrid/api-key"), env("SENDGRID_KEY")),
});
```

Links carry color: the SSM link is async, so this binding is an `AsyncBinding` no matter that the `env` fallback behind it is sync.

## Reading

Request paths pin. `snapshot()` returns a coherent, immutable view: if the cache is fresh it costs zero I/O; if some keys have exceeded `maxStalenessMs`, it issues one batched `read` for exactly those keys.

```ts
app.use(async (req, _res, next) => {
  req.config = await cfg.snapshot(); // usually zero-I/O
  next();
});

app.get("/orders", (req, res) => {
  const limit = req.config.unwrap(rateLimit); // sync and stable for the whole request
  // ...
});
```

For scripts and jobs there is the colored read: `await cfg.unwrap(rateLimit)`.

`maxStalenessMs` is an honest hard bound, not a hint. An expired key **blocks** on a re-read ‚ÄĒ there is no silent stale-while-revalidate serving old values while claiming freshness. Expiry under load coalesces: a hot tessera expiring during a traffic spike triggers **one** provider read via per-key single-flight, not one per request.

"Coherent" is a claim about the store, and core does not take it on faith. `consistentReads` is a declared capability demoted from honor-claim to round-trip optimization: when a provider declares it and a coherence-requiring read fits in one `maxReadBatch` batch, that single `read()` call is one consistent cut and core is done in one round trip. When the capability is falsy ‚ÄĒ or core had to chunk the read across batches ‚ÄĒ core buys coherence itself with a bounded re-check loop: read the batch, re-read the discriminant, and retry (up to 3√ó) if it moved. **DynamoDB pitfall:** `BatchGetItem` is per-item consistency, not one cut ‚ÄĒ a provider built on it must declare `consistentReads: false` and let the re-check loop do its job, or switch to `TransactGetItems` and declare `maxReadBatch: 100`.

## Writing: `set()` is write-through

```ts
await cfg.set(rateLimit, "200");
```

Three rules make `set()` predictable.

**It takes the schema's input type ‚ÄĒ the raw ‚ÄĒ not the parsed output.** Standard Schema has no inverse: given `z.string().transform((s) => s.split(","))`, no library can turn the array back into the string ‚ÄĒ the transform only runs forward. So the caller owns the raw representation, and core never serializes or inverts:

```ts
snap.unwrap(corsOrigins);
// ["https://app.example.com"]  ‚Üź parsed output

await cfg.set(corsOrigins, "https://app.example.com, https://admin.example.com");
//                          ‚ÜĎ raw input, exactly what a source would supply
```

The pipeline: validate the raw against the tessera's schema (an invalid write is rejected before the provider ever sees it) ‚Üí `provider.write([{ key, raw }])` ‚ÄĒ a single-element batch ‚ÄĒ verbatim ‚Üí install the **parsed** value into the local cache ‚Üí bump the epoch.

**Write-through, never invalidate-and-refetch.** Installing the parsed value locally is what guarantees read-your-writes on the writing replica, even when the provider's read path lags its write path (read replicas, eventually-consistent reads). Refetching after a write could hand back the pre-write value from a lagging reader.

| Reader | Sees the write |
| --- | --- |
| The writing replica | Immediately (write-through) |
| Every other replica | Within `maxStalenessMs` plus provider replication lag |
| Concurrent writers | Last-write-wins unless `ifVersion` is used; divergence bounded by `maxStalenessMs` |

**It targets the highest-precedence ref in the tessera's wiring.** A write that lands below a shadowing ref would change every replica except the operator's own ‚ÄĒ the one machine where the change appears to work is the one machine where it hasn't. Forbidden:

```ts
import { chain } from "tessellum";
import { file } from "tessellum/async";

const logLevel = tessera({
  key: "log.level",
  schema: z.enum(["debug", "info", "warn", "error"]),
  default: "info",
  freshness: "live",
});

const cfg = bind({
  logLevel: chain(logLevel, file("./config.local.json"), provider(pgProvider)),
});

await cfg.set(logLevel, "debug");
// NotWritableError: "log.level" is currently supplied by file "./config.local.json",
// which shadows provider "pg-config". Remove the local override or write to the file.
```

The binding records which ref supplied each value, so the refusal names the shadowing ref precisely ‚ÄĒ the same provenance that `explain()` and `safe` report.

## Verifying writes: did the store keep what you gave it?

`set()` validates what you *intend* to write ‚ÄĒ nothing yet proves what the store *kept*.
Column truncation, charset mangling, JSON number precision loss, a `BEFORE UPDATE`
trigger: all of these can silently store something other than your bytes, and the mangled
row then reaches every replica at its next staleness expiry. Worst of all is the
mangled-but-*valid* case ‚ÄĒ an endpoint URL truncated at column width to
`https://api.example.com/v` is still a URL, so the poisoned-slot machinery never fires
and the fleet silently adopts corrupted config. Write verification is the only thing that
catches that class.

```ts
const result = await cfg.set(endpointUrl, url, { ifVersion: version, verify: "revert" });
// SetResult: { version, outcome: "installed" | "verified" | "inconclusive" | "superseded", reason? }
```

**How verification works, in order of preference:**

- **Write-echo** (`capabilities.echoWrite`): the provider's `write()` returns
  `{ version, storedRaw }` for each entry ‚ÄĒ Postgres `UPDATE ‚Ä¶ RETURNING`, DynamoDB
  `ReturnValues: ALL_NEW`. Verification compares in the *same round trip*: zero extra
  requests, no replica-lag ambiguity. This is the primary mechanism.
- **Strong read-back** (`capabilities.strongRead`): one re-read from the
  primary/consistent path, matched by version token.
- Plain read-back works but is chronically **inconclusive** behind lagging replicas ‚ÄĒ the
  read may predate your own write.

**Equality is judged on the raw, canonically:** byte equality for string raws, RFC 8785
canonical-JSON equality for JSON raws. A JSONB store reordering keys therefore passes
canonical comparison in report mode ‚ÄĒ but such a store should still declare
`normalizes: true`, which keeps revert gated off. Truncation, charset mangling, and
number-precision loss all fail ‚ÄĒ they change the canonical form. A per-tessera `equals()`
hook is the escape hatch for stores whose rewrites are semantic-preserving but
comparison-visible.

**Outcomes are honest about certainty:**

- `verified` ‚ÄĒ version-matched, content equal. Done.
- `inconclusive` ‚ÄĒ the write *committed* but could not be confirmed (stale read-back,
  missing row on a fresh create, provider omitted the version). This is a non-throwing
  return, and the parsed value is still installed locally ‚ÄĒ the write succeeded; the
  residual uncertainty is exactly what unverified writes carry anyway.
- `superseded` ‚ÄĒ a concurrent writer landed after you; their write wins, no revert.
- Confirmed mangling ‚ÄĒ version-matched, content different ‚ÄĒ throws **`WriteMangledError`**
  with expected/actual digests and `nowLive: "prev" | "unknown"`. In `"report"` mode the
  writer's local cache entry is immediately expired so it converges with the fleet
  instead of split-braining until its own staleness window.

**Secret tesserae verify differently.** Echo-verify is disabled for them regardless of
provider ‚ÄĒ the secret write result carries no `storedRaw`, so there is nothing raw to
compare. Precedence becomes: **version-token equality** first (the write's version equals
the read-back version ‚áí `verified`, with zero raw comparison; a *newer* read-back version
‚áí the non-throwing `superseded` ‚ÄĒ a concurrent rotation is not corruption), then **digest
read-back** via `strongRead`, then the honest **unverified** outcome ‚ÄĒ the write installs,
and a per-tessera `requireVerifiedWrites` refuses it where policy demands. `WriteMangledError`
for a secret carries expected/actual digests plus `nowLiveDigest`, never raw values.

**Auto-revert (`verify: "revert"`) restores the previous value on confirmed mangling ‚ÄĒ and
it is deliberately hard to hold wrong.** Revert requires version-confirmed corruption
(same version token, different content ‚ÄĒ *never* on inconclusive), which in turn requires
the full precondition bundle, checked at the `set()` call with a typed
`VerifyCapabilityError`:

- the provider declares **unique** version tokens (a per-write row counter or etcd
  revision ‚ÄĒ timestamps are not versions: two writes in one second would let a revert
  clobber a concurrent writer's good value);
- `echoWrite` or `strongRead` is available, and the provider does not declare
  `normalizes: true`;
- revert implies CAS on the original write, so `rawPrev` is provably the immediate
  predecessor;
- `rawPrev` exists (a *create* has nothing to revert to ‚ÄĒ downgraded to report) and
  re-validates before being written back;
- **secret tesserae refuse revert unless opted in per tessera** ‚ÄĒ auto-revert of a credential
  rotation silently puts the *leaked* key back in service; the error's `nowLive` field
  and previous-value digest exist so a rotation-undo is unmissable.

The result reports `revertWritten` (the revert write was accepted) and `revertVerified`
(one further read-back confirmed it) separately ‚ÄĒ against a store that rewrites every
value, the revert is rewritten too, and pretending otherwise would be the same lie the
feature exists to catch.

Verification is a **writer-side early warning**, not the fleet's safety net: the fleet is
protected by the poisoned-slot machinery and alerting on `InvalidLiveValueError` ‚ÄĒ except
for mangled-but-valid values, which only verification sees. `setMany()` supports
`verify: "report"`; batch revert remains deferred ‚ÄĒ when it lands, it additionally gates
on `atomicWrites` and restores the *entire* previous post-state, never a partial revert
that strands a never-validated combination.

## Concurrent writers

Read-modify-write without `ifVersion` loses updates silently: two admins both read `100`, both add 50, both save ‚ÄĒ the value lands on `150` and one increment vanishes without a trace. With `ifVersion`, the second write is refused instead:

```ts
async function bumpRateLimit(delta: number) {
  for (let attempt = 0; attempt < 3; attempt++) {
    const snap = await cfg.snapshot();
    const current = snap.unwrap(rateLimit);
    const { version } = snap.explain(rateLimit); // the provider version this cache last saw
    try {
      return await cfg.set(rateLimit, String(current + delta), { ifVersion: version });
    } catch (e) {
      if (e instanceof WriteConflictError) continue; // someone else won ‚ÄĒ re-read, recompute, retry
      throw e;
    }
  }
  throw new Error("gave up after 3 conflicts");
}
```

## Multi-tessera updates: `setMany()`

An admin panel's Save button rarely writes one tessera. `setMany()` validates the **post-state as a whole** ‚ÄĒ every schema, variant completeness, cross-tessera rules ‚ÄĒ before a single byte reaches the provider. A plain `set()` validates against its own tessera's schema only; that is the documented difference.

```ts
await cfg.setMany([
  { tessera: rateLimit, raw: "200", ifVersion: rlVersion },
  { tessera: corsOrigins, raw: "https://app.example.com,https://admin.example.com" },
]);
```

Atomicity follows the declared capability. Against a provider with `atomicWrites: true`, the batch goes down as **one** `write()` call ‚ÄĒ one transaction, all-or-nothing, including the CAS abort: a version miss on any entry means nothing persisted, and the `WriteConflictError` names every conflicting key. Against a provider without it, `setMany()` **refuses before attempting anything** ‚ÄĒ a half-applied batch is worse than a rejected one. Passing `{ allowNonAtomic: true }` opts into core sequencing single-entry writes itself, in dependency order ‚ÄĒ branch values first, discriminant last ‚ÄĒ stopping at the first failure, so a crash mid-batch tears in the benign direction (staged-but-unused values, never an active branch missing its values). Either way the ordering is core's knowledge, never the provider's loop to get right.

**One transaction means one batch.** Chunking is core's job, and an atomic write is the one thing core never chunks ‚ÄĒ splitting it across `write()` calls would make it *several* transactions, not one. So an atomic `setMany` is capped at the provider's `limits.maxWriteBatch`: the number of entries the store can commit in a single all-or-nothing write (DynamoDB `TransactWriteItems` caps at 100; a Postgres provider may declare far more). A larger atomic batch is refused up front with **`AtomicBatchTooLargeError`**, before any write reaches the store ‚ÄĒ because a partially-applied multi-tessera write (discriminant written, branch values not) is exactly the torn durable state this feature exists to prevent, and a silent chunk-split would smuggle it back in. Keep a rotate-together set within one batch ‚ÄĒ a discriminant and its branch values almost always are ‚ÄĒ or, when a large multi-tessera update genuinely cannot be one transaction, pass `{ allowNonAtomic: true }` and accept the dependency-ordered, benign-tear sequencing above. This is the write-side of the coherent-group rule: a coherent *read* is one `read()` to one store, and an atomic *write* is one `write()` to one store ([ADR 0006](https://github.com/rejifald/tessellum/blob/main/docs/adr/0006-atomic-setmany-vs-write-batch.md)).

## Switching a variant from the database

Discriminant tesserae are `freshness: "snapshot"` by rule: pinned at `ready()` and exempt from staleness re-reads. A database flip must never switch your process topology mid-flight.

`set()` on a discriminant is therefore a **staged write**: it writes for the fleet but does not install locally and does not bump the epoch. The local binding adopts it only through an explicit `refresh()`, which re-reads the discriminant and its full active branch as one coherence-requiring batched read ‚ÄĒ one consistent cut when the provider's `consistentReads` holds and the branch fits in one batch, the bounded discriminant re-check loop otherwise. Given a payments variant group:

```ts
// 1. Stage the target branch's values ‚ÄĒ legal while the branch is inactive.
//    Writes validate against the target tessera's own branch schema;
//    InactiveVariantError is read-side only and never applies to writes.
await cfg.set(adyenApiKey, "aq_live_9f‚Ä¶");
await cfg.set(adyenMerchant, "ACME_EU");

// 2. Flip the discriminant ‚ÄĒ staged: the fleet will see it, this process not yet.
await cfg.set(paymentsProvider, "adyen");

// 3. Adopt the new topology here, all-or-nothing.
await cfg.refresh();
```

A single `setMany()` with `allowNonAtomic: true` encodes the same runbook mechanically.

## When things fail

Six different events, six different behaviors ‚ÄĒ never blurred together:

| Event | Behavior | Error / hook |
| --- | --- | --- |
| Re-read returns an **invalid** value | Tessera is poisoned: pins still succeed, the tessera occupies an error slot | `InvalidLiveValueError` on `unwrap`; `onInvalid` fires once per distinct invalid raw |
| An `{ error }` **slot** answers one key | Per-key unavailable: that key serves its cached value within `maxStalenessMs`, beyond it throws ‚ÄĒ the other keys in the batch are untouched | `SourceUnavailableError` naming the key; `onSourceError` |
| Provider **throws** the whole call | Provider-down for every requested key: each serves its cached value within `maxStalenessMs` (that budget is the contract, not inherited state); beyond it, throw | `SourceUnavailableError`; `onSourceError` ‚ÄĒ never routed through `onInvalid` |
| A re-read is **too slow** (`readTimeoutMs`) | Deadline breach: the blocking re-read is abandoned so the caller's wait stays bounded ‚ÄĒ a store too slow to answer within budget is unreachable. The late result may still refresh the cache for the next caller; it never un-throws the caller who breached | `SourceUnavailableError` (cause: the deadline); `onSourceError` |
| Store **down at `ready()`** (boot, no cache) | No cached value can exist yet, so bounded retry with backoff (each attempt ‚Č§ `readTimeoutMs`) until it answers or `readyTimeoutMs` elapses, then the boot fails | `SourceUnavailableError` past `readyTimeoutMs`; `readyTimeoutMs: 0` fails on the first strike |
| **Write** fails | Cache untouched, no epoch bump, core never retries (no loops ‚ÄĒ the provider or caller retries) | `WriteFailedError` with a scrubbed cause |

The slot-versus-throw split is deliberate: one poisoned row must not DoS every tessera on the provider. A single unreadable key degrades exactly that key; only connection-level failure degrades them all. And an `{ error }` slot is final ‚ÄĒ the provider already retried transients internally, so core adds no retry loop on top. One more rule for secret-class and `holdsSecrets` providers: their errors are **quarantined** ‚ÄĒ never propagated as error objects or `cause` chains, with `SourceUnavailableError` carrying only `providerId`, `key`, `errorName`, and a `messageDigest` ‚ÄĒ because a store client's own connection error can embed the provider's credential in its message, which no tessera-value scrub can find.

**`maxStalenessMs` bounds staleness; `readTimeoutMs` bounds the wait.** The rows above that "serve within `maxStalenessMs`, then throw" govern how *stale* a served value may be ‚ÄĒ not how long a re-read may *block*. A provider that is not down but merely slow would otherwise wedge every coalesced request for as long as it takes, and `maxStalenessMs: 5_000` would do nothing about a 30-second read. Instead each blocking re-read ‚ÄĒ an expired key, or a `coherent` group's cut and its bounded re-check loop ‚ÄĒ is bounded by `readTimeoutMs` (default `maxStalenessMs`), and a breach throws `SourceUnavailableError` rather than serve a value already past its budget. At boot `ready()` has no cache at all, so a store outage there is bounded retry with backoff, then fail-fast past `readyTimeoutMs` ‚ÄĒ enough to ride out the usual database-and-app-starting-together race without hanging, and still fail the boot loudly if the store is genuinely down ([ADR 0005](https://github.com/rejifald/tessellum/blob/main/docs/adr/0005-async-read-deadline-and-boot-outage.md)).

Because `set()` validates before writing, a poisoned slot rarely comes from your own admin panel ‚ÄĒ it arrives via raw SQL edits or schema skew (below). Current truth wins: the invalid value poisons exactly that tessera.

```ts
const snap = await cfg.snapshot(); // never throws wholesale
snap.unwrap(rateLimit);
// InvalidLiveValueError: "api.rateLimit" invalid since 2026-07-04T10:12:03Z (incident 01JZK‚Ä¶):
//   expected positive integer, got "unlimited". Last valid value observed at 10:07:41Z.
snap.safe(rateLimit); // { ok: false, error } ‚ÄĒ the non-throwing form
```

Every other tessera keeps working; blast radius is proportional to actual usage of the broken one. Poisoned state also bypasses staleness memoization ‚ÄĒ the next read re-reads immediately, so recovery lands within one read of the fix, not one `maxStalenessMs`.

Where coasting beats throwing, opt in at the bind site (never in the descriptor ‚ÄĒ the same tessera may deserve different policies in different apps):

```ts
import { keepLastGood } from "tessellum";

const cfg = bind(
  {
    rateLimit: provider(rateLimit, pgProvider),
    corsOrigins: provider(corsOrigins, pgProvider),
  },
  {
    maxStalenessMs: 5_000,
    invalid: [keepLastGood(rateLimit, { maxCoastMs: 300_000 })], // maxCoastMs is required
  },
);
// past the coast budget ‚Üí CoastBudgetExceededError
```

`health()` reports per tessera: `"valid" | "invalid-was-valid" | "never-valid" | "coasting"`, plus `lastValidAt`, `lastInvalidAt`, `lastThrownAt`, `throwCount` ‚ÄĒ wire it into your health endpoint.

## Reacting to change: epochs, not events

There are no change events. The epoch is a number you compare ‚ÄĒ and it also bumps on the invalid transition, which is why epoch consumers reach for `safe`:

```ts
let pool = makePool((await cfg.snapshot()).unwrap(poolSettings));
let seen = cfg.epoch(poolSettings);

app.use(async (req, _res, next) => {
  const now = cfg.epoch(poolSettings);
  if (now !== seen) {
    const result = (await cfg.snapshot()).safe(poolSettings);
    if (result.ok) {
      const old = pool;
      pool = makePool(result.value);
      seen = now;
      void old.drain();
    } // invalid transition: keep the old pool; health() shows "invalid-was-valid"
  }
  next();
});
```

## Rolling deploys and schema skew

`set()` validates against the **calling process's** schema version. During a rolling deploy, a v1 replica can write a value that v2 replicas reject at read time ‚ÄĒ poisoning that slot on half the fleet. Containment is the story above: error slots keep the blast radius to one tessera, `health()` makes the skew visible per replica, and `keepLastGood` is recommended for DB-sourced tesserae precisely because admin writes and deploys overlap in real fleets.

## Secrets in the database

`set()` on an `exposure: "secret"` tessera is allowed ‚ÄĒ refusing would push operators to raw SQL with zero validation, which is strictly worse. Redaction is structural, not advisory: `onInvalid` receives `{ rawDigest, length }` and never the raw; core substring-scrubs the raw value out of every validator and provider error message it surfaces (best-effort, and documented as such); `WriteFailedError` carries a scrubbed cause. And `bake` never resolves secret values, so a DB-stored secret cannot reach a client bundle by construction.

## What this is not

Four clauses keep database-backed config from mutating into a remote-config SDK:

1. **Pull-only.** No `watch`, no `subscribe`, no `onChange` ‚ÄĒ ever. The epoch is a number you compare, not an event you receive.
2. **Context-free reads.** The moment `unwrap()` takes a per-request or per-user context, it is a feature-flag SDK. We refuse.
3. **Uniform.** One value per process per tessera, never per-caller.
4. **Writes are operator-plane.** `set()` exists for admin panels, CLIs, and migrations. The litmus test: if `set()` appears in a request handler, config is being used as application state ‚ÄĒ use a database library.

## Testing staleness

The clock is a seam, so staleness tests need no timers:

