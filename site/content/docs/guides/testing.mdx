## The headline: zero `process.env` mutation

Most config testing pain comes from one root cause: the library reads the ambient environment at import time and caches the result at module scope. Testing then means mutating a global (`process.env`), resetting module registries, and praying your tests never run concurrently.

This library has no ambient anything. Values exist only inside explicit bindings — there is no global binding, no module-scope cache, no cache on the tessera object. So a test never mutates the environment. It constructs its own snapshot, hands it to the code under test, and throws it away. The composition-root discipline that makes production wiring explicit is exactly what makes tests trivial: config arrives through a parameter, and a parameter is the easiest thing in the world to fake — except here you don't even fake it, you build a real one.

## Literal snapshots with `literal()`

`literal(entry, value)` wires a tessera to an input-typed literal value, and a `bind` whose entries are all literals produces a pinned, immutable `Snapshot`. Both come from the core `tessellum` entry — and core `bind` supplies no natural sources, so an all-literal binding *cannot* touch `process.env`, the filesystem, or anything else ambient. That is a structural guarantee, not a convention. There is no separate testing entry point, either: pinning a value in a test is the same `literal` combinator production wiring uses.

Your app's descriptors are inert — zero I/O, zero validation at construction — so tests import the *real* tesserae and exercise the *real* schemas. No test-only duplicates to drift out of sync.

```ts
// app/config/db.ts — the app's real descriptors; tests import these as-is
import { tessera, defineTesserae } from "tessellum";
import { z } from "zod";

export const dbUrl = tessera({
  key: "DATABASE_URL",
  schema: z.string().url(),
  doc: "Primary Postgres connection string",
  // exposure defaults to "secret" — fail closed
});

export const poolSize = tessera({
  key: "DB_POOL_SIZE",
  schema: z.coerce.number().int().min(1).max(64),
  exposure: "public",
  default: 10,
});

export const dbTesserae = defineTesserae([dbUrl, poolSize]);
```

```ts
import { it, expect } from "vitest";
import { bind, literal } from "tessellum";
import { dbUrl, poolSize } from "../app/config/db";

it("reads typed values from a literal snapshot", () => {
  const cfg = bind({
    dbUrl: literal(dbUrl, "postgres://127.0.0.1:5432/app_test"),
    poolSize: literal(poolSize, 2),
  });

  expect(cfg.unwrap(dbUrl)).toBe("postgres://127.0.0.1:5432/app_test");
  expect(cfg.unwrap(poolSize)).toBe(2); // typed number, inferred from the schema
});
```

The result is an ordinary `Snapshot` — the binding type is computed from the entries, and all-literal entries yield the plainest one — with the same `unwrap`, `safe`, `pick`, and `explain` your production code sees. And it is still validated: `literal` values run the schema at `bind`, so a bad test value fails loudly with the same aggregated error any eager bind produces. A test snapshot can never hold a value the production path would refuse.

```ts
import { MissingConfigError } from "tessellum";

it("a bad test value fails loudly", () => {
  expect(() => bind({ dbUrl: literal(dbUrl, "not-a-url") })).toThrow(
    MissingConfigError,
  );
});
```

One rule, now enforced: `literal()` on a secret-exposure tessera is `SecretLiteralError` outside the test entry point — a live credential in committed wiring JSON would bypass every name-based guard. In tests, use `secretLiteral()` from the testing kit (same shape, test-scoped), and `fakeSecretProvider()` — a pre-certified in-memory `SecretProvider` double, the `manualClock` pattern — for testing secret wiring end to end. Placeholders only, never real values.

## Inject at the composition root

Factories take config as a parameter and never reach for the environment themselves. Production binds once at the root; tests build a snapshot per test. Same seam, two suppliers.

```ts
// app/db.ts — config arrives as a parameter, always
import type { Scope, ConfigOf } from "tessellum";
import { dbTesserae, dbUrl, poolSize } from "./config/db";

export function createPool(cfg: Scope<ConfigOf<typeof dbTesserae>>) {
  return new Pool({ connectionString: cfg.unwrap(dbUrl), max: cfg.unwrap(poolSize) });
}
```

```ts
// app/main.ts — the only place that touches the real environment
import { bind } from "tessellum/node";
import { dbTesserae } from "./config/db";
import { createPool } from "./db";

const root = bind(dbTesserae); // bare entries — declared refs resolve against process.env + files
const pool = createPool(root.pick(dbTesserae));
```

```ts
// db.test.ts — the same seam, fed by a literal snapshot
it("caps the pool at the configured size", () => {
  const cfg = bind({
    dbUrl: literal(dbUrl, "postgres://127.0.0.1:5432/app_test"),
    poolSize: literal(poolSize, 2),
  });

  const pool = createPool(cfg.pick(dbTesserae));
  expect(pool.options.max).toBe(2);
});
```

## Parallel tests need no isolation dance

Because there is no global state at all, two tests holding contradictory config run concurrently without stubbing, sandboxing, or serial execution. Each snapshot is its own world.

```ts
it.concurrent("small pool behaviour", () => {
  const cfg = bind({
    dbUrl: literal(dbUrl, "postgres://127.0.0.1:5432/a"),
    poolSize: literal(poolSize, 1),
  });
  expect(createPool(cfg.pick(dbTesserae)).options.max).toBe(1);
});

it.concurrent("large pool behaviour", () => {
  const cfg = bind({
    dbUrl: literal(dbUrl, "postgres://127.0.0.1:5432/b"),
    poolSize: literal(poolSize, 64),
  });
  expect(createPool(cfg.pick(dbTesserae)).options.max).toBe(64);
});
```

## Testing failure paths

### Missing config at bind: `MissingConfigError`

Eager `bind` validates the whole bound set up front and throws one aggregated error naming every missing or invalid tessera. Testing the boot-failure path needs no setup at all — absence is the natural state of a test process's environment.

```ts
import { MissingConfigError } from "tessellum";
import { bind } from "tessellum/node";

it("boot refuses with every problem listed at once", () => {
  try {
    bind(dbTesserae); // bare entries; the test env has no DATABASE_URL
    expect.unreachable("bind must throw");
  } catch (err) {
    expect(err).toBeInstanceOf(MissingConfigError);
    expect(String(err)).toContain("DATABASE_URL");
    // DB_POOL_SIZE is absent too, but its default (10) resolves — defaults
    // are validated like any sourced value, not a privileged path.
  }
});
```

### Present-but-invalid values: `safe`

`safe` returns `{ ok: true, value }` or `{ ok: false, error }` instead of throwing. To produce a present-but-invalid raw value without touching `process.env`, declare a file ref on a tessera pointed at a temp file, bind it as a bare entry under `tessellum/node`, and take the half-lazy door — `bind.lazy` checks presence eagerly but defers validator execution to the first read.

```ts
import { mkdtemp, writeFile } from "node:fs/promises";
import { tmpdir } from "node:os";
import { join } from "node:path";
import { tessera } from "tessellum";
import { bind } from "tessellum/node";
import { z } from "zod";

it("surfaces validation issues without throwing", async () => {
  const file = join(await mkdtemp(join(tmpdir(), "tesserae-")), "timeout");
  await writeFile(file, "soon"); // present, not a number

  const timeoutMs = tessera({
    key: "API_TIMEOUT_MS",
    schema: z.coerce.number().int().positive(),
    exposure: "public",
    sources: [{ file }],
  });

  const cfg = bind.lazy([timeoutMs]); // presence passes
  const result = cfg.safe(timeoutMs);

  expect(result.ok).toBe(false);
  if (!result.ok) expect(result.issues.length).toBeGreaterThan(0);
});
```

One property worth knowing when you snapshot-test error reports: for secret tesserae, issue reports carry counts and paths only — never validator messages or values, because BYO validators echo what they received and their text is untrusted. A failed secret validation can't leak the secret into your test snapshots or CI logs.

### Secrets refuse to serialize: `ExposureError`

Secrets are fail-closed all the way down to serialization. `Snapshot.toJSON` redacts secret values by construction, and a secret value itself throws if code tries to force it through `JSON.stringify` or `structuredClone` — the guard that closes SSR payload leaks. Both behaviors are directly testable.

```ts
import { ExposureError } from "tessellum";

it("secrets cannot ride a serialized payload", () => {
  const cfg = bind({
    dbUrl: literal(dbUrl, "postgres://127.0.0.1:5432/app_test"),
  });

  // the snapshot redacts secrets by construction
  expect(JSON.stringify(cfg)).not.toContain("app_test");

  // the secret value itself is guarded
  const url = cfg.unwrap(dbUrl);
  expect(() => JSON.stringify({ pageProps: { url } })).toThrow(ExposureError);
});
```

## Testing live config

`freshness: "live"` only means anything for re-readable refs — files and providers. That makes live behavior easy to test: point a tessera's file ref at a temp file you control and bind it as a bare entry under `tessellum/node`. Don't sleep past the poll throttle (`maxStalenessMs`, a `bind` option — roughly one second for files by default) — that's how flaky tests are born. `refresh(live)` forces one coherent re-read pass and returns a new pinned `Snapshot`, so the test is deterministic.

```ts
import { mkdtemp, writeFile } from "node:fs/promises";
import { tmpdir } from "node:os";
import { join } from "node:path";
import { tessera, refresh } from "tessellum";
import { bind } from "tessellum/node";
import { z } from "zod";

async function tempUpstream() {
  const file = join(await mkdtemp(join(tmpdir(), "tesserae-")), "upstream-url");
  await writeFile(file, "https://api.internal.test");
  const upstreamUrl = tessera({
    key: "UPSTREAM_URL",
    schema: z.string().url(),
    exposure: "public",
    sources: [{ file }],
    freshness: "live",
  });
  return { file, upstreamUrl };
}

it("bumps the epoch when an accepted value changes", async () => {
  const { file, upstreamUrl } = await tempUpstream();
  const live = bind([upstreamUrl]); // LiveBinding — a live tessera, no async entries
  const before = live.epoch(upstreamUrl);

  await writeFile(file, "https://api-blue.internal.test");
  const next = await refresh(live);

  expect(live.epoch(upstreamUrl)).toBe(before + 1);
  expect(next.unwrap(upstreamUrl)).toBe("https://api-blue.internal.test");
});
```

There is deliberately no watch or subscribe API — code that must react to change rebuilds clients keyed on the epoch, so the epoch is also what tests assert.

Invalid content after boot poisons the tessera — current truth wins, and the epoch bumps on the *transition* so consumers can observe it with `safe`:

```ts
import { InvalidLiveValueError, keepLastGood } from "tessellum";

it("poisons the tessera when the file goes bad", async () => {
  const { file, upstreamUrl } = await tempUpstream();
  const rejected: unknown[] = [];
  const live = bind([upstreamUrl], {
    onInvalid: (k, raw, issues) => rejected.push(issues),
  });
  const bootEpoch = live.epoch(upstreamUrl);

  await writeFile(file, "not a url");
  await refresh(live).catch(() => {});

  expect(() => live.unwrap(upstreamUrl)).toThrow(InvalidLiveValueError);
  expect(live.safe(upstreamUrl).ok).toBe(false);          // non-throwing form
  expect(live.epoch(upstreamUrl)).toBeGreaterThan(bootEpoch); // transition bumps
  expect(rejected).toHaveLength(1);                          // once per distinct raw
});

it("keepLastGood is the explicit opt-in, bounded by maxCoastMs", async () => {
  const { file, upstreamUrl } = await tempUpstream();
  const live = bind([upstreamUrl], {
    invalid: [keepLastGood(upstreamUrl, { maxCoastMs: 60_000 })],
  });

  await writeFile(file, "not a url");
  await refresh(live).catch(() => {});

  expect(live.unwrap(upstreamUrl)).toBe("https://api.internal.test"); // coasting
  expect(live.health()[upstreamUrl.key].status).toBe("coasting");
});
```

A live tessera that has *never* held a valid value is a different, named failure. Eager `bind` refuses at boot with `MissingConfigError`; if you took the half-lazy door, the first read is where truth arrives:

```ts
import { NoValidValueError } from "tessellum";

it("a never-yet-valid live tessera throws NoValidValueError", async () => {
  const { file, upstreamUrl } = await tempUpstream();
  await writeFile(file, "not a url"); // invalid from the start

  const live = bind.lazy([upstreamUrl]);
  expect(() => live.unwrap(upstreamUrl)).toThrow(NoValidValueError);
});
```

## Testing library code that takes a `Scope`

A library ships its tesserae as inert descriptors and types its factory against `Scope<ConfigOf<typeof itsTesserae>>`. Testing it is the same move the app makes at its composition root: build an all-literal binding, `pick`, pass it in. An all-literal `bind` returns a real `Snapshot`, so `pick` works on it directly.

```ts
import { bind, literal, defineTesserae, NotInScopeError } from "tessellum";
import { createMailer, mailerTesserae, smtpUrl, smtpFrom } from "@acme/mailer";

it("wires the mailer from a picked scope", () => {
  const cfg = bind({
    smtpUrl: literal(smtpUrl, "smtp://127.0.0.1:1025"),
    smtpFrom: literal(smtpFrom, "noreply@test.local"),
  });

  const mailer = createMailer(cfg.pick(mailerTesserae));
  // createMailer(cfg) does not compile: the root Snapshot is deliberately
  // not assignable to Scope — the pick() call is the audit point.
});

it("a scope is runtime-restricted, not just typed", () => {
  const cfg = bind({
    smtpUrl: literal(smtpUrl, "smtp://127.0.0.1:1025"),
    smtpFrom: literal(smtpFrom, "noreply@test.local"),
  });

  const scope = cfg.pick(defineTesserae([smtpUrl]));
  expect(() => scope.unwrap(smtpFrom)).toThrow(NotInScopeError);
});
```

Honest boundary, in tests as in production: `pick` is least-privilege for mistakes and auditability, not a sandbox — hostile in-process code can read `process.env` itself.

## Anti-pattern gallery

Every pattern below exists to work around ambient config state. None of them is needed here, and each one quietly costs you something.

```ts
// Anti-pattern: vi.stubEnv
vi.stubEnv("DATABASE_URL", "postgres://127.0.0.1:5432/app_test");
```

`stubEnv` mutates a process-wide global: it races under `it.concurrent`, leaks into the next test if cleanup is forgotten, and carries untyped, unvalidated strings. Here it is also pointless — nothing reads the environment unless you bound under an entry point whose natural defaults include it (`tessellum/node`), and the core `bind` these tests use supplies no natural sources at all.

```ts
// Anti-pattern: mutating process.env by hand
process.env.DB_POOL_SIZE = "64";
delete process.env.DATABASE_URL;
```

Same problems without even the automatic cleanup. If a test needs this, the code under test is reaching for ambient state instead of accepting a `Scope` — fix the factory, not the environment.

```ts
// Anti-pattern: module-mocking a "config module"
vi.mock("../config", () => ({
  config: { dbUrl: "postgres://127.0.0.1:5432/app_test", poolSize: "64" },
}));
```

This mocks a shape, not a contract: validation is skipped entirely (`"64"` where a number belongs, and nobody notices), the mock drifts silently when the real module refactors, and hoisting rules make failures cryptic. In this design there is also nothing to intercept — there is no central all-tesserae module (groups are declared near their call sites) and no ambient binding hiding behind an import.

```ts
// Instead: a real, validated snapshot per test
const cfg = bind({
  dbUrl: literal(dbUrl, "postgres://127.0.0.1:5432/app_test"),
  poolSize: literal(poolSize, 2),
});
const pool = createPool(cfg.pick(dbTesserae));
```

If a test is hard to write, that's the design telling you something: somewhere, code is reading configuration instead of receiving it. Move the read to the composition root and the test writes itself.
