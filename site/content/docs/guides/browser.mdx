## The rule

In the browser, config arrives exactly one way — the module `komirka bake` generates:

```ts
import { config } from "./config.baked";
```

That is the entire client-side surface. There is no runtime `bind()` path for secrets, no `VITE_`-prefix convention, no `import.meta.env` reads, no ambient global. The generated module contains public literal values plus a small inlined shim; it imports nothing at runtime, so a baked SPA carries **zero runtime dependency** on the library.

The rest of this page walks a Vite SPA through the pipeline, then covers the three failure modes you should see once each on purpose: skipping bake, listing a secret in the client target, and upgrading a dependency after baking.

## Walkthrough: a Vite SPA

### 1. Declare zernos near their call sites

A zerno is an inert descriptor — zero I/O, zero validation at construction — so importing one costs nothing. Declare them next to the code that reads them, not in one central all-zernos module:

```ts
// src/config.zernos.ts
import { zerno, defineZernos } from "komirka";
import { z } from "zod";

export const apiBaseUrl = zerno({
  key: "API_BASE_URL",
  schema: z.string().url(),
  doc: "Origin the SPA calls for its API",
  exposure: "public", // exposure defaults to "secret" — mark client values public explicitly
});

export const releaseChannel = zerno({
  key: "RELEASE_CHANNEL",
  schema: z.enum(["stable", "beta"]),
  doc: "Release ring this build belongs to",
  exposure: "public",
  default: "stable", // JSON default; validated like any sourced value — no privileged path
});

export const clientZernos = defineZernos([apiBaseUrl, releaseChannel]);
```

Neither zerno declares any refs beyond its `key`, so each carries the default declared env ref named by its key — bake resolves `API_BASE_URL` and `RELEASE_CHANNEL` from the **build** environment.

### 2. Declare the client target

```ts
// config.manifest.ts — must be side-effect-free; bake runs it in a constrained
// context, and because zerno construction is inert, I/O here is detectable.
import { clientZernos } from "./src/config.zernos";

export default {
  targets: {
    client: [...clientZernos],
  },
};
```

Listing a zerno under `targets.client` is an **app attestation**. A package author declaring `exposure: "public"` is necessary but not sufficient — env-backed values are deny-by-default for the client, and nothing reaches a browser bundle unless the app names it here. This replaces Vite's `VITE_` prefix convention entirely: the manifest is the allowlist, and nothing in client code reads `import.meta.env` at all.

A `server` target is optional. Mandatory-ness attaches to the trust boundary: untrusted client targets must be baked; a trusted server may runtime-bind with `bind()` from `komirka/node` instead.

### 3. Run bake

```bash
$ komirka bake
✓ target "client" → src/config.baked.ts   (2 public values, validated)
✓ .env.example                             (generated from key + doc fields)
✓ CONFIG.md
✓ manifest.json
✓ leak scan: 0 findings

$ komirka bake --check   # CI drift gate — fails if the module is missing or stale
```

Bake resolves each listed zerno's declared refs in the build environment, runs your schema, and inlines the validated literal. A missing or invalid public value fails the bake with one aggregated report — the same eager, list-everything contract `bind()` gives you at runtime.

For local development, run bake in watch mode; the generated file is an ordinary module, so Vite HMR reloads it like any other:

```bash
# terminal 1
$ komirka bake --watch

# terminal 2
$ vite dev
```

### 4. What the generated module is

Abridged, but this is the whole idea:

```ts
// src/config.baked.ts — GENERATED by `komirka bake`. Do not edit.
import type { Scope } from "komirka"; // type-only: erased at build; nothing is imported at runtime

// 1. PUBLIC literal values, keyed by zerno identity hash:
const VALUES = {
  "zernov1:xxh128:1f2ab3…": "https://api.example.com", // API_BASE_URL
  "zernov1:xxh128:8bd04c…": "stable",                  // RELEASE_CHANNEL
};

// 2. An inlined ~30-line snapshot shim. Its unwrap() fetches nothing and
//    computes nothing — it checks the caller's descriptor against the
//    identity hash recorded at bake time:
//      hash matches                    -> return the baked literal, typed
//      key known, hash DIFFERENT       -> throw StaleBakeError
//      key not part of this target     -> throw NotInScopeError
export const config: Scope<{
  API_BASE_URL: string;
  RELEASE_CHANNEL: "stable" | "beta";
}> = /* …inlined shim over VALUES… */;
```

Three properties matter. It contains **public literals only** — bake never resolves secret values, so nothing secret exists to leak. It **imports nothing at runtime** — the shim is inlined, keeping the artifact zero-dependency. And it **carries the `Scope` brand**, so any library API typed `Scope<ConfigOf<typeof someZernos>>` accepts it exactly like a `pick()`-produced scope. Bake also strips all wiring metadata from the client output — a pinned snapshot needs no refs, so the generated module discloses nothing about your env-name topology.

### 5. Use it

```ts
// src/api/client.ts
import { config } from "../config.baked";
import { apiBaseUrl, releaseChannel } from "../config.zernos";

export function apiFetch(path: string) {
  return fetch(new URL(path, config.unwrap(apiBaseUrl)), {
    headers: { "x-release-channel": config.unwrap(releaseChannel) },
  });
}
```

`config.unwrap(apiBaseUrl)` is a typed, synchronous read of a build-time constant. Asking for a zerno outside the client target throws `NotInScopeError` — the baked scope is runtime-restricted, same as any `pick()`.

In tests, build the same shape without touching `process.env`:

```ts
import { bind, literal } from "komirka";
import { clientZernos, apiBaseUrl, releaseChannel } from "../src/config.zernos";

const cfg = bind({
  apiBaseUrl: literal(apiBaseUrl, "https://api.test.local"),
  releaseChannel: literal(releaseChannel, "beta"),
}).pick(clientZernos); // validation runs at bind; pass it wherever a Scope is expected
```

## Skipping bake

If the manifest declares a client target and you build without baking, the build fails — the generated module simply does not exist:

```bash
$ vite build
✗ [ERROR] Could not resolve "./config.baked"
    src/api/client.ts:1:23

$ komirka bake --check
✗ target "client" is declared in config.manifest.ts but no generated module exists.
  Run `komirka bake` (dev: `komirka bake --watch`).
```

This is deliberate: there is no runtime fallback the browser could silently degrade to. A declared client target either bakes or the build dies.

## Putting a secret in the client target

`exposure` defaults to `"secret"`, so the common mistake is listing a zerno you never marked public:

```ts
// config.manifest.ts
export default {
  targets: {
    client: [...clientZernos, apiToken], // ✗ apiToken is exposure: "secret"
  },
};
```

```bash
$ komirka bake
✗ ExposureError: secret zerno "API_TOKEN" is listed in client target "client".
  Secrets never enter client artifacts; the value was not resolved.
  If this value is genuinely public, set exposure: "public" on the zerno
  AND keep it listed here (both declarations are required).
exit status 1
```

The refusal report contains key names only. Bake never resolves secret values for any target — the server target emits key names and dynamic lookups, never values — so there is nothing to redact and nothing for CI to hold.

## Why there is no runtime bind() for browser secrets

The guarantee is structural, not procedural. The universal `komirka` entry point ships a `bind` with **no natural sources by construction**: a bare env or file ref under it has nothing to resolve against and throws `NoWiringError`. The wiring that can actually reach an environment lives elsewhere — `komirka/node`'s `bind` supplies `process.env` and the filesystem as natural defaults, `komirka/workers` only reads the per-request bag you pass explicitly as `bind(entries, { env: envBag })`, and provider-backed entries require `bind` from `komirka/async` — and none of those resolve under the browser export condition. In a browser bundle there is nothing you could hand to `bind()` that can read your deployment's secrets — and reading a secret zerno from a browser context throws `ExposureError` regardless.

Baking is one layer of a defense stack, and no single mechanism is claimed sufficient on its own:

1. Required `targets` declaration — a client target that exists must be baked.
2. Generated-module-only delivery — client code has exactly one config import.
3. No-natural-sources browser entry — the structural guarantee above.
4. An AST lint banning raw `process.env` and `import.meta.env` reads in client-designated code.
5. A coverage report comparing the manifest against client-reachable zernos.
6. The allowlist leak scan: any high-entropy string in emitted client assets — including sourcemaps and decompressed `.gz` / `.br` — that is not a registered public value gets flagged. Findings never print values.

Honest boundaries: the scan is entropy- and length-gated, so a short value like `"8080"` is unscannable — bake warns explicitly and you rely on the structural controls for it. And SSR request-time payloads are outside build-time scope; those are closed by the runtime serialization guard on snapshots, not by the scan.

## Baked means constant-per-build

A baked value is a constant of the build artifact. That is the contract, stated plainly: if a value can change after the build, it is not baked client config. This is the browser-side edition of the core rule — if it can change while the process runs, it's state, not config — and it is what makes the shim so small: no fetching, no polling, no cache invalidation, just literals.

### Open decision: build once, promote many

The constant-per-build contract has a real cost we are not papering over. If your pipeline builds one artifact and promotes it through environments — staging, then prod — an env-varying public value like `API_BASE_URL` is a genuine problem: bake froze whatever the build environment held. Today you have two honest options:

- **Re-bake per environment** in the promote step. Only public values are resolved, so no secrets are needed in the pipeline, but you give up bit-identical artifacts.
- **Keep env-varying values out of the client target** and have the SPA fetch them from your own API at boot. That is userland, and it reintroduces a network read the baked path exists to avoid.

A runtime-injected JSON mode — the shim reading a small, per-environment JSON document deployed next to the artifact instead of inlined literals — is under consideration but not designed. It is an open decision, not a promised feature.

## Upgrading a dependency after baking: StaleBakeError

Zerno identity is content-addressed. If `@acme/widgets` ships a zerno and a new package version changes its schema (bumping `version`), its identity hash changes — and the baked module still holds the old one:

```ts
import { config } from "./config.baked";
import { widgetTheme } from "@acme/widgets/zernos";

config.unwrap(widgetTheme);
// StaleBakeError: "WIDGET_THEME" was baked under identity
//   zernov1:xxh128:8bd04c…
// but the descriptor passed to unwrap() has identity
//   zernov1:xxh128:44c19e…
// (@acme/widgets was upgraded after `komirka bake` ran)
```

The fix is one command — `komirka bake` — and the reason you rarely see this in production is one CI line: `komirka bake --check` fails on exactly this drift before it ships. Note the distinction from `NotInScopeError`: an unknown key was never part of the target; a known key under a different descriptor is a stale bake.
