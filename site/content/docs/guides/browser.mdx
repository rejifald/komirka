## The rule

In the browser, config arrives exactly one way — the module `tessellum bake` generates:

```ts
import { config } from "./config.baked";
```

That is the entire client-side surface. There is no runtime `bind()` path for secrets, no `VITE_`-prefix convention, no `import.meta.env` reads, no ambient global. The generated module contains public literal values plus a small inlined shim; it imports nothing at runtime, so a baked SPA carries **zero runtime dependency** on the library.

The rest of this page walks a Vite SPA through the pipeline, then covers the three failure modes you should see once each on purpose: skipping bake, listing a secret in the client target, and upgrading a dependency after baking.

## Walkthrough: a Vite SPA

### 1. Declare tesserae near their call sites

A tessera is an inert descriptor — zero I/O, zero validation at construction — so importing one costs nothing. Declare them next to the code that reads them, not in one central all-tesserae module:

```ts
// src/config.tesserae.ts
import { tessera, defineTesserae } from "tessellum";
import { z } from "zod";

export const apiBaseUrl = tessera({
  key: "API_BASE_URL",
  schema: z.string().url(),
  doc: "Origin the SPA calls for its API",
  exposure: "public", // exposure defaults to "secret" — mark client values public explicitly
});

export const releaseChannel = tessera({
  key: "RELEASE_CHANNEL",
  schema: z.enum(["stable", "beta"]),
  doc: "Release ring this build belongs to",
  exposure: "public",
  default: "stable", // JSON default; validated like any sourced value — no privileged path
});

export const clientTesserae = defineTesserae([apiBaseUrl, releaseChannel]);
```

Neither tessera declares any refs beyond its `key`, so each carries the default declared env ref named by its key — bake resolves `API_BASE_URL` and `RELEASE_CHANNEL` from the **build** environment.

### 2. Declare the client target

```ts
// config.manifest.ts — must be side-effect-free; bake runs it in a constrained
// context, and because tessera construction is inert, I/O here is detectable.
import { clientTesserae } from "./src/config.tesserae";

export default {
  targets: {
    client: [...clientTesserae],
  },
};
```

Listing a tessera under `targets.client` is an **app attestation**. A package author declaring `exposure: "public"` is necessary but not sufficient — env-backed values are deny-by-default for the client, and nothing reaches a browser bundle unless the app names it here. This replaces Vite's `VITE_` prefix convention entirely: the manifest is the allowlist, and nothing in client code reads `import.meta.env` at all.

A `server` target is optional. Mandatory-ness attaches to the trust boundary: untrusted client targets must be baked; a trusted server may runtime-bind with `bind()` from `tessellum/node` instead.

### 3. Run bake

```bash
$ tessellum bake
✓ target "client" → src/config.baked.ts   (2 public values, validated)
✓ .env.example                             (generated from key + doc fields)
✓ CONFIG.md
✓ manifest.json
✓ leak scan: 0 findings

$ tessellum bake --check   # CI drift gate — fails if the module is missing or stale
```

Bake resolves each listed tessera's declared refs in the build environment, runs your schema, and inlines the validated literal. A missing or invalid public value fails the bake with one aggregated report — the same eager, list-everything contract `bind()` gives you at runtime.

For local development, run bake in watch mode; the generated file is an ordinary module, so Vite HMR reloads it like any other:

```bash
# terminal 1
$ tessellum bake --watch

# terminal 2
$ vite dev
```

### 4. What the generated module is

Abridged, but this is the whole idea:

```ts
// src/config.baked.ts — GENERATED by `tessellum bake`. Do not edit.
import type { Tessera, Scope } from "tessellum"; // type-only: erased at build; nothing imported at runtime

// 1. PUBLIC literal values:
const VALUES = {
  API_BASE_URL: "https://api.example.com",
  RELEASE_CHANNEL: "stable",
};

// 2. Lightweight HANDLES — key + identity hash + a phantom Tessera<V> type, but
//    NO schema. Client code imports THESE for the unwrap() key, so no descriptor
//    (and therefore no zod/schema) ever reaches the browser bundle.
export const apiBaseUrl = /* handle */ {} as Tessera<string>;
export const releaseChannel = /* handle */ {} as Tessera<"stable" | "beta">;

// 3. An inlined ~30-line shim. unwrap(handle) returns the baked literal by key;
//    a handle outside this target throws NotInScopeError. Staleness is caught at
//    BUILD time by `tessellum bake --check` (see below), not at runtime — a runtime
//    check would need the real descriptor, which would reship the schema.
export const config: Scope<{
  API_BASE_URL: string;
  RELEASE_CHANNEL: "stable" | "beta";
}> = /* …inlined shim over VALUES… */;
```

Three properties matter. It contains **public literals only** — bake never resolves secret values, so nothing secret exists to leak. It **imports nothing at runtime and exports its own handles**, so client code never imports a descriptor — the artifact is genuinely zero-dependency (measured: ~0.4 KB gzip; importing a real descriptor for the `unwrap()` key instead drags in ~65 KB of `zod`). And it **carries the `Scope` brand**, so any library API typed `Scope<ConfigOf<typeof someTesserae>>` accepts it exactly like a `pick()`-produced scope — the phantom `Tessera<V>` types on the handles flow through `import type`, erased at build. Bake also strips all wiring metadata from the client output — a pinned snapshot needs no refs, so the generated module discloses nothing about your env-name topology.

### 5. Use it

```ts
// src/api/client.ts
import { config, apiBaseUrl, releaseChannel } from "../config.baked";

export function apiFetch(path: string) {
  return fetch(new URL(path, config.unwrap(apiBaseUrl)), {
    headers: { "x-release-channel": config.unwrap(releaseChannel) },
  });
}
```

The handles come from `../config.baked`, **not** from `../config.tesserae` — importing the real descriptor for the `unwrap()` key is exactly what would pull `zod` into the bundle, and the client-graph lint (no `*.tesserae.ts` reachable from a client entry) fails the build if you do. `config.unwrap(apiBaseUrl)` is a typed, synchronous read of a build-time constant. Asking for a tessera outside the client target throws `NotInScopeError` — the baked scope is runtime-restricted, same as any `pick()`.

In tests, build the same shape without touching `process.env`:

```ts
import { bind, literal } from "tessellum";
import { clientTesserae, apiBaseUrl, releaseChannel } from "../src/config.tesserae";

const cfg = bind({
  apiBaseUrl: literal(apiBaseUrl, "https://api.test.local"),
  releaseChannel: literal(releaseChannel, "beta"),
}).pick(clientTesserae); // validation runs at bind; pass it wherever a Scope is expected
```

## Skipping bake

If the manifest declares a client target and you build without baking, the build fails — the generated module simply does not exist:

```bash
$ vite build
✗ [ERROR] Could not resolve "./config.baked"
    src/api/client.ts:1:23

$ tessellum bake --check
✗ target "client" is declared in config.manifest.ts but no generated module exists.
  Run `tessellum bake` (dev: `tessellum bake --watch`).
```

This is deliberate: there is no runtime fallback the browser could silently degrade to. A declared client target either bakes or the build dies.

## Putting a secret in the client target

`exposure` defaults to `"secret"`, so the common mistake is listing a tessera you never marked public:

```ts
// config.manifest.ts
export default {
  targets: {
    client: [...clientTesserae, apiToken], // ✗ apiToken is exposure: "secret"
  },
};
```

```bash
$ tessellum bake
✗ ExposureError: secret tessera "API_TOKEN" is listed in client target "client".
  Secrets never enter client artifacts; the value was not resolved.
  If this value is genuinely public, set exposure: "public" on the tessera
  AND keep it listed here (both declarations are required).
exit status 1
```

The refusal report contains key names only. Bake never resolves secret values for any target — the server target emits key names and dynamic lookups, never values — so there is nothing to redact and nothing for CI to hold.

## Why there is no runtime bind() for browser secrets

The guarantee is structural, not procedural. The universal `tessellum` entry point ships a `bind` with **no natural sources by construction**: a bare env or file ref under it has nothing to resolve against and throws `NoWiringError`. The wiring that can actually reach an environment lives elsewhere — `tessellum/node`'s `bind` supplies `process.env` and the filesystem as natural defaults, `tessellum/workers` only reads the per-request bag you pass explicitly as `bind(entries, { env: envBag })`, and provider-backed entries require `bind` from `tessellum/async` — and none of those resolve under the browser export condition. In a browser bundle there is nothing you could hand to `bind()` that can read your deployment's secrets — and reading a secret tessera from a browser context throws `ExposureError` regardless.

Baking is one layer of a defense stack, and no single mechanism is claimed sufficient on its own:

1. Required `targets` declaration — a client target that exists must be baked.
2. Generated-module-only delivery — client code has exactly one config import.
3. No-natural-sources browser entry — the structural guarantee above.
4. An AST lint banning raw `process.env` and `import.meta.env` reads in client-designated code.
5. A coverage report comparing the manifest against client-reachable tesserae.
6. The allowlist leak scan: any high-entropy string in emitted client assets — including sourcemaps and decompressed `.gz` / `.br` — that is not a registered public value gets flagged. Findings never print values.

Honest boundaries: the scan is entropy- and length-gated, so a short value like `"8080"` is unscannable — bake warns explicitly and you rely on the structural controls for it. And SSR request-time payloads are outside build-time scope; those are closed by the runtime serialization guard on snapshots, not by the scan.

## Baked means constant-per-build

A baked value is a constant of the build artifact. That is the contract, stated plainly: if a value can change after the build, it is not baked client config. This is the browser-side edition of the core rule — if it can change while the process runs, it's state, not config — and it is what makes the shim so small: no fetching, no polling, no cache invalidation, just literals.

### Build once, promote many

The constant-per-build contract has a real cost, and it has a decided answer ([ADR 0008](https://github.com/rejifald/tessellum/blob/main/docs/adr/0008-env-varying-public-values.md)). If your pipeline builds one artifact and promotes it through environments — staging, then prod — an env-varying public value like `API_BASE_URL` is a genuine problem: bake froze whatever the build environment held. There are two supported paths, and the trade-off between them is explicit.

**The default: re-bake per environment.** Run `tessellum bake` once per environment in the promote step. Only public values are resolved, so no secrets are needed in the pipeline. The cost, stated plainly: you give up bit-identical artifacts. For teams that can rebuild, this is the simplest correct answer and needs no new machinery.

**The escape: a runtime-injected public document.** For pipelines that must promote one bit-identical artifact, deploy a small **public** JSON document — conventionally `/config.json` — next to the artifact, one per environment. The client fetches it **once at boot** through an async provider entry and pins; every read after that is synchronous.

```ts
import { tessera, provider } from "tessellum"; // descriptor + wiring data (pure, universal)
import { bind } from "tessellum/async";        // the async door: provider entries
import { z } from "zod";

const apiBase = tessera({
  key: "API_BASE_URL",
  schema: z.string().url(),
  exposure: "public", // public, or the bind is an ExposureError
  sources: [{ provider: "runtime-config" }, { env: "API_BASE_URL" }],
});

// runtimeConfigProvider is a userland provider that fetches /config.json exactly once
const cfg = await bind({ apiBase: provider(apiBase, runtimeConfigProvider) }).snapshot();
// one await at boot; every read after this line is synchronous
```

```json
{ "API_BASE_URL": "https://api.example.com", "CDN_ORIGIN": "https://cdn.example.com" }
```

Four properties make this a sanctioned mode rather than a hole in the model:

- **Values still validate.** The document flows through the same tessera schemas at boot, so a bad deploy is one aggregated `MissingConfigError` — not a mystery at first use.
- **Secrets are structurally impossible.** The browser entry ships no secret-capable sources, so an `exposure: "secret"` tessera pointed at the document fails with `ExposureError`. The document cannot become a secret channel; that is construction, not policy.
- **It is an untrusted input boundary.** Injected browser config is revalidated on read and never used for secret-tier values. Whoever can serve `/config.json` can serve a *valid-but-wrong* URL — the same trust you already place in whoever serves your JS bundle.
- **It never touches `StaleBakeError`.** Runtime-injected tesserae are not in the `client` bake target: never inlined, no baked handle, no identity hash. `StaleBakeError` stays exactly what it is — a build-time `bake --check` gate over baked handles.

Bake emits a **JSON Schema** for the document, so your deploy pipeline can validate `/config.json` *before serving it* rather than discovering the error in someone's browser.

**Variant: HTML injection.** Your server can template the same JSON into a script tag and have the provider read it from the DOM instead of fetching. It saves a request and couples you to a serving layer. Same schemas, same `ExposureError` guard, same untrusted-input rule — one document shape, two transports.

The honest cost of the escape, in one line: it reintroduces a network read on the boot path that the baked path exists to avoid, and it makes boot depend on that document being served. That is why re-baking is the default.

## Upgrading a dependency after baking: StaleBakeError

Tessera identity is content-addressed. If `@acme/widgets` ships a tessera and a new package version changes its schema (bumping `version`), its identity hash changes — and a baked module generated against the old one is now stale. **The production guarantee is at build time:** `tessellum bake --check` re-derives every artifact and fails CI on exactly this drift before it ships. It is the load-bearing gate — the client imports handles, not the upgraded descriptor, so there is no runtime descriptor to compare against in prod.

**In dev you also get an automatic runtime check.** The generated module carries a dev-only hash check — change a dependency's schema, keep the dev server running, and the next read throws so you fix it locally instead of in CI:

```ts
config.unwrap(widgetTheme);
// (dev only) StaleBakeError: "WIDGET_THEME" was baked under identity
//   tesserav1:xxh128:8bd04c… but the current descriptor is tesserav1:xxh128:44c19e…
//   — @acme/widgets changed after `tessellum bake` ran. Re-run it.
```

That check is **stripped from production builds** entirely: it reaches the real descriptor only through a dynamic import in a `NODE_ENV`-gated branch, so a prod bundle drops it — and zod with it (a static import would silently re-ship the schema). The fix is always one command — `tessellum bake`. Note the distinction from `NotInScopeError`: an unknown key was never part of the target; a known key under a different descriptor is a stale bake. The honest cost: in production a stale bake that slips past `tessellum bake --check` reaches users as a wrong or missing value, not a `StaleBakeError` — so keep `bake --check` required in CI. See [ADR 0003](https://github.com/rejifald/tessellum/blob/main/docs/adr/0003-client-read-api-and-bake-staleness.md).
