A configuration library earns trust as much by what it refuses to do as by what it does. Every exclusion below was adversarially litigated during design, and every one protects a gate the library refuses to trade away: browser-first, bundle-frugal, contract-not-dependency, no background machinery. The scope line is a feature. Each entry states what is out, why, and what to reach for instead.

One sentence decides half of them:

> If it can change while the process runs, it is state, not config.

The single carve-out is explicitly declared `freshness: "live"` on re-readable sources — files and providers — and even that is poll-on-read with no background loops.

## Feature flags, targeting, subscriptions

Percentage rollouts, per-user targeting, kill switches keyed on session attributes — these change per request, per user, mid-process. By the litmus above, that is state, and state belongs to a flag service with its own delivery pipeline. A flag SDK runs perfectly well *next to* your config; it does not belong *inside* it.

There is also a mechanical reason: core runs no background loops, timers, or watchers, ever. A subscription API implies a push channel, and a push channel implies machinery that keeps a process awake. So there is no `subscribe()`, no `onChange` callback — by design. What exists instead is a monotonic epoch and per-request pinning:

```ts
import { tessera } from "tessellum";
import { bind } from "tessellum/node";
import { z } from "zod";

// In scope: an operational dial that ops flips by updating a mounted file.
const maintenanceMode = tessera({
  key: "MAINTENANCE_MODE",
  schema: z.enum(["on", "off"]),
  exposure: "public",
  sources: [{ file: "/etc/app/maintenance" }],
  freshness: "live",
  doc: "Serve the maintenance page instead of the app",
});

const live = bind([maintenanceMode]); // bare entry — the declared file ref resolves against node's natural defaults

app.use((req, res, next) => {
  const pinned = live.snapshot(); // per-request pin — the documented pattern
  if (pinned.unwrap(maintenanceMode) === "on") return res.status(503).end();
  next();
});
```

A slow-moving operational dial in a ConfigMap-mounted file is config. "Show the new checkout to 10% of EU users" is not.

## .env parsing in core

Core does not read `.env` files, and never grows a dotenv dependency. The runtimes already own this job — and one of them actively fights libraries that try to do it too. Bun loads `.env` natively, so a library that self-loads produces double-loading conflicts and precedence fights. Node has had this built in for a while:

```bash
# The runtime already owns .env loading:
node --env-file=.env --env-file=.env.local server.js

# Bun loads .env natively — a self-loading library just gets in its way:
bun run server.ts
```

`bind` reads whatever the runtime placed in the environment — on the node entry, `process.env` is the natural default for bare entries; how it got there is not core's business:

```ts
import { bind } from "tessellum/node";

// Works identically whether env came from --env-file, Bun, your PaaS,
// or dotenv preloaded by your own bootstrap.
const cfg = bind(appTesserae);
```

If you want dotenv semantics that a runtime does not provide, dotenv is a userland provider impl — `{ id: "dotenv", color: "sync", read }` is a few lines you own — or, simpler still, a bootstrap line that populates `process.env` before `bind` runs. Note that the `.env` file format is still a first-class *output*: `tessellum bake` generates `.env.example` from your descriptors' `key` and `doc` fields. Core writes the template; it never parses the file.

## Secret-manager clients

Core ships no AWS SSM, Secrets Manager, Vault, or GCP client. Those SDKs are heavy, churny, and platform-specific — any one of them would end the zero-dependency budget and the browser-safe universal entry in a single `import`. Provider impls are plain objects; adapters are community packages:

```ts
import { bind } from "tessellum/async";
import { provider } from "tessellum";
import { ssm } from "@acme/tessellum-ssm"; // community provider impl — not part of core

const cfg = await bind({
  dbUrl,                                              // bare entry: process.env, as usual
  apiKey: provider(apiKey, ssm, "/prod/app/API_KEY"), // async-colored entry
}).snapshot();

cfg.unwrap(apiKey); // resolved once at the pin — every read after this is sync, forever
```

An impl declares `{ id: "ssm", color: "async", read }` — the `id` is what the lockfile records (anonymous impls are a bind-time error on bake-covered targets), and an untyped `color` is a compile error.

The async boundary is enforced, not advisory — and it lives in the types. One async link colors the whole binding an `AsyncBinding`, so an async source cannot sneak into the sync door:

```ts
import { bind } from "tessellum/node";

bind({ apiKey: provider(apiKey, ssm, "/prod/app/API_KEY") });
// Compile error — an async-colored entry needs the `tessellum/async` bind and an
// awaited pin. There is no spelling that hands an async read a sync face.
```

## Sealed boxes, TTL, rotation, audit trails

Excluded from v1, for three litigated reasons.

**Scope creep.** TTLs, rotation schedules, and audit logs are the feature set of a secrets manager. Building a worse one inside a config library helps nobody; a real one already exists in your stack.

**Boxes are ceremony.** Every downstream SDK takes strings — `new Pool({ connectionString })`, `new Stripe(apiKey)`. A hypothetical `Sealed` wrapper would be unwrapped at every single call site, which means the "protection" evaporates exactly where the value is used. What actually protects secrets here is structural: `exposure: "secret"` is the fail-closed default, the browser entry has no secret-capable sources by construction, bake never resolves secret values, and the serialization guard closes the SSR-payload leak that boxes only pretended to:

```ts
cfg.explain(apiKey);              // provenance with the value redacted
cfg.toJSON();                     // secrets redacted by construction
structuredClone(cfg.unwrap(apiKey)); // throws — serialization guard, no box required
```

**Rotation cannot be sync-read-safe.** A token that expires mid-process would force `unwrap()` to block, go async, or hand out a stale credential — each of which breaks the contract that all reads are sync once a snapshot is pinned. A credential that rotates every 15 minutes fails the litmus: it is state. Fetch it with your vault's SDK. For config that legitimately re-resolves, rotation is spelled honestly as a deliberate act:

```ts
// Re-resolve at a moment you choose — not on a timer core hides from you:
const next = await refresh(live); // new pinned Snapshot; all-or-nothing on partial failure
```

## Environment-cascade DSL

There is no per-tessera environment map, no `NODE_ENV` switch inside descriptors, no cascade file format. Environments are just different sources and different overrides — and ordered `sources` already states precedence in the open, where a cascade DSL would hide it:

```ts
const logLevel = tessera({
  key: "LOG_LEVEL",
  schema: z.enum(["debug", "info", "warn", "error"]),
  exposure: "public",
  // The cascade, spelled once, in order:
  sources: [{ env: "LOG_LEVEL" }, { file: "/etc/app/log-level" }, { literal: "info" }],
});
```

Staging and production run the *same descriptor* against different environment contents. Nothing in the descriptor knows which environment it is in — which is precisely what keeps `identityOf(logLevel)` stable across every deploy target, and keeps descriptors JSON-round-trippable instead of mini-programs.

## Runtime registry

No global `provide()`, no ambient binding, no side-effectful registration at import time. Ever. Three reasons:

- **Tree-shaking dies.** A registry entangles every tessera with every other; importing one descriptor must cost nothing, because libraries ship their tesserae as inert, zero-cost imports.
- **Import order becomes load-bearing.** Registration-at-import is exactly the side-effect bug class the inert-descriptor rule exists to kill.
- **DI honesty.** With no registry, a dependency structurally *cannot* redirect where your secrets come from at import time. Wiring happens once, at your composition root, in code you review.

Reads resolve against an explicit binding — never an ambient lookup:

```ts
import { bind } from "tessellum/node";

const cfg = bind([dbUrl]);

cfg.unwrap(dbUrl);    // bound here — fine
cfg.unwrap(poolSize); // UnboundTesseraError — not in this binding, and no registry to fall back to
```

"What config exists?" is still answerable — at build time. Enumeration is bake's job, and the manifest is the explicit, PR-reviewable registry:

```ts
// config.manifest.ts
export default {
  targets: {
    server: [...allTesserae],
    client: [...appPublicTesserae],
  },
};
```

## Derived-tessera expression language

No `derive()` tessera, no expression strings in descriptors. Derivation is code, and a derivation DSL re-implements your language badly. It would also break the contract-not-dependency gate: descriptors round-trip as JSON, and closures do not serialize — while expression *strings* would be an eval surface. Compute derived values in the host language:

```ts
const cfg = bind([dbHost, dbPort]);

const connectionString = `pg://${cfg.unwrap(dbHost)}:${cfg.unwrap(dbPort)}/app`;
```

For live sources, the recompute-when question already has an answer — the epoch, plus user code:

```ts
let seen = -1;
let pool: Pool | undefined;

function getPool() {
  const now = live.epoch(dbHost);
  if (!pool || now !== seen) {
    const pinned = live.snapshot(); // one coherent pass — host and port cannot tear
    pool = new Pool({ host: pinned.unwrap(dbHost), port: pinned.unwrap(dbPort) });
    seen = now;
  }
  return pool;
}
```

## Reachability-analysis codegen

Bake does not scan your bundle to discover which tesserae the client uses. It requires an explicit manifest, because static reachability analysis is unsound in both directions — and for a mechanism guarding a trust boundary, both failure modes are disqualifying:

- **False negatives.** Dynamic access and conditional imports hide real usage; a tessera the analyzer missed is config that is simply absent in production.
- **False positives.** Dead-but-imported code inflates the set, shipping public values the client never needed and silently widening the artifact.

A human-written target list is the auditable artifact: your security review reads the manifest, not an analyzer's opinion. Analysis is demoted to what it is good at — an advisory signal:

```bash
tessellum bake --check
# CI drift gate. The coverage report compares the manifest against
# client-reachable tesserae and flags drift in either direction —
# a lint on top of the explicit manifest, never a substitute for it.
```

## Instead-of, at a glance

| You want | Out of scope because | Reach for |
| --- | --- | --- |
| Feature flags, targeting | State, not config; core runs no loops | A flag SDK alongside; `freshness: "live"` files for ops dials |
| `.env` parsing | Runtimes load env natively; Bun fights self-loaders | `node --env-file`, Bun's loader, dotenv as a userland provider impl |
| Vault/SSM clients in core | Zero-dep, browser-safe core | Community provider impls via the `tessellum/async` bind |
| Sealed boxes, TTL, rotation | SDKs take strings; sync reads; secrets-manager scope | Serialization guard, redaction, `refresh()`, a real secrets manager |
| Environment cascades | Environments are sources plus overrides | Ordered `sources`; same descriptor, different deploys |
| Runtime registry | Tree-shaking, import-order side effects | Explicit `bind` at the composition root; the bake manifest |
| Derived-tessera expressions | Derivation is code; JSON round-trip | Plain functions; `epoch()`-keyed rebuilds |
| Bundle-scanning codegen | Unsound in both directions | Explicit manifest plus the `bake --check` coverage report |

Every "no" on this page keeps a "yes" cheap: inert descriptors stay inert, the browser entry stays secret-free by construction, the bundle stays small, and the contract stays a contract instead of a dependency.
