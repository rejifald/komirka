A design document that hides its open questions is marketing. Everything else on this site reads as contract; this page is the delta between contract and certainty. Each entry states the question, the options on the table, and the current lean. A lean is not a promise — if you have production experience that bears on any of these, that is exactly the feedback this phase exists to collect.

Once a question is settled with a full decision record, its entry graduates to a one-line pointer to an [ADR](https://github.com/rejifald/komirka/tree/main/docs/adr) (see §10 for the first) — the ADR carries the context, options, and consequences; this page stays the map of decided-with-a-record versus still-open. This registry is the single source of truth: `principles.md` §6 now points here instead of keeping a second copy of the list.

## 1. Is `default` part of the identity hash?

The identity contract on the rest of this site includes `default` — presence and value — in the hashed subset. That reflects the current lean, but it is the most revisitable line in the identity design, so it is recorded here as open.

**Option A — include it (current lean).** A changed default *is* different configuration: two copies of a library that fall back to different values are not running the same contract. Including `default` means a dual-installed package whose newer copy bumped a default trips `DescriptorMismatchError` instead of silently resolving different values depending on which copy your import graph happened to reach.

**Option B — exclude it (treat it like `doc`).** Library authors could bump a default in a patch release without breaking identity across duplicated package copies. The price: the mismatch becomes invisible, and `cfg.unwrap()` quietly returns `3` in one module and `5` in another.

The lean is A: a loud error beats a silent wrong value. The escape hatch already exists and is cheap:

```ts
// @acme/mailer v1.4.0
const retries = zerno({
  key: "MAILER_RETRIES",
  schema: z.coerce.number().int(),
  exposure: "public",
  default: 3,
});

// @acme/mailer v1.5.0 — only the default changed
const retries = zerno({
  key: "MAILER_RETRIES",
  schema: z.coerce.number().int(),
  exposure: "public",
  default: 5,
});
```

With `default` in the hash, an app that binds one copy and reads through the other fails loudly, and the error hands you the fix:

```text
DescriptorMismatchError: "MAILER_RETRIES" is bound under a different descriptor.
  bound:    default: 3   (zernov1:xxh128:9f2a…)
  reading:  default: 5   (zernov1:xxh128:c41b…)
  diff: default
  To accept the newer descriptor at your composition root:
    bind(zernos, { unify: [{ accept: retries, aliases: ["zernov1:xxh128:9f2a…"] }] })
```

The residual cost lands on library authors: a default bump is an identity-breaking change and should be released like one. `unify` resolutions always re-validate with the caller's schema, so accepting the newer descriptor never downgrades validation. If real-world friction shows this generating more `unify` boilerplate than it prevents bugs, option B gets a second hearing — but the burden of proof sits on the silent side.

## 2. Env-varying public values for build-once-promote-many SPAs

This is the predicted number-one GitHub issue, so it gets named before anyone files it.

The bake pipeline inlines public literal values into the client artifact at build time. For secrets that is a feature — bake never resolves secret values, which is what preserves build-once-deploy-many for the server side. But *public* values are frozen at bake time, and teams that promote one immutable artifact from staging to production need `API_BASE_URL` to differ per environment without a rebuild.

**Option A — do nothing; document rebuild-per-environment.** Cheapest and honest (`komirka bake` once per environment in the CI matrix). It also angers precisely the teams with the strictest promotion pipelines — the ones most likely to care about this library.

**Option B — a runtime-injected JSON mode (lean).** The deploy environment serves a small public document, conventionally `/config.json`; the client fetches it once at boot through an async provider entry and pins (`await bind(entries).snapshot()`), so every read after boot is sync. Values still validate against the same zerno schemas. The browser entry still contains no secret-capable sources, so a secret zerno pointed at that document fails with `ExposureError` — the document cannot become a side-channel for secrets.

**Option C — HTML injection.** The server templates the same JSON into a script tag; saves a request, couples you to a serving layer. Probably a documented variant of B rather than a separate mode.

The undecided part is the shape, not the need. In the current contract the pieces already compose as a userland recipe — a provider entry plus an async pin:

```ts
// candidate recipe — sketch, not contract; the blessed shape is TBD
const apiBase = zerno({
  key: "API_BASE_URL",
  schema: z.string().url(),
  exposure: "public",
  sources: [{ provider: "runtime-config" }, { env: "API_BASE_URL" }],
});

// browser boot: runtimeConfigProvider is a userland provider that
// fetches /config.json once; all reads are sync forever after
const cfg = await bind({ apiBase: provider(apiBase, runtimeConfigProvider) }).snapshot();
```

```json
{
  "API_BASE_URL": "https://api.example.com",
  "CDN_ORIGIN": "https://cdn.example.com"
}
```

Open shape questions: does core bless a first-class browser source or only document the provider recipe; should bake emit a JSON Schema for the document so the *deploy* pipeline can validate it before serving; and how the baked shim's identity-hash check (`StaleBakeError`) interacts with values that arrive at runtime instead of bake time. Current lean: bless exactly one runtime-injected mode, and keep it out of v0.1 unless the proof-gate demo (below) demands it.

## 3. Ship the Node file source in v0.1?

**Option A — ship the file source in v0.1 (lean yes).** **Option B — env-only v0.1, files in v0.2.**

The argument for shipping is that the file source is what makes `freshness: "live"` real. Live env vars are physically impossible on Node — a running process's OS environment never changes. Files are the one re-readable source with a mainstream deployment story: Kubernetes volume-mounted ConfigMaps update in place via the `..data` symlink swap (`subPath` mounts never update). Without a file source, `LiveBinding`, `epoch()`, `refresh()`, `health()`, and keep-last-good are paper features that no one can run — and a v0.1 that documents machinery it cannot demonstrate fails the honesty bar this project set for itself.

```ts
import { zerno } from "komirka";
import { bind } from "komirka/node";

const rateLimit = zerno({
  key: "RATE_LIMIT",
  schema: z.coerce.number().int().positive(),
  exposure: "public",
  sources: [{ file: "/etc/app-config/RATE_LIMIT" }],
  freshness: "live",
});

const live = bind(rateLimit); // : LiveBinding — the declared { file } ref resolves via komirka/node

// per-request pin — the documented pattern
app.use((req, res, next) => {
  res.locals.cfg = live.snapshot(); // one coherent Snapshot per request
  next();
});

app.get("/healthz", (req, res) => {
  res.json(live.health()); // staleness + last validation error, per zerno
});
```

The argument against is scope: poll-on-read throttling (`maxStalenessMs`), single-coherent-pass reads so multi-zerno files never tear, keep-last-good plus `onInvalid`, and filesystem quirks across platforms are a real test surface for a first release. The lean stays yes — the file source is the difference between "live is designed" and "live exists."

## 4. Server bake targets: zero-runtime-dep or runtime validation?

For the client target there is no tension: public values are validated at bake time, then inlined next to a ~30-line shim that imports nothing. Zero runtime dependency *and* validated — validation just happened earlier, which is safe because the values are build-constant.

For servers the tension is structural. Validators are BYO Standard Schema objects — closures. They do not serialize into a generated module, and bake deliberately never resolves secret values. So for any value that varies per deployment (every secret, every per-environment var), **zero-runtime-dep and runtime schema validation are mutually exclusive**. You cannot inline what you refuse to resolve, and you cannot re-validate without the validator.

The lean is a two-tier story rather than pretending one mechanism covers both:

- **Tier 1 — build-constant values: bake them.** This is how StitchAPI consumes the library — a devDependency, bake at build time, zero runtime dependency in the published package.
- **Tier 2 — runtime-varying values: runtime-bind them.** The server imports the library as a normal dependency and binds at boot with full schema validation. The contract already permits this: mandatory bake attaches to the *trust boundary* — untrusted client targets — and trusted servers may runtime-bind.

```ts
// Tier 2: the library is a real dependency here, and that is the point —
// secrets validate at runtime, where the value actually lives
import { bind } from "komirka";
import { serverZernos } from "./config/zernos";

const cfg = bind(serverZernos);
// anything missing or invalid dies here, at boot, in ONE aggregated
// MissingConfigError listing every offender — not request by request
```

The open sub-question is what the *optional* baked server wiring module checks. It already reads env dynamically — computed access over a key array, never static `process.env.KEY`, so DefinePlugin cannot fold values into literals. Candidates: presence-only checks at boot (schema-free, still catches the deploy that forgot a var), or pure enumeration with no checks at all. What is **not** on the table: serializing validators via generated code, or a mini validation DSL embedded in baked output — both collide head-on with BYO Standard Schema and the bundle-frugality gate.

## 5. Nested `variants()` / `when()`

**Decision for v1: rejected at construction** (`VariantsNestingError`) — compose two flat
groups instead. Nesting semantics (transitive activity propagation, chained
`InactiveVariantError`, condition algebra) are the hard 20% and nobody has presented the
real case yet. **But the relation wire format is recursive and tagged from day one** —
entries are a zerno descriptor, a `$when` node, or a `$variants` node — so enabling nesting
later is a feature flag, not a format break. Non-negotiable either way: transitive
inactivity is specced now, so a default can never resurrect a subtree whose ancestor
branch is inactive.

## 6. Client bake for variants: the attested branch-freeze

v1 ships whole-group client-ineligibility (`VariantsInClientTargetError`): baking would
freeze a build-time branch choice that runtime can silently contradict, and
mixed-exposure branches would ship torn half-branches. The v2 escape hatch is designed
but not shipped: an **attested branch-freeze** — `{ group, frozenBranch }` declared in
the manifest, bake resolves the discriminant, ships only that branch's public literals,
and records the frozen value in the committed lockfile so the attestation diff shows
exactly which branch the client was frozen to.

## 7. Async reads: stale-while-revalidate?

Not in v1. Expired reads block on a per-key single-flight re-read, so `maxStalenessMs`
stays an honest hard bound — serve-stale-while-revalidating would make the name a lie.
If demand materializes it arrives as a loudly-named opt-in, never a silent default.
Related deferral: `cfg.update(zerno, fn, { retries })` read-modify-write sugar — it is a
five-line userland loop over `get`/`set`/`WriteConflictError`; core adds it only if
demand proves out.

**Write verification ship order** — `verify: "report"` ships first (it delivers most of
the value: catching store-side mangling before fleet-wide adoption, including the
mangled-but-valid class nothing else sees). `verify: "revert"` ships only once the
capability plumbing exists in real providers (unique version tokens, `echoWrite` /
`strongRead`) — every data-losing failure mode found in review routes through a revert
taken without its preconditions. `setMany` verification is report-only until atomic
revert-of-many has an answer for partially superseded batches.

## 8. Secrets in `.values`

The `.values` view ships with **raw values plus a hardened bag** (null prototype,
redacting `toJSON`, redacting Node inspect hook — so `JSON.stringify` and `console.log`
never dump secrets; spread is the documented residual, equivalent to calling `unwrap()` on
every zerno). Two alternatives were considered and are recorded as revisitable:
`Redacted<T>` brands for secret zernos (safest, but under secret-by-default exposure it
guts the flagship destructure and pressures everyone toward `exposure: "public"` — a
worse equilibrium), and omitting secret zernos from `.values` entirely (type-level
omission requires exposure as a type parameter, and a type-widened zerno would then
silently *vanish* from the record — a stranger confusion than the spread residual).
The AST lint rule (no `cfg.values` as a function argument or spread in client/library
code) is the misuse backstop.

## 9. Attestation strictness for rewired deployments

Open: should production binds on bake-covered targets *require* the lockfile digest check
(fail-closed when the composition root's effective wiring diverges from the committed
attestation) or is the check opt-in via `bind(..., { attest })` with a loud dev warning?
Mandatory is the only thing that makes manifest-invisible wiring truly unable to sneak past the
manifest; the cost is coupling server boot to a committed artifact. Leaning mandatory for
client-covered targets only. Related settled points: mapping *functions* are dropped from
the public source-transform API pre-1.0 (data-only `prefix`/`rename` — nothing in the
motivating cases needs a function), and the TypeScript floor is **>= 5.0** (`const` type
parameters carry the tuple inference the call shapes depend on).

## 10. Recently resolved

- **Secret value representation** — resolved in **[ADR 0002](https://github.com/rejifald/komirka/blob/main/docs/adr/0002-secret-value-representation.md)**: `unwrap()` of a secret returns a Proxy-based `Secret<T>` — **assignable to `T`** (no `.reveal()` ceremony for coercion) — that throws on `JSON.stringify` and is non-cloneable (`structuredClone`/`postMessage` throw `DataCloneError`), redacts in logs, and offers `.reveal()` for the typecheck sinks (`crypto`, `z.string().parse`), those flagged by a lint rather than the type. A spike (real zod v4 + Node crypto) proved the bare-`string`-that-throws claim unrealizable and characterized boxed-`String` (porous — no `structuredClone` guard, spread leak) vs `Proxy` (complete). The **hybrid** (string-assignable brand + targeted lint) was chosen over a `Redacted<T>` tax (§8) and an honest-but-guardless primitive. Resolves #19; supersedes the earlier claim.
- **Client read API & bake staleness** — resolved in **[ADR 0003](https://github.com/rejifald/komirka/blob/main/docs/adr/0003-client-read-api-and-bake-staleness.md)**: bake emits schema-free per-zerno **handles**; client code imports handles from `config.baked` (never the real descriptor), proven to drop `zod` from the browser bundle (~0.4 KB gzip vs ~65 KB with the descriptor). `StaleBakeError` becomes a **build-time** `bake --check` guarantee (load-bearing) in prod, plus an automatic **dev-mode** runtime check that is stripped from prod builds (proven: dynamic-import prod bundle 414 B gzip, zod absent; a *static* descriptor import re-ships ~65 KB of zod). Resolves #20; also fixes the browser-guide-vs-client-graph-lint contradiction.
- **Zerno inheritance & derivation** — resolved in **[ADR 0001](https://github.com/rejifald/komirka/blob/main/docs/adr/0001-zerno-inheritance.md)**: a declared, identity-bearing `inherit(target, base, ops?)` relation (not a source ref, not deployment wiring); data-only transforms (never a function); layered validation (the base's own declaration, then the target's); loud-by-default with a bind-site `useDefault` opt-in; single-base and transitive, multiple inheritance deferred. The ADR is the full record — this registry only points to it. Deferred spin-off: a `latch` freshness mode ([#14](https://github.com/rejifald/komirka/issues/14)).
- **Live-invalid default** — resolved by author decision: **current truth wins** (poisoned
  slots, `InvalidLiveValueError`, `safe`), with `keepLastGood(zerno, { maxCoastMs })` as
  the explicit bind-site opt-in. See [Freshness](/docs/concepts/freshness).
- **`freshness` in the identity hash** — resolved: **yes**. Live-vs-snapshot is
  behaviorally load-bearing; relations reference discriminants by identity, and a live
  twin must not be identity-equal. Pre-release was the last cheap day to make this change.
- **Manifest shape** — resolved: the manifest is compose-only *source*; the attestation
  artifact is a **committed lockfile** (`config.manifest.lock.json`) with per-entry
  provenance, gated by `bake --check` in a zero-credential CI job. See
  [Bake](/docs/bake/overview).
- **Call shapes** — resolved: `bind()` accepts single/array/record entries and a single
  source without array wrapping; the snapshot mirrors the input shape via `.value` /
  `.values` (a real frozen tuple for arrays — snapshots are deliberately *not* iterable,
  because positional iterable types are a lie in TypeScript). See
  [Binding](/docs/concepts/binding).
- **Secret-class providers** — resolved (author proposal): provider-sourced secrets flow
  through a dedicated `SecretProvider` — same batch shape (`class: "secret"` pin; the
  write result is derived as `Omit<WriteResult, "storedRaw">`, so echo-leaking is
  unrepresentable and `echoWrite` does not exist), bidirectional pairing at bind and bake,
  `holdsSecrets` opt-in inheriting the full obligation set, error QUARANTINE (never
  propagate provider error objects — connection errors embed the provider's own
  credential), runtime result reshaping, exposure-segregated batches, `cacheUntil` as a
  pure cache ceiling (leased/dynamic credentials are a written non-goal),
  `SecretLiteralError` gating `literal()` on secrets outside tests, and public-only
  discriminants. Secret write verification: version equality > `strongRead` digest >
  honest `"inconclusive"` outcome (per-zerno `requireVerifiedWrites` refuses). Open:
  extending `read()` to version-pinned keys is deferred until conformance data across
  real stores shows the `"superseded"` outcome is insufficient.
- **Batch-only provider contract** — resolved (author proposal): `read`/`readMany` and
  `write`/`writeMany` collapsed into batch-shaped methods — "assume plural". Results are
  input-aligned slot arrays (`{ raw }` / `{ absent: true }` / `{ error }`); chunking is
  exclusively core's job (`limits` declared, never provider-internal); the two signals the
  old shape carried moved into declared capabilities: `atomicWrites` (gates multi-entry
  writes and `setMany` atomicity; conformance-kit-verified) and `consistentReads` (demoted
  to a round-trip optimization by the discriminant re-check loop). Deferred with
  prerequisites: same-tick micro-batching of `unwrap()` (only after per-slot errors + limits;
  microtask-drain only; `snapshot()`/`refresh()` always issue dedicated batches).
  Single-key providers get a documented 3-line recipe, deliberately not an exported
  helper (a shipped adapter becomes a shadow second contract form).
- **Wiring (supersedes the sources argument AND `rebind`)** — resolved: per-entry
  combinators — `env(zerno, "NAME")` renames a declared ref in place preserving fallbacks,
  `chain()` is the only full replacement, groups wire as one entry with
  `{ prefix, overrides }`, bare entries resolve declared refs against the bind entry
  point's natural defaults. Combinators are pure JSON data (provider impls declare
  `{ id, color }`); capability enters only at the bind entry point (the Workers bag is a
  required bind option). `load()` and `snapshotOf()` died into the same vocabulary
  (async pin, `literal()` entries). Effective names still propagate to every artifact,
  error message, and the leak scan's name union; library renames use declared fallback
  chains, not deployment wiring. See [Binding](/docs/concepts/binding).

## 11. Final naming — resolved

Both names are settled.

**Package: `komirka`** (комірка — Ukrainian for a memory/spreadsheet cell), published on npm as a 0.0.0 placeholder; the repo lives at github.com/rejifald/komirka.

**Primitive: `zerno`** (зерно — Ukrainian for a grain/seed: one small unit of config). It carries through the whole surface: the group helper `defineZernos`, the type `Zerno<T>`, the `$zerno` structural brand, and the `zernov1:` identity-hash prefix. It reads well as the unit, is search-unique (greppable — the deciding criterion), has no reactivity connotation, and stays on-brand with the package's Ukrainian metaphor: grains kept in a memory-cell.

Why not keep `knob`: a knob is something you *turn*, which quietly promises the runtime tuning/reactivity this library refuses. An inert descriptor you never adjust should not borrow a control-panel verb. **atom** was rejected for the same reason and then some — jotai and recoil own it, and it implies subscription and re-render; there is no watch or subscribe API here, by design.

Runner-up and also-rans, kept for the record: **`grain`** was the closest contender (same metaphor, self-documenting in English) and lost only on greppability and brand cohesion. Others considered: `slot` / `cell` (storage), `panel` / `dial` / `setpoint` (control-room), `notions` / `bobbin` / `loom` / `weft` (sewing — pairs with the Stitch ecosystem), `mote` / `filament` / `quark` (smallest-unit), `confit` / `confetti` (wordplay).

## 12. The v0.1 proof gate

The last open decision is whether to ship at all, and it has a pass/fail procedure attached. Before a v0.1 tag exists, one public demo repository must show a **single zernos file, consumed unchanged** — never forked per runtime — by:

1. **A Node server** — runtime `bind` from `komirka/node` over bare entries, plus one `freshness: "live"` file-ref zerno proving poisoning, `keepLastGood`, and `epoch()` against an edited file.
2. **A Cloudflare Worker** — per-request `bind(entries, { env })` from `komirka/workers`: the case import-time validators structurally cannot handle, demonstrated rather than asserted.
3. **A Vite client** — a baked target running on the zero-runtime-dep shim, with `komirka bake --check` green in CI.
4. **A planted leak** — a hand-rolled high-entropy string smuggled into client code, and `komirka bake --check` failing the build via the allowlist scan. The gate is not that the scan exists; it is that it visibly catches the plant.
5. **A vitest suite with zero `process.env` mutation** — every test constructs its own fixture and passes it into factories:

```ts
import { bind, literal } from "komirka";

const cfg = bind([
  literal(apiBase, "https://test.local"),
  literal(poolSize, 2),
]); // still validated; no globals touched; safe under parallel test runners
```

The bar is comparative, not aesthetic: the demo must **clearly beat t3-env plus hand-rolling** — fewer lines at the call sites, guarantees the incumbents structurally cannot make (per-request Worker binding, fail-closed secrets, a build that fails on a planted leak, tests without env mutation), and no asterisks. If it does not clearly win, the honest outcome is to publish the demo and the learnings and stop. The ecosystem does not need a fifth config library that is merely different.
