A design document that hides its open questions is marketing. Everything else on this site reads as contract; this page is the delta between contract and certainty. Each entry states the question, the options on the table, and the current lean. A lean is not a promise — if you have production experience that bears on any of these, that is exactly the feedback this phase exists to collect.

Once a question is settled with a full decision record, its entry graduates to a one-line pointer to an [ADR](https://github.com/rejifald/tessellum/tree/main/docs/adr) (see §10 for the first) — the ADR carries the context, options, and consequences; this page stays the map of decided-with-a-record versus still-open. This registry is the single source of truth: `principles.md` §6 now points here instead of keeping a second copy of the list.

## 1. Is `default` part of the identity hash?

The identity contract on the rest of this site includes `default` — presence and value — in the hashed subset. That reflects the current lean, but it is the most revisitable line in the identity design, so it is recorded here as open.

**Option A — include it (current lean).** A changed default *is* different configuration: two copies of a library that fall back to different values are not running the same contract. Including `default` means a dual-installed package whose newer copy bumped a default trips `DescriptorMismatchError` instead of silently resolving different values depending on which copy your import graph happened to reach.

**Option B — exclude it (treat it like `doc`).** Library authors could bump a default in a patch release without breaking identity across duplicated package copies. The price: the mismatch becomes invisible, and `cfg.unwrap()` quietly returns `3` in one module and `5` in another.

The lean is A: a loud error beats a silent wrong value. The escape hatch already exists and is cheap:

```ts
// @acme/mailer v1.4.0
const retries = tessera({
  key: "MAILER_RETRIES",
  schema: z.coerce.number().int(),
  exposure: "public",
  default: 3,
});

// @acme/mailer v1.5.0 — only the default changed
const retries = tessera({
  key: "MAILER_RETRIES",
  schema: z.coerce.number().int(),
  exposure: "public",
  default: 5,
});
```

With `default` in the hash, an app that binds one copy and reads through the other fails loudly, and the error hands you the fix:

```text
DescriptorMismatchError: "MAILER_RETRIES" is bound under a different descriptor.
  bound:    default: 3   (tesserav1:xxh128:9f2a…)
  reading:  default: 5   (tesserav1:xxh128:c41b…)
  diff: default
  To accept the newer descriptor at your composition root:
    bind(tesserae, { unify: [{ accept: retries, aliases: ["tesserav1:xxh128:9f2a…"] }] })
```

The residual cost lands on library authors: a default bump is an identity-breaking change and should be released like one. `unify` resolutions always re-validate with the caller's schema, so accepting the newer descriptor never downgrades validation. If real-world friction shows this generating more `unify` boilerplate than it prevents bugs, option B gets a second hearing — but the burden of proof sits on the silent side.

## 2. Env-varying public values for build-once-promote-many SPAs


**Resolved** in **[ADR 0008](https://github.com/rejifald/tessellum/blob/main/docs/adr/0008-env-varying-public-values.md)**. This was the predicted number-one GitHub issue, so it gets an answer rather than a hedge. Bake inlines public literals at build time, so `API_BASE_URL` is frozen per build; teams promoting one immutable artifact need it to vary without a rebuild. **Both paths are supported, and the trade-off is explicit:**

- **Default — re-bake per environment.** `tessellum bake` once per environment in the promote step. Only public values resolve, so no secrets enter the pipeline. Cost: artifacts are no longer bit-identical.
- **Blessed escape — exactly one runtime-injected public document.** A per-environment `/config.json` fetched **once at boot** through an async provider entry and pinned (`await bind(entries).snapshot()`), so every read after boot is synchronous. Values validate against the same tessera schemas; a secret tessera pointed at the document is an `ExposureError` (the browser entry ships **no** secret-capable sources — construction, not policy); the document is an untrusted input boundary, revalidated on read; and because these tesserae are never baked they carry no baked handle and can never raise `StaleBakeError`. Bake emits a **JSON Schema** for the document so the deploy pipeline validates it *before serving*.
- **Variant — HTML script-tag injection.** The same document, templated into a script tag: DOM transport instead of a fetch. Saves a request, couples you to a serving layer.

It stays a **userland recipe** — no first-class core browser source in v0.1 (P6). **P13 was narrowed** to say what it always meant: bake is the only channel that *inlines* client config. See [Browser](/docs/guides/browser).

## 3. Ship the Node file source in v0.1?

**Option A — ship the file source in v0.1 (lean yes).** **Option B — env-only v0.1, files in v0.2.**

The argument for shipping is that the file source is what makes `freshness: "live"` real. Live env vars are physically impossible on Node — a running process's OS environment never changes. Files are the one re-readable source with a mainstream deployment story: Kubernetes volume-mounted ConfigMaps update in place via the `..data` symlink swap (`subPath` mounts never update). Without a file source, `LiveBinding`, `epoch()`, `refresh()`, `health()`, and keep-last-good are paper features that no one can run — and a v0.1 that documents machinery it cannot demonstrate fails the honesty bar this project set for itself.

```ts
import { tessera } from "tessellum";
import { bind } from "tessellum/node";

const rateLimit = tessera({
  key: "RATE_LIMIT",
  schema: z.coerce.number().int().positive(),
  exposure: "public",
  sources: [{ file: "/etc/app-config/RATE_LIMIT" }],
  freshness: "live",
});

const live = bind(rateLimit); // : LiveBinding — the declared { file } ref resolves via tessellum/node

// per-request pin — the documented pattern
app.use((req, res, next) => {
  res.locals.cfg = live.snapshot(); // one coherent Snapshot per request
  next();
});

app.get("/healthz", (req, res) => {
  res.json(live.health()); // staleness + last validation error, per tessera
});
```

The argument against is scope: poll-on-read throttling (`maxStalenessMs`), single-coherent-pass reads so multi-tessera files never tear, keep-last-good plus `onInvalid`, and filesystem quirks across platforms are a real test surface for a first release. The lean stays yes — the file source is the difference between "live is designed" and "live exists."

## 4. Server bake targets: zero-runtime-dep or runtime validation?

For the client target there is no tension: public values are validated at bake time, then inlined next to a ~30-line shim that imports nothing. Zero runtime dependency *and* validated — validation just happened earlier, which is safe because the values are build-constant.

For servers the tension is structural. Validators are BYO Standard Schema objects — closures. They do not serialize into a generated module, and bake deliberately never resolves secret values. So for any value that varies per deployment (every secret, every per-environment var), **zero-runtime-dep and runtime schema validation are mutually exclusive**. You cannot inline what you refuse to resolve, and you cannot re-validate without the validator.

**Resolved** in **[ADR 0010](https://github.com/rejifald/tessellum/blob/main/docs/adr/0010-server-bake-target-semantics.md)**. The answer is a two-tier story rather than pretending one mechanism covers both — and it is now settled contract, not a lean:

- **Tier 1 — build-constant values: bake them.** This is how StitchAPI consumes the library — a devDependency, bake at build time, zero runtime dependency in the published package.
- **Tier 2 — runtime-varying values: runtime-bind them.** The server imports the library as a normal dependency and binds at boot with full schema validation. The contract already permits this: mandatory bake attaches to the *trust boundary* — untrusted client targets — and trusted servers may runtime-bind.

```ts
// Tier 2: the library is a real dependency here, and that is the point —
// secrets validate at runtime, where the value actually lives
import { bind } from "tessellum";
import { serverTesserae } from "./config/tesserae";

const cfg = bind(serverTesserae);
// anything missing or invalid dies here, at boot, in ONE aggregated
// MissingConfigError listing every offender — not request by request
```

The sub-question — what the *optional* baked server wiring module checks — is resolved as **presence-only at boot**. It enumerates the declared env key names, reads each via computed access (`process.env[k]`, never static `process.env.KEY`, so DefinePlugin cannot fold values into literals), and throws **one** aggregated error naming every missing key. It is schema-free and **imports nothing** — reusing the *name* `MissingConfigError` without taking the dependency — so a package that bakes at build keeps its zero-runtime-dependency guarantee.

Presence is not validity: `DATABASE_URL=""` passes the module and still fails the real schema at the tier-2 `bind()`. Pure enumeration with no checks was rejected (it gives nothing over reading `process.env` yourself). Permanently **not** on the table: serializing validators via generated code, or a mini validation DSL embedded in baked output — both collide head-on with BYO Standard Schema and the bundle-frugality gate, and a generated check that silently diverged from the real schema would be worse than no check at all.

## 5. Nested `variants()` / `when()`

**Decision for v1: rejected at construction** (`VariantsNestingError`) — compose two flat
groups instead. Nesting semantics (transitive activity propagation, chained
`InactiveVariantError`, condition algebra) are the hard 20% and nobody has presented the
real case yet. **But the relation wire format is recursive and tagged from day one** —
entries are a tessera descriptor, a `$when` node, or a `$variants` node — so enabling nesting
later is a feature flag, not a format break. Non-negotiable either way: transitive
inactivity is specced now, so a default can never resurrect a subtree whose ancestor
branch is inactive.

## 6. Client bake for variants: the attested branch-freeze

v1 ships whole-group client-ineligibility (`VariantsInClientTargetError`): baking would
freeze a build-time branch choice that runtime can silently contradict, and
mixed-exposure branches would ship torn half-branches. The v2 escape hatch is designed
but not shipped: an **attested branch-freeze** — `{ group, frozenBranch }` declared in
the manifest, bake resolves the discriminant, ships only that branch's public literals,
and records the frozen value in the committed lockfile so the attestation diff shows
exactly which branch the client was frozen to.

## 7. Async reads: stale-while-revalidate?

Not in v1. Expired reads block on a per-key single-flight re-read, so `maxStalenessMs`
stays an honest hard bound — serve-stale-while-revalidating would make the name a lie.
If demand materializes it arrives as a loudly-named opt-in, never a silent default.
Related deferral: `cfg.update(tessera, fn, { retries })` read-modify-write sugar — it is a
five-line userland loop over `get`/`set`/`WriteConflictError`; core adds it only if
demand proves out.

**Separately settled — cross-key coherence within an async `snapshot()`** (a different
async-read question, resolved in **[ADR 0004](https://github.com/rejifald/tessellum/blob/main/docs/adr/0004-async-snapshot-coherence.md)**): a declared `coherent([...])` group is
pinned and re-read as one coherent cut (ungrouped keys stay expired-keys-only), `version`
joins `Provenance` as the CAS token, and a `WriteConflictError` marks the conflicting key
stale — which is what makes the five-line loop above provably make progress. Stale-while-
revalidate itself stays deferred as above.

**Separately settled — the blocking-read deadline and boot outage** (the read-*liveness*
axis of the same async-read question, resolved in **[ADR 0005](https://github.com/rejifald/tessellum/blob/main/docs/adr/0005-async-read-deadline-and-boot-outage.md)**): the single-flight
re-read above (and the coherent-group cut just settled) blocks only within `readTimeoutMs`
(a new bind option, default `maxStalenessMs`) — a slow provider that breaches the deadline
throws `SourceUnavailableError` instead of wedging every coalesced request unbounded (distinct
from `CoherenceError`, a store that answers fast but keeps changing), and `ready()`, which has
no cache to fall back on, handles a store down at boot with bounded retry then fail-fast
(`readyTimeoutMs`, default 30 000 ms; `0` = pure fail-fast). `maxStalenessMs` bounds
*staleness*; `readTimeoutMs` bounds the *wait*. Stale-while-revalidate itself stays deferred,
as above.

**Write verification ship order** — `verify: "report"` ships first (it delivers most of
the value: catching store-side mangling before fleet-wide adoption, including the
mangled-but-valid class nothing else sees). `verify: "revert"` ships only once the
capability plumbing exists in real providers (unique version tokens, `echoWrite` /
`strongRead`) — every data-losing failure mode found in review routes through a revert
taken without its preconditions. `setMany` verification is report-only until atomic
revert-of-many has an answer for partially superseded batches — and, per **[ADR 0006](https://github.com/rejifald/tessellum/blob/main/docs/adr/0006-atomic-setmany-vs-write-batch.md)**,
an atomic `setMany` is capped at the provider's `maxWriteBatch` (refused above it with
`AtomicBatchTooLargeError`, because a chunked atomic write cannot be one transaction), so the
post-state report mode observes is always exactly one transaction, never a smear across chunks.

## 8. Secrets in `.values`

The `.values` view ships with **raw values plus a hardened bag** (null prototype,
redacting `toJSON`, redacting Node inspect hook — so `JSON.stringify` and `console.log`
never dump secrets; spread is the documented residual, equivalent to calling `unwrap()` on
every tessera). Two alternatives were considered and are recorded as revisitable:
`Redacted<T>` brands for secret tesserae (safest, but under secret-by-default exposure it
guts the flagship destructure and pressures everyone toward `exposure: "public"` — a
worse equilibrium), and omitting secret tesserae from `.values` entirely (type-level
omission requires exposure as a type parameter, and a type-widened tessera would then
silently *vanish* from the record — a stranger confusion than the spread residual).
The AST lint rule (no `cfg.values` as a function argument or spread in client/library
code) is the misuse backstop.

## 9. Attestation strictness for rewired deployments


**Resolved** in **[ADR 0009](https://github.com/rejifald/tessellum/blob/main/docs/adr/0009-attestation-strictness.md)**. The lockfile wiring-digest check is **mandatory on client-covered targets** — always on, not disableable, and keyed off *bake coverage* (a structural fact) rather than `NODE_ENV`, so no ambient environment string ever decides whether a guarantee is enforced. Divergence is `WiringSkewError`, fail-closed. **Server targets opt in** via `bind(entries, { attest: true })` (`attest?: boolean`, default `false`), and a bake-covered server target bound without it gets a dev-mode warning.

The honest cost, stated rather than hidden: **manifest-invisible wiring on a server is possible by default.** `tessellum bake --check` in a zero-credential CI job is the primary control; `attest: true` is the boot-time backstop for operators who want boot coupled to the committed attestation. Mandatory-everywhere was rejected because it turns a stale lockfile into an outage on the tier that most legitimately rewires per deployment; opt-in-everywhere was rejected because it ships a core security guarantee in the off position.

Related settled points: mapping *functions* are dropped from the public source-transform API pre-1.0 (data-only `prefix`/`rename` — nothing in the motivating cases needs a function), and the TypeScript floor is **>= 5.0** (`const` type parameters carry the tuple inference the call shapes depend on).

## 10. Recently resolved

- **Secret value representation** — resolved in **[ADR 0002](https://github.com/rejifald/tessellum/blob/main/docs/adr/0002-secret-value-representation.md)**: `unwrap()` of a secret returns a Proxy-based `Secret<T>` — **assignable to `T`** (no `.reveal()` ceremony for coercion) — that throws on `JSON.stringify` and is non-cloneable (`structuredClone`/`postMessage` throw `DataCloneError`), redacts in logs, and offers `.reveal()` for the typecheck sinks (`crypto`, `z.string().parse`), those flagged by a lint rather than the type. A spike (real zod v4 + Node crypto) proved the bare-`string`-that-throws claim unrealizable and characterized boxed-`String` (porous — no `structuredClone` guard, spread leak) vs `Proxy` (complete). The **hybrid** (string-assignable brand + targeted lint) was chosen over a `Redacted<T>` tax (§8) and an honest-but-guardless primitive. Resolves #19; supersedes the earlier claim.
- **Client read API & bake staleness** — resolved in **[ADR 0003](https://github.com/rejifald/tessellum/blob/main/docs/adr/0003-client-read-api-and-bake-staleness.md)**: bake emits schema-free per-tessera **handles**; client code imports handles from `config.baked` (never the real descriptor), proven to drop `zod` from the browser bundle (~0.4 KB gzip vs ~65 KB with the descriptor). `StaleBakeError` becomes a **build-time** `bake --check` guarantee (load-bearing) in prod, plus an automatic **dev-mode** runtime check that is stripped from prod builds (proven: dynamic-import prod bundle 414 B gzip, zod absent; a *static* descriptor import re-ships ~65 KB of zod). Resolves #20; also fixes the browser-guide-vs-client-graph-lint contradiction.
- **Async snapshot cross-key coherence** — resolved in **[ADR 0004](https://github.com/rejifald/tessellum/blob/main/docs/adr/0004-async-snapshot-coherence.md)**: coherence within an async `snapshot()` is an explicit, bind-site `coherent([...])` group — pinned and re-read as one cut whenever any member is expired (ungrouped keys stay expired-keys-only; a group must resolve to one provider or bind fails `CoherenceGroupSourceError`, `CoherenceError` if a cut can't be captured in the re-check budget). `version` joins `Provenance` as the provider CAS token, and a `WriteConflictError` marks the conflicting key stale so the database-guide read-modify-write loop provably progresses. The ADR is the full record. Its siblings are now settled too — the async re-read deadline / boot-outage ([ADR 0005](https://github.com/rejifald/tessellum/blob/main/docs/adr/0005-async-read-deadline-and-boot-outage.md)) and `setMany` atomicity vs. chunking ([ADR 0006](https://github.com/rejifald/tessellum/blob/main/docs/adr/0006-atomic-setmany-vs-write-batch.md)).
- **Async re-read deadline & boot outage** — resolved in **[ADR 0005](https://github.com/rejifald/tessellum/blob/main/docs/adr/0005-async-read-deadline-and-boot-outage.md)**: every blocking async re-read (an expired key; ADR 0004's `coherent`-group cut and its re-check loop; the post-conflict re-read) is bounded by a new `readTimeoutMs` bind option (default `maxStalenessMs`) — a slow provider that breaches it throws `SourceUnavailableError`, so `maxStalenessMs` bounds *staleness* while `readTimeoutMs` bounds the *wait*, and it stays distinct from `CoherenceError` (a store that answers fast but keeps changing). `ready()`, cacheless at boot, meets a store outage with bounded retry then fail-fast (`readyTimeoutMs`, default 30 000 ms; `0` = pure fail-fast). Resolves [#22](https://github.com/rejifald/tessellum/issues/22).
- **Atomic `setMany` vs. `maxWriteBatch`** — resolved in **[ADR 0006](https://github.com/rejifald/tessellum/blob/main/docs/adr/0006-atomic-setmany-vs-write-batch.md)**: an atomic write is one `write()` to one provider and cannot span chunks (the write-side mirror of ADR 0004's single-source rule), so an atomic `setMany` larger than the provider's `maxWriteBatch` is refused up front with `AtomicBatchTooLargeError` — a chunked atomic write would be the torn durable state the guarantee prevents (discriminant flipped, branch values missing). Shrink the rotate-together set to one batch, or opt into `{ allowNonAtomic: true }` (dependency-ordered singles, benign tear direction only). Resolves [#17](https://github.com/rejifald/tessellum/issues/17).
- **Env-varying public values for build-once-promote-many SPAs** — resolved in **[ADR 0008](https://github.com/rejifald/tessellum/blob/main/docs/adr/0008-env-varying-public-values.md)** (§2): **re-bake per environment is the default**; the blessed escape is exactly **one runtime-injected public document** (`/config.json`, fetched once at boot through an async provider entry and pinned), with HTML script-tag injection as a transport variant. Values still validate; a secret pointed at the document is an `ExposureError` because the browser entry ships no secret-capable sources; the document is an untrusted input boundary revalidated on read; and runtime-injected tesserae are never baked, so they carry no baked handle and can never raise `StaleBakeError`. Bake emits a JSON Schema for the document. It stays a userland recipe — no core browser source in v0.1 — and **P13 was narrowed** to "bake is the only channel that *inlines* client config." Resolves [#28](https://github.com/rejifald/tessellum/issues/28).
- **Attestation strictness for rewired deployments** — resolved in **[ADR 0009](https://github.com/rejifald/tessellum/blob/main/docs/adr/0009-attestation-strictness.md)** (§9): the lockfile wiring-digest check is **mandatory and always on for client-covered targets** (not disableable; keyed off bake coverage, never `NODE_ENV`), and **opt-in for server targets** via `bind(entries, { attest: true })` with a dev-mode warning when a bake-covered server omits it. Honest cost: manifest-invisible wiring on a server is possible by default, so `bake --check` in CI is the primary control. Resolves [#29](https://github.com/rejifald/tessellum/issues/29).
- **Server bake-target semantics** — resolved in **[ADR 0010](https://github.com/rejifald/tessellum/blob/main/docs/adr/0010-server-bake-target-semantics.md)** (§4): the two-tier story is settled (build-constant → bake; runtime-varying → runtime bind), and the optional server wiring module performs **presence-only** checks at boot — schema-free, importing nothing, dynamic computed access, one aggregated error naming every missing key. Presence is not validity; serialized validators and embedded DSLs stay permanently off the table. Resolves [#30](https://github.com/rejifald/tessellum/issues/30).
- **Cross-install branded-type identity & library consumption** — resolved in **[ADR 0007](https://github.com/rejifald/tessellum/blob/main/docs/adr/0007-cross-install-identity-and-library-consumption.md)**: the `Scope`/`ConfigOf` brand is **structural** (a well-known string key), never a `unique symbol` — a `tsc` spike proved a nominal brand *loses* identity across duplicate installs while a structural one survives and still blocks `Snapshot → Scope` (P12); it is also what lets ADR 0003's baked shim carry the brand while importing nothing. Library authors declare tessellum a `peerDependency` (dedup + no second bundled copy), but content-addressed identity + revalidate-on-fallback is the load-bearing floor, not the peer-dep (P17). A documented escape — a plain-object accessor plus an optional `…/tessellum` adapter entry — lets libraries serve consumers who don't use tessellum; public values cross bare, secret values cross as `Secret<T>` (ADR 0002; assignable to `T`, so the adapter passes `unwrap()` through with no `.reveal()` and the Proxy guards ride along). Resolves [#31](https://github.com/rejifald/tessellum/issues/31).
- **Tessera inheritance & derivation** — resolved in **[ADR 0001](https://github.com/rejifald/tessellum/blob/main/docs/adr/0001-tessera-inheritance.md)**: a declared, identity-bearing `inherit(target, base, ops?)` relation (not a source ref, not deployment wiring); data-only transforms (never a function); layered validation (the base's own declaration, then the target's); loud-by-default with a bind-site `useDefault` opt-in; single-base and transitive, multiple inheritance deferred. The ADR is the full record — this registry only points to it. Deferred spin-off: a `latch` freshness mode ([#14](https://github.com/rejifald/tessellum/issues/14)).
- **Live-invalid default** — resolved by author decision: **current truth wins** (poisoned
  slots, `InvalidLiveValueError`, `safe`), with `keepLastGood(tessera, { maxCoastMs })` as
  the explicit bind-site opt-in. See [Freshness](/docs/concepts/freshness).
- **`freshness` in the identity hash** — resolved: **yes**. Live-vs-snapshot is
  behaviorally load-bearing; relations reference discriminants by identity, and a live
  twin must not be identity-equal. Pre-release was the last cheap day to make this change.
- **Manifest shape** — resolved: the manifest is compose-only *source*; the attestation
  artifact is a **committed lockfile** (`config.manifest.lock.json`) with per-entry
  provenance, gated by `bake --check` in a zero-credential CI job. See
  [Bake](/docs/bake/overview).
- **Call shapes** — resolved: `bind()` accepts single/array/record entries and a single
  source without array wrapping; the snapshot mirrors the input shape via `.value` /
  `.values` (a real frozen tuple for arrays — snapshots are deliberately *not* iterable,
  because positional iterable types are a lie in TypeScript). See
  [Binding](/docs/concepts/binding).
- **Secret-class providers** — resolved (author proposal): provider-sourced secrets flow
  through a dedicated `SecretProvider` — same batch shape (`class: "secret"` pin; the
  write result is derived as `Omit<WriteResult, "storedRaw">`, so echo-leaking is
  unrepresentable and `echoWrite` does not exist), bidirectional pairing at bind and bake,
  `holdsSecrets` opt-in inheriting the full obligation set, error QUARANTINE (never
  propagate provider error objects — connection errors embed the provider's own
  credential), runtime result reshaping, exposure-segregated batches, `cacheUntil` as a
  pure cache ceiling (leased/dynamic credentials are a written non-goal),
  `SecretLiteralError` gating `literal()` on secrets outside tests, and public-only
  discriminants. Secret write verification: version equality > `strongRead` digest >
  honest `"inconclusive"` outcome (per-tessera `requireVerifiedWrites` refuses). Open:
  extending `read()` to version-pinned keys is deferred until conformance data across
  real stores shows the `"superseded"` outcome is insufficient.
- **Batch-only provider contract** — resolved (author proposal): `read`/`readMany` and
  `write`/`writeMany` collapsed into batch-shaped methods — "assume plural". Results are
  input-aligned slot arrays (`{ raw }` / `{ absent: true }` / `{ error }`); chunking is
  exclusively core's job (`limits` declared, never provider-internal); the two signals the
  old shape carried moved into declared capabilities: `atomicWrites` (gates multi-entry
  writes and `setMany` atomicity; conformance-kit-verified) and `consistentReads` (demoted
  to a round-trip optimization by the discriminant re-check loop). Deferred with
  prerequisites: same-tick micro-batching of `unwrap()` (only after per-slot errors + limits;
  microtask-drain only; `snapshot()`/`refresh()` always issue dedicated batches).
  Single-key providers get a documented 3-line recipe, deliberately not an exported
  helper (a shipped adapter becomes a shadow second contract form).
- **Wiring (supersedes the sources argument AND `rebind`)** — resolved: per-entry
  combinators — `env(tessera, "NAME")` renames a declared ref in place preserving fallbacks,
  `chain()` is the only full replacement, groups wire as one entry with
  `{ prefix, overrides }`, bare entries resolve declared refs against the bind entry
  point's natural defaults. Combinators are pure JSON data (provider impls declare
  `{ id, color }`); capability enters only at the bind entry point (the Workers bag is a
  required bind option). `load()` and `snapshotOf()` died into the same vocabulary
  (async pin, `literal()` entries). Effective names still propagate to every artifact,
  error message, and the leak scan's name union; library renames use declared fallback
  chains, not deployment wiring. See [Binding](/docs/concepts/binding).

## 11. Final naming — resolved

Both names are settled.

**Package: `tessellum`** — from Latin *tessella*, a small mosaic tile. The package is the *tessellation*: many small config tiles composing one complete surface, no gaps and no overlaps. Published on npm as a 0.0.0 placeholder; the repo lives at github.com/rejifald/tessellum.

**Primitive: `tessera`** — one mosaic tile, and in Rome a *token*: the *tessera militaris* carried the camp watchword, and the *tessera hospitalis* was split between two parties so that rejoining the halves proved a bond of trust. One config value is one tessera: a single tile in the surface that carries its own identity and exposure. It threads through the whole API — the group helper `defineTesserae`, the type `Tessera<T>`, the `$tessera` structural brand, and the `tesserav1:` identity-hash prefix. (Plural: `tesserae`, the established English plural, over the anglicized `tesseras`.)

**Why these, and why the change.** The names replaced an earlier Ukrainian pair — `komirka` (комірка, a memory/spreadsheet cell) and `zerno` (зерно, a grain), chosen 2026-07-05 — on 2026-07-09. The trade was deliberate: the Ukrainian names carried real cultural resonance but were opaque to an English-speaking audience, and a primitive you type a hundred times a day earns its keep by being *decodable*. *Tessellation* and *tessera* are half-known English already (mosaics, 3-D graphics, board-game tokens), the tile-composing-a-surface metaphor maps cleanly onto atomic config, and `tessera`'s token/trust sense reinforces the secrets-and-identity model for free. The honest cost — recorded here because this page records costs: `tessera` is a real word, so it greps a shade noisier than the search-unique `zerno` it replaced, the very criterion that decided the earlier round. We knowingly traded a little greppability for readability. `komirka@0.0.0` stays a parked npm placeholder and the old repo URL redirects.

**Rejected framing: anything you *turn*.** A `knob` promises the runtime tuning this library refuses; **atom** was out for the same reason and because jotai and recoil own it, implying subscription and re-render. There is no watch or subscribe API here, by design — and a tessera is *placed*, not adjusted, so the metaphor keeps the primitive honestly inert. (Earlier also-rans from the Ukrainian round — `grain`, `slot`, `cell`, `bobbin`, `loom`, `weft`, `mote`, `quark` — are superseded; the lineage is recorded above.)

## 12. The v0.1 proof gate

The last open decision is whether to ship at all, and it has a pass/fail procedure attached. Before a v0.1 tag exists, one public demo repository must show a **single tesserae file, consumed unchanged** — never forked per runtime — by:

1. **A Node server** — runtime `bind` from `tessellum/node` over bare entries, plus one `freshness: "live"` file-ref tessera proving poisoning, `keepLastGood`, and `epoch()` against an edited file.
2. **A Cloudflare Worker** — per-request `bind(entries, { env })` from `tessellum/workers`: the case import-time validators structurally cannot handle, demonstrated rather than asserted.
3. **A Vite client** — a baked target running on the zero-runtime-dep shim, with `tessellum bake --check` green in CI.
4. **A planted leak** — a hand-rolled high-entropy string smuggled into client code, and `tessellum bake --check` failing the build via the allowlist scan. The gate is not that the scan exists; it is that it visibly catches the plant.
5. **A vitest suite with zero `process.env` mutation** — every test constructs its own fixture and passes it into factories:

```ts
import { bind, literal } from "tessellum";

const cfg = bind([
  literal(apiBase, "https://test.local"),
  literal(poolSize, 2),
]); // still validated; no globals touched; safe under parallel test runners
```

The bar is comparative, not aesthetic: the demo must **clearly beat t3-env plus hand-rolling** — fewer lines at the call sites, guarantees the incumbents structurally cannot make (per-request Worker binding, fail-closed secrets, a build that fails on a planted leak, tests without env mutation), and no asterisks. If it does not clearly win, the honest outcome is to publish the demo and the learnings and stop. The ecosystem does not need a fifth config library that is merely different.
