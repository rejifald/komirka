title: Secrets model
description: Secrets are the default exposure and fail closed at every layer — structurally in the browser, at serialization boundaries, and in every error report.
---

Most configuration libraries treat secrecy as a naming convention: prefix the
variable with `PUBLIC_` or `NEXT_PUBLIC_` and hope every human and every build
tool agrees on what the prefix means. Tessellum inverts this. **Every tessera is a
secret unless you say otherwise**, and everything downstream — binding,
serialization, error reporting, build output — is built so that the safe
behavior is the one you get when you forget something.

## Secret is the default

`exposure` has two values: `"secret"` and `"public"`. Omitting the field means
`"secret"`. Fail closed: the mistake of *not declaring* costs you a build error
or a withheld value, never a leak.

```ts
import { tessera } from "tessellum";
import { z } from "zod";

// No `exposure` field — this is a secret.
const stripeKey = tessera({
  key: "STRIPE_SECRET_KEY",
  schema: z.string().startsWith("sk_"),
  doc: "Stripe key for server-side charge creation",
});

// Public is a deliberate, reviewable opt-in.
const apiBaseUrl = tessera({
  key: "PUBLIC_API_BASE_URL",
  schema: z.string().url(),
  exposure: "public",
  doc: "Base URL the browser client calls",
});
```

`exposure` is an **identity field**: it participates in `identityOf(...)`, so a
tessera flipped from secret to public is a *different tessera*. If two copies of a
package disagree about a key's exposure, reads fail with
`DescriptorMismatchError` — carrying both canonical descriptors and a
field-level diff — instead of silently resolving to the leakier of the two.

Deployment wiring cannot launder exposure either: wiring a non-secret tessera
onto a name in the secret-name set fails with `WiringExposureError`, a ban
backed by a well-known-secret-suffix denylist (`*_SECRET`, `*_TOKEN`,
`*_PASSWORD`, `*_PRIVATE_KEY`, `*_API_KEY`) with an explicit allowlist escape.

The same denylist gates the other end of the pipeline. `tessellum bake` refuses
to put into a `client` target any tessera whose declared `key` — or any effective
source name it resolves through — matches that set as a whole token, *even when
the tessera is declared `exposure: "public"`*, failing with
`SecretNameInClientTargetError`. This does not contradict the section above: the
`exposure` flag is still the mechanism, and a name is only a heuristic. But a
value that is public *and* named like a credential is almost always a
misclassification, so the name check is the net under the flag, not a
replacement for it. The escape is the same per-name allowlist — one reviewed
entry in the bake manifest's client attestation, recorded in the committed
lockfile — never a global flag; see [bake](/docs/bake/overview) for the shape.

Exposure cannot be laundered through the inheritance relation either. A public
tessera that inherits from a secret base fails with `InheritanceExposureError` at
construction or bind: a chain's effective exposure is the max secrecy of any
link, so neither a public read nor bake's client inlining can ever derive its
value from a secret. The reverse — a secret inheriting a *public* base — stays
legal, since a secret may hold anything. See
[ADR 0001](https://github.com/rejifald/tessellum/blob/main/docs/adr/0001-tessera-inheritance.md)
for the inheritance contract.

One more asymmetry worth internalizing early: `exposure: "public"` is
*necessary but not sufficient* for a value to reach a client bundle. The app
still has to attest to it in the bake manifest's `client` target, behind a
deny-by-default allowlist. A dependency cannot promote its own tessera into your
browser artifact. See the [bake pages](/docs/bake) for that half of the story.

## A secret cannot resolve in a browser — structurally

The core entry point ships **no source resolvers**, and core `bind` supplies
**no natural defaults**. This is not a lint rule or a runtime `if (isBrowser)`
check; the code that can read an environment simply is not in the browser
bundle. The `env` and `file` combinators live in `tessellum/node`, which imports
`node:process` and `node:fs` at module scope — bundling it for a non-Node
target fails at build time, not in production.

```ts
import { tessera, defineTesserae, bind } from "tessellum"; // browser-safe by construction
import { env, file } from "tessellum/node";          // Node-only — imports node:process and node:fs
// tessellum/workers ships its own bind: bind(entries, { env: envBag }) — the bag is required
```

A browser build that imports only `"tessellum"` has nothing to resolve a secret
*from*, no matter what the code does. Even core `bind` cooperates: it wires
entries but has no ambient environment to fall back to, so a bare `env` or
`file` ref reaching core `bind` fails with `NoWiringError` instead of
resolving. The only delivery channel for client values is `tessellum bake`, which
emits a generated module containing **public literal values only** plus a
small inlined snapshot shim that imports nothing at runtime. Bake never
resolves secret values — it ships secret key *names* (into `.env.example` and
`CONFIG.md`) but never their values, so prod secrets never enter CI and
build-once-deploy-many survives.

Reading a secret through a public context is a named failure, not a fallback:

```ts
// Browser code, using the module `tessellum bake` emitted for the client target.
import { config } from "./config.baked";

config.unwrap(apiBaseUrl);
// "https://api.example.com" — a public literal, inlined at build time

config.unwrap(stripeKey);
// throws ExposureError: secret tessera "STRIPE_SECRET_KEY" read in a public context.
// The value was never in this artifact; there is nothing to fall back to.
```

## Redacted wherever values surface

Diagnostics are where secrets traditionally escape — a snapshot dumped into a
log line, a provenance trace pasted into a bug report. Both surfaces redact by
construction:

```ts
import { bind } from "tessellum/node";

const cfg = bind([stripeKey, apiBaseUrl]);

cfg.explain(stripeKey);
// Full provenance — which declared ref supplied the value, under which
// effective name — value redacted:
//   STRIPE_SECRET_KEY  <- env "STRIPE_SECRET_KEY"  value: [redacted]

cfg.toJSON();
// {
//   "PUBLIC_API_BASE_URL": "https://api.example.com",
//   "STRIPE_SECRET_KEY": "[redacted]"
// }
```

The same redaction applies to `literal(...)` entries — a secret stays
`[redacted]` in every diagnostic even when the value came from your own test
wiring.

## The serialization guard

Redacted `toJSON()` covers the snapshot itself, but the classic server-side
leak doesn't serialize the snapshot — it serializes a *value* that wandered
into an SSR payload. Every framework that hydrates the client from server
props (the `__NEXT_DATA__` class of bug) will happily JSON-encode whatever you
hand it.

So reading a secret tessera does not hand you a bare string — it hands you a
`Secret<string>`. It behaves as a string for every *coercion* (`` `${key}` ``,
`"Bearer " + key`, `String(key)`, `key.startsWith(...)`, `Buffer.from(key)`,
`new URL(base + key)`) but **refuses to serialize**: it throws `ExposureError`
on `JSON.stringify` and is non-cloneable, so `structuredClone` and `postMessage`
throw `DataCloneError`. It redacts under `console.log` / `util.inspect`, and
`{ ...key }` yields nothing.

```ts
const key = cfg.unwrap(stripeKey); // : Secret<string> — assignable to string, usable anywhere a string is

await stripe.charges.create(params, { apiKey: key }); // fine — coerces to the raw string

JSON.stringify({ props: { key } });
// throws ExposureError — a secret value refuses to enter a JSON payload

structuredClone({ key });
// throws DataCloneError — the same refusal on the structured-clone / postMessage
// channel (the React-Server-Components / SSR-hydration path a bare string can't guard)
```

`Secret<string>` is **assignable to `string`**, so the common case carries no
ceremony — you hand it straight to an SDK and it coerces. The rare boundary that
*typechecks* its input rather than coercing — `z.string().parse(key)`,
`crypto.createHmac("sha256", key)`, a strict SDK doing `typeof x === "string"` —
needs the raw, so call `key.reveal()` there deliberately. Because the type is
assignable to `string`, the compiler will not force this, so a lint rule flags
exactly those sinks (see [the defense stack](/docs/bake/overview)); `isSecret(v)`
is the runtime brand check.

**The boundary, stated honestly:** the guard travels with the value tessellum
hands you, not with strings you *derive* from it. `` `${key}` ``,
`"Bearer " + key`, and `key.trim()` produce plain strings the guard cannot
follow — it catches the *structural* accident (a whole secret dumped into a
payload, a clone, a logged object), never the *coercion* one (a secret
interpolated into a log message). It is a tripwire, not a taint-tracking system.
It exists precisely because the build-time leak scan cannot see request-time
payloads: bake scans what the compiler emits, the serialization guard covers
what the server emits per request. Neither claims the other's territory. Full
rationale and the representation trade-offs:
[ADR 0002](https://github.com/rejifald/tessellum/blob/main/docs/adr/0002-secret-value-representation.md).

## Validator messages are untrusted for secrets

BYO validators echo the values they receive — that is most of what makes their
error messages useful. A zod message like *"expected string starting with
sk_, received nk_live_9f2..."* is a leak wearing a stack trace. For secret
tesserae, every error report carries **issue counts and paths only** — never
messages, never values:

```ts
// STRIPE_SECRET_KEY is set but fails the schema;
// PUBLIC_API_BASE_URL is set to "not a url".
const cfg = bind([stripeKey, apiBaseUrl]);
// throws MissingConfigError (aggregated — every problem in one pass):
//   PUBLIC_API_BASE_URL  invalid: Invalid url            (public: full message)
//   STRIPE_SECRET_KEY    invalid: 1 issue at (root)      (secret: counts + paths only)
```

The same rule applies to `safe` — the `issues` for a secret tessera identify
*where* validation failed, never *what* was seen:

```ts
const result = cfg.safe(stripeKey);
if (!result.ok) {
  // result.issues: paths and counts only — no validator text, no received value
}
```

This also holds across package copies: hash-fallback resolutions re-validate
with the caller's schema, and those re-validation reports are stripped the
same way.

## Why there are no Sealed boxes

A recurring request: wrap secrets in a `Sealed<string>` box so raw values
never exist unwrapped. We looked hard at it and rejected it for v1, for one
empirical reason: **every SDK takes strings**. A database driver wants a
connection string; an HTTP client wants a header value. A box you must
`.reveal()` on the very next line is ceremony, not confinement — the unwrapped
string exists either way, one call later, in the same scope. (The
`Secret<string>` you *do* get from `unwrap()` is **not** a Sealed box: it is
assignable to `string` and needs no `.reveal()` for ordinary use — it adds
serialization *refusal*, not mandatory confinement. See
[the serialization guard](#the-serialization-guard).)

The same logic rules out TTLs, rotation schedules, and access audit trails in
v1: each is a real feature of a *secrets manager*, and building them here is a
secrets manager growing inside a configuration library. Provider wiring stays
thin — if you need rotation, wire the tessera to your secrets manager with
`provider(...)`, declare it `freshness: "live"`, and drive re-resolution with
`refresh(...)`. The orchestration stays where it belongs. What the wiring
*does* get is a dedicated provider class with its own guard rails — the next
section.

What v1 spends its budget on instead is the *accidental* disclosure surface:
fail-closed exposure, the resolver-free browser entry, the serialization
guard, redacted diagnostics, and stripped validator output — none of which tax
the read path with ceremony.

## Secret providers

`env` and `file` cover ambient configuration, but secrets increasingly live in
a store with an API — Vault, SSM, Secrets Manager. The moment a tessera is wired
to `provider(...)`, the provider itself joins the disclosure surface: its
errors, its write results, its caching behavior. So provider-sourced secrets
get a dedicated **provider class**, and everything in this section hangs off
that one declaration.

### One contract, one extra class

A payload fork — a separate secret-flavored slot shape alongside the general
one — was explicitly rejected: dual contracts are a drift class, two shapes
that agree today and diverge under maintenance. A secret provider is the
*same* batch contract every provider implements, with a class marker and two
deliberate omissions:

```ts
interface SecretProvider {
  id: string;
  color: "async";
  class: "secret"; // general providers: class?: "config" (the default)
  limits?: {
    maxReadBatch?: number;
    maxWriteBatch?: number;
    maxConcurrentBatches?: number;
  };
  // The same ReadSlot every provider returns — the shared slot type,
  // no secret-specific payload fork.
  read(keys: readonly string[]): Promise<readonly ReadSlot[]>;
  // SecretWriteResult = Omit<WriteResult, "storedRaw"> — derived from the
  // general result type, not hand-written, so the two cannot drift.
  write?(
    entries: ReadonlyArray<{ key: string; raw: Json; ifVersion?: string }>,
  ): Promise<readonly SecretWriteResult[]>;
  capabilities?: {
    versionTokens?: "unique" | "coarse" | "none";
    strongRead?: boolean;
    normalizes?: boolean;
    atomicWrites?: boolean;
    consistentReads?: boolean;
    // no echoWrite — a secret provider cannot even claim to echo writes
  };
}
```

The omissions are the type-level move: the secret write result has no
`storedRaw` field and `capabilities` has no `echoWrite`, so an author cannot
*accidentally* echo a secret — the field to put it in does not exist. But
TypeScript is structural, and an object carrying extra fields still satisfies
the type. So core does not trust the annotation: every slot and every write
result from a secret-serving provider is **runtime-reshaped** through a
whitelist copy. Reads keep `raw`, `version`, and `cacheUntil` — or absence, or
a quarantined error (below) — and nothing else; write results keep `version`
only, and a `storedRaw` the provider returns anyway is dropped on the floor.
The type is the lint; the reshape is the enforcement.

### Pairing runs both directions

Classing would be decoration if it only labeled providers. It is checked at
bind and again at bake, and the pairing is recorded in the lockfile — in both
directions:

- A `class: "secret"` provider may serve **only** secret-exposure tesserae.
  Wiring a public tessera to it fails with `WiringExposureError`: either the tessera
  is misclassified, or the wiring is laundering values out of a secret store
  into public config.
- A provider-sourced **secret** tessera requires a secret-class provider — or a
  general provider that explicitly declares `holdsSecrets: true`. That flag is
  a greppable opt-in for teams whose config database legitimately holds
  secrets, and it lands in its own flagged lockfile section, so the decision
  is visible in review rather than buried in wiring.

`env`- and `file`-sourced secrets are unaffected: the Kubernetes secret
mounted as an env var works exactly as it does today. What changes is the
provider path. Today the tessera's `exposure` flag is the only assertion of
secrecy; with classing, the *store* asserts it too, and disagreement between
the two is an error — defense in depth applied to the classification itself.

Bake participates in the pairing as well: `tessellum bake --check` hard-errors if
a secret-class provider id appears anywhere in the client-reachable wiring
closure, and warns when bake would have to contact a `holdsSecrets` provider
at build time — build-time surfaces are adversary-triggerable, and a build
should not be quietly talking to a store that holds secrets.

### Quarantine, not scrub

The classic mitigation for provider errors is a per-tessera substring scrub: take
the resolved raw value and strip it from any error text before it surfaces.
That scrub has a blind spot no tuning can fix: **when the value never
resolved, there is no needle to search for**. A Vault client that fails to
connect throws something like

```
connect ECONNREFUSED https://user:token@vault.internal:8200/v1/secret/data/app
```

— an error carrying the *provider's own credential*, a value no tessera ever
declared and no tessera-value scrub can find.

So core never propagates error objects or `cause` chains from secret-serving
providers. When a read fails, the per-key `SourceUnavailableError` carries
exactly `{ providerId, key, errorName, messageDigest, safeMessage? }`. The
digest lets you correlate incidents across logs without carrying the message;
`safeMessage` exists only because the secret conformance profile requires
providers to pre-sanitize their error objects — a claim the conformance kit
probes with poisoned fakes rather than takes on faith. For local debugging
there is a `debugUnsafeErrors` escape; it is dev-only and refused in
production binds. The substring scrub is not gone — it remains as
belt-and-braces over raw values that *did* resolve in-process — but it is no
longer the mechanism the model relies on for provider errors.

### `holdsSecrets` inherits the full obligation set

The opt-in buys pairing legality, never reduced hygiene. A general provider
declaring `holdsSecrets: true` is held to everything a secret-class provider
is held to: error quarantine, the runtime reshape, the secret conformance
profile, and echo-verify disabled for the secret tesserae it serves.

Core adds one segregation on top: batches are split by exposure. Secret and
public tesserae never share a wire batch, even on the same provider — chunking is
core's job anyway, so this is free to enforce — which means a batch-level
error on the public side can never enumerate secret rows in its message.

### Write verification without echo

Rotating a secret through `write` poses a puzzle: the standard way to verify a
write is to compare what you stored against what the provider echoes back, and
echoing is exactly what secret providers cannot do. Verification runs a
capability precedence instead, with each rung honestly labeled:

1. **Version tokens** (`versionTokens`). The write returns a version token; if
   a read-back returns the same token, the write is verified with zero raw
   comparison — and the check is immune to stores that normalize values on
   write. If the read-back version is *newer*, the outcome is `"superseded"`,
   a non-throwing outcome: a concurrent rotation is not corruption.
2. **Digest read-back** (`strongRead`). Read the value back and compare
   digests — never raws.
3. **Neither capability.** The outcome is `"inconclusive"` — the write installs,
   and the result says plainly what could not be checked. Where policy demands
   proof, a per-tessera `requireVerifiedWrites` opt-in refuses writes that would
   land inconclusive.

When verification does catch a mismatch, `WriteMangledError` for a secret tessera
carries digests and `nowLiveDigest` — never a raw value.

### Cache ceilings

The shared `ReadSlot` carries `cacheUntil` — deliberately named for what core
does with it, not `expiresAt`, which would claim knowledge of the credential's
lifetime that core does not have. Effective staleness for a provider-sourced
tessera is `min(maxStalenessMs, cacheUntil - now)`: the provider's ceiling can
tighten the tessera's own staleness budget, never extend it.

Three clauses, stated honestly:

- A pinned snapshot keeps serving a value past its ceiling within a request —
  **by design**. Snapshot consistency wins inside a request boundary.
- Ceiling passed and the provider is down: the key fails closed as per-key
  unavailable — **by design**. An expired credential is usually dead anyway;
  serving it is a slower failure, not a kindness.
- **Non-goal:** leased and dynamic credentials — Vault dynamic database
  credentials, STS session tokens — must not be served through this library.
  There is no renewal machinery here; resolution is poll-on-read only. A lease
  needs a lifecycle manager, and that is a secrets manager's job (see
  [Why there are no Sealed boxes](#why-there-are-no-sealed-boxes)).

### Two gates that close side channels

Two rules land early — at descriptor construction and at bind — because both
channels bypass every value-level guard on this page:

- **`literal(...)` on a secret-exposure tessera is `SecretLiteralError`**, at
  bind and at lock. A live credential written into committed wiring JSON
  bypasses every name-based guard — it is a secret sitting in the repository,
  whatever the tessera declares. Tests use `secretLiteral()` from the testing kit
  instead, alongside `fakeSecretProvider()`, a pre-certified in-memory
  double — the same pattern as `manualClock`.
- **`variants` discriminants must be `exposure: "public"`**, enforced when the
  descriptor is constructed. Branch selection is a function of the
  discriminant's *value*, and that value leaks through
  `InactiveVariantError` messages and lockfile surfaces that no error scrub
  ever sees. A secret must not steer branches.

### What classing buys — and what it does not

| Classic provider-author mistake | What classing does about it |
| --- | --- |
| Raw value in error text | Quarantine — the error object never propagates; digest, name, and an optional pre-sanitized message only |
| Echoing a written secret | Type omission (`SecretWriteResult` has no `storedRaw`) plus the runtime reshape — echoed raws are dropped |
| Secret routed to a casual store | Pairing — refused at bind and at bake unless the store opts in with `holdsSecrets: true`. The headline win |
| Absent-vs-empty confusion | Conformance kit — probed for every provider, class-independent |
| Provider-internal caching or logging of raws | **Not addressed.** Kit-probed only; no runtime mechanism sees inside the provider |

Classing buys routing, a quarantine perimeter, and a conformance profile — not
author infallibility.

## Threat model

No single mechanism here is claimed sufficient; they stack. And each guarantee
has a stated boundary.

**What this model defends against — accidental disclosure:**

- A secret compiled into a client bundle: the core entry ships no source
  resolvers and its `bind` supplies no natural defaults, bake never resolves
  secret values, and the allowlist leak scan flags unregistered high-entropy
  strings in every emitted asset (see [bake](/docs/bake)).
- A secret in an SSR/JSON payload: the runtime serialization guard throws.
- A secret in logs and diagnostics: `toJSON()` and `explain()` redact by
  construction.
- A secret echoed by a validator: counts and paths only.
- A provider's own credential in its error text: errors from secret-serving
  providers are quarantined — digests and names, never the error object or
  its `cause` chain.
- Over-broad reads across package boundaries: `pick()` gives least privilege
  and auditability — a dependency sees only the scope you hand it, and
  anything outside it fails with `NotInScopeError`.

**What it does not defend against — and does not pretend to:**

- Hostile code already running in your process. It can call
  `process.env` itself; no configuration library changes that. `pick()` is
  least-privilege for *mistakes*, not a sandbox. The exception worth knowing:
  provider-resolved values (fetched from SSM, Vault, and the like rather than
  ambient env) exist only inside your bindings, so they gain real confinement
  that ambient env values structurally cannot have.
- Strings derived from a secret. Concatenate, slice, or template a secret and
  the guard no longer follows it.
- A provider that caches or logs raw values internally. The conformance kit
  probes for it; no runtime mechanism can see inside the provider.
- Memory inspection, debuggers, and core dumps.
- Exfiltration *after* a legitimate `unwrap()` — once handed to an SDK, the value
  is subject to that SDK's behavior.

## Build-time layers

This page covers the runtime half of the model. The build-time half — the
required targets declaration, the deny-by-default client allowlist, the
secret-name bake gate, generated-module-only delivery, the AST lint banning
raw `process.env` reads in client-designated code, the coverage report, the
entropy leak scan
(including sourcemaps and decompressed assets), and the secret-provider
checks (`bake --check` hard-errors on a secret-class provider id in the
client-reachable wiring closure and warns when bake must contact a
`holdsSecrets` provider) — lives on the [bake](/docs/bake) and
[leak scan](/docs/bake/leak-scan) pages.
