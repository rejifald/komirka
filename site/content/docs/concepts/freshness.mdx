## Pinned by default

Every zerno defaults to `freshness: "snapshot"`. When you bind, the value is resolved and validated once; the resulting `Snapshot` returns the same value for the life of the binding, on every call, from every consumer.

```ts
import { zerno } from "komirka";
import { bind } from "komirka/node";
import { z } from "zod";

const dbUrl = zerno({
  key: "DATABASE_URL",
  schema: z.string().url(),
  doc: "Primary Postgres connection string",
});

const cfg = bind([dbUrl]); // : Snapshot — pinned
cfg.unwrap(dbUrl); // same value on every read, forever
```

This is a stance, not a limitation: **if it can change while the process runs, it's state, not config.** Feature flags, targeting rules, per-user toggles — those belong in a flag system with subscriptions and targeting, which this library deliberately does not have. The one exception is a value backed by a source that is genuinely re-readable, declared explicitly with `freshness: "live"`.

## The physics: env vars can never be live

> **A running process's OS environment never changes.** The env block is copied into the process at spawn; nothing on Node can mutate it from outside afterwards. On Kubernetes, editing a Deployment's env vars rolls new pods — your "updated" env var arrives as a *new process*, never inside the old one. Any library that offers "live env config" on Node is reading a cache of its own making, not the environment.

So `freshness: "live"` on an env source emits a dev-mode warning and behaves as a snapshot. Live is only meaningful for sources that can actually be re-read:

- **Files.** Kubernetes ConfigMaps mounted as volumes update in place — kubelet swaps a `..data` symlink atomically, so a re-read sees the new content. (Caveat: `subPath` mounts never update. If your mount uses `subPath`, live reads will faithfully re-read a file that never changes.)
- **Providers** (SSM, Vault-shaped functions) — re-resolved on demand via `refresh()`.

## Declaring a live zerno

```ts
import { zerno } from "komirka";
import { bind } from "komirka/node";
import { z } from "zod";

const upstreamUrl = zerno({
  key: "UPSTREAM_URL",
  schema: z.string().url(),
  exposure: "public",
  sources: [{ file: "/etc/app-config/UPSTREAM_URL" }],
  freshness: "live",
});

const rateLimit = zerno({
  key: "RATE_LIMIT",
  schema: z.coerce.number().int().positive(),
  exposure: "public",
  sources: [{ file: "/etc/app-config/RATE_LIMIT" }],
  freshness: "live",
});

const live = bind([upstreamUrl, rateLimit]); // : LiveBinding
```

The zernos declare their file refs themselves, so bare entries are all the bind site needs — under `komirka/node`, declared file refs resolve against the filesystem naturally. If any bound zerno is live, the bind result type is a `LiveBinding` instead of a `Snapshot`. This is a hard type split, and it is the point.

## LiveBinding is not a Snapshot — deliberately

`Snapshot` is the only type cross-package APIs accept, and it is always pinned and immutable. `LiveBinding` is not assignable to `Snapshot` or `Scope`, so a library function typed against pinned config can never silently receive a drifting view:

```ts
// A library types its API against pinned, scoped config:
export function createProxy(cfg: Scope<ConfigOf<typeof proxyZernos>>) { /* ... */ }

createProxy(live);                          // ✗ compile error: LiveBinding is not a Scope
createProxy(live.snapshot().pick(proxyZernos)); // ✓ pin first, then narrow
```

The consequence for DI: *the caller decides when time stops.* A dependency receives values from one coherent instant and cannot observe changes mid-call. Anything that wants fresh values must hold the `LiveBinding` itself — visible in its signature, auditable at the composition root.

## How a live read works

`live.unwrap(zerno)` uses **poll-on-read** — there are no watchers, no timers, no background loops in core, ever. The cost of freshness is paid at the read site, and three mechanisms keep it cheap and coherent:

- **Throttle:** raw re-reads are throttled per source by the `maxStalenessMs` bind option (files default to about one second). Reads inside the staleness window return the current value without touching the filesystem.
- **Memoize-by-raw:** the validator only runs when the raw content actually changed. A hot loop calling `live.unwrap()` re-validates zero times while the file is stable.
- **One coherent pass per source:** a single file read serves all zernos backed by that file, so two zernos from the same source never mix generations within one poll.

```ts
const live = bind([upstreamUrl, rateLimit], { maxStalenessMs: 5_000 });
```

## Torn config and the per-request pin

Coherence *within* a poll does not protect two reads separated by time. This handler is torn:

```ts
app.get("/proxy", async (req, res) => {
  const target = live.unwrap(upstreamUrl); // config generation N
  await authenticate(req);              // ...ConfigMap swaps here...
  const limit = live.unwrap(rateLimit);    // generation N+1 — a mismatched pair
});
```

The documented pattern is to **pin once per request** in middleware. `live.snapshot()` returns a real `Snapshot` — one coherent generation for the whole request, and the type every downstream function already accepts:

```ts
app.use((req, _res, next) => {
  req.cfg = live.snapshot(); // pinned: this request sees exactly one generation
  next();
});

app.get("/proxy", (req, res) => {
  const target = req.cfg.unwrap(upstreamUrl);
  const limit = req.cfg.unwrap(rateLimit); // guaranteed same generation as target
});
```

Requests get per-request freshness; nothing inside a request ever tears.

## When the new value is invalid: current truth wins

Someone fat-fingers the ConfigMap and `RATE_LIMIT` becomes `"unlimited"`. The default is **current truth wins**: there are no magically inherited values. A live re-read that yields an invalid value **poisons** that zerno — reads of it fail until the source is fixed. The honest trade, in one sentence: throw-by-default converts a bad config push from silent staleness into a loud *partial* outage that lasts exactly until the push is fixed — opt into `keepLastGood` per zerno where availability beats freshness.

Partial is load-bearing. `live.snapshot()` never throws wholesale: pinning succeeds, and the poisoned zerno occupies an **error slot** in the snapshot. Only reading *that* zerno throws; every other zerno keeps serving:

```ts
const cfg = live.snapshot(); // ✓ pins fine — the poison rides along in an error slot

cfg.unwrap(upstreamUrl); // ✓ unaffected zernos keep working
cfg.unwrap(rateLimit);
// ✗ InvalidLiveValueError: RATE_LIMIT is invalid
//   carries { issues, lastValidAt, invalidSince, incidentId }
//   (and the raw value — for non-secret zernos only)
```

Blast radius is proportional to actual usage: one invalid zerno that nothing reads must never 500 the site.

A revocation list or a kill switch — worse stale than absent — needs no spelling at all anymore: throw-on-invalid is simply the default.

### safe: the non-throwing read

`safe(zerno)` returns `{ ok: true, value }` or `{ ok: false, error }`. It is the first thing epoch-consumer code should reach for, and the right shape anywhere you would otherwise wrap `unwrap` in try/catch:

```ts
const r = live.safe(rateLimit);
if (r.ok) {
  applyLimit(r.value);
} else {
  degrade(r.error); // InvalidLiveValueError — your fallback, chosen explicitly
}
```

### Observation and consumption are different planes

The `onInvalid` bind option is the **observation plane**: it fires once per `(zerno, distinct invalid raw)` — not once per failing read — so a hot loop can't melt your metrics. The throws are the **consumption plane**. Both carry the same `incidentId`, so the hook's log line and a request's stack trace join (and dedupe) on one id:

```ts
const live = bind([upstreamUrl, rateLimit], {
  onInvalid: (k, raw, issues) => {
    metrics.increment("config_invalid", { key: k.key });
  },
});
```

For secret zernos the hook receives `{ rawDigest, length }` instead of the raw — the invalid value never reaches your logging pipeline.

### Invalid is not unavailable

A source that serves garbage and a source that is down are different failures. If the file is unreadable or the provider is unreachable, the binding serves the cached value **within `maxStalenessMs`** — that is not inherited state, it's the staleness budget the binder signed at the bind site. Past the budget, reads throw `SourceUnavailableError` (`{ cause, staleSince }`) and the `onSourceError` hook fires. Unavailability is never routed through `onInvalid`.

### Opting into keep-last-good

Some values genuinely prefer availability over freshness — a rate limit, a retry budget. Keep-last-good still exists, as a **bind-site opt-in** with a mandatory coasting budget:

```ts
import { keepLastGood } from "komirka";

const live = bind([upstreamUrl, rateLimit], {
  invalid: [keepLastGood(rateLimit, { maxCoastMs: 300_000 })],
});
```

`maxCoastMs` is required — there is no unbounded coasting. While the zerno coasts on its last good value, `health()` reports it as `"coasting"`; past the budget, reads throw `CoastBudgetExceededError` and the status flips to `"invalid-was-valid"`. Reach for this exactly where availability beats freshness, and nowhere else.

Why a bind option and not a descriptor field? Identity. On the descriptor's identity class, two teams that disagree about coasting would hold *different zernos* — a descriptor mismatch over an operational preference. On the cosmetic class, real runtime behavior would hide from identity. How *this process* copes with a bad push belongs to this process's bind site.

### Never-valid is its own error

Current-truth-wins semantics apply after a first valid value. With the default eager `bind`, an invalid-at-boot value still dies at bind inside the aggregated `MissingConfigError`. Through the lazy doors, a zerno that has *never* validated in this process throws `NoValidValueError`:

```ts
const lazy = bind.unchecked([rateLimit]);
lazy.unwrap(rateLimit);
// ✗ NoValidValueError: RATE_LIMIT has never produced a valid value in this process
```

### Recovery is one read, not one poll

Poisoned state bypasses the staleness memoization: while a zerno is poisoned, the next read re-reads the source immediately instead of waiting out `maxStalenessMs`. Fix the file or the row, and the very next read recovers. (On Kubernetes the `..data` symlink swap is atomic for single-file, single-pass reads, and multi-file sources are read in one resolved-dir pass — the genuine tear risk is plain files outside k8s, and the poisoned bypass collapses that window too.)

### What health() reports

`live.health()` gives each zerno a status — `"valid"`, `"invalid-was-valid"`, `"never-valid"`, or `"coasting"` — plus `lastValidAt`, `lastInvalidAt`, `lastThrownAt`, and `throwCount`. Source observation and consumer impact stay distinct on purpose: a zerno that went invalid but nothing reads ("invalid but dormant") is a different incident than one actively failing requests, and that split is how operators triage. Two probe rules: **liveness probes must never consult zerno validity** — a bad config push must not get pods killed — and readiness may gate on `"never-valid"` at most.

```ts
live.health();
// per-zerno: status ("valid" | "invalid-was-valid" | "never-valid" | "coasting"),
// lastValidAt, lastInvalidAt, lastThrownAt, throwCount
```

## epoch(): rebuilding what you constructed

Honesty first: **live buys you nothing for construction-time consumption.** If a value is read once to build a long-lived object, the object holds a copy — the binding staying fresh doesn't reach inside it:

```ts
const client = makeUpstreamClient(live.unwrap(upstreamUrl));
// live.unwrap() returned the freshest value *at that instant* — but the client
// captured it. Later file swaps change what live.unwrap() returns, not the client.
```

There is deliberately no `watch()` or `subscribe()` — that's a background loop by another name. Instead, `live.epoch(zerno)` is a monotonic counter that bumps whenever the zerno's observable state changes: a newly accepted value **and** the transition to invalid alike. One clock — consumers observe validity changes through the same counter they already poll, and use `safe` to learn which kind of change fired:

```ts
let epoch = live.epoch(upstreamUrl);
let client = makeUpstreamClient(live.unwrap(upstreamUrl));

function getClient() {
  const now = live.epoch(upstreamUrl);
  if (now !== epoch) {
    epoch = now;
    const r = live.safe(upstreamUrl);
    if (r.ok) {
      client.close();
      client = makeUpstreamClient(r.value); // input actually changed: rebuild
    } else {
      degrade(r.error); // input went invalid: that's a change too — degrade, don't guess
    }
  }
  return client;
}
```

Rebuild-on-epoch is the whole pattern: pools, agents, and SDK clients get recreated exactly when their input actually changed, under your control, on your thread.

## refresh(): a new pinned snapshot on demand

Files re-read themselves on poll; providers don't. `refresh(live)` re-resolves the binding's sources and returns a **new pinned Snapshot**:

```ts
import { refresh } from "komirka";

const next = await refresh(live); // : Snapshot — a fresh, coherent generation
```

Three guarantees:

- **All-or-nothing.** If any source fails mid-refresh, no partial generation escapes — the refresh fails and the binding keeps serving the previous values intact. Opt in to `partial: "keep-old"` to accept updates from the sources that succeeded while keeping old values for the ones that failed: `refresh(live, { partial: "keep-old" })`.
- **Single-flight.** Concurrent `refresh()` calls coalesce into one in-flight resolution in core — a thundering herd of requests triggers one provider round-trip, not hundreds.
- **Pinned result.** The returned `Snapshot` is immutable like any other; hand it to anything typed `Snapshot` or `Scope` without ceremony.

Since core never schedules anything, *you* decide the cadence: refresh on a deploy hook, an admin endpoint, or a cron — all outside the library.

## Workers: env is fixed per deployment

On Cloudflare Workers, the env bag is part of the deployment. It never changes for a running deployment — updating a binding or secret rolls a new deployment, and isolates see the new env from birth. `freshness: "live"` on a Workers env source is therefore a no-op with a dev-mode warning:

```ts
import { bind } from "komirka/workers";

export default {
  async fetch(request: Request, env: Env) {
    const cfg = bind([upstreamUrl, rateLimit], { env });
    // Any freshness:"live" zerno here degrades to snapshot semantics
    // (dev-mode warning): workerd env is immutable per deployment.
    return handle(request, cfg);
  },
};
```

## Freshness by source, at a glance

| Source | Can be live? | Why / how |
| --- | --- | --- |
| `env` on Node | No — physics | OS env of a running process never changes; k8s env edits restart the pod. Dev warning, snapshot semantics. |
| `env` on Workers | No — platform | Env bag is fixed per deployment. Dev warning, snapshot semantics. |
| `file` | Yes — poll-on-read | k8s ConfigMap volume mounts update via `..data` symlink swap; `subPath` mounts never update. |
| `provider` | Yes — on demand | Re-resolved by `refresh(live)`; core adds single-flight and all-or-nothing. |

If nothing on this page applied to your value, that's the healthy case: leave `freshness` at its default, bind once, and let the `Snapshot` be boring.
