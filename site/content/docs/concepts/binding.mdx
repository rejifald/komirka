## Values exist only inside bindings

A `Tessera<T>` is an inert descriptor. Importing it costs nothing, reads nothing, validates nothing. The value comes into existence in exactly one place: a **binding**, where you hand a set of tesserae and a set of source implementations to `bind()`.

```ts
import { z } from "zod";
import { tessera, defineTesserae } from "tessellum";
import { bind } from "tessellum/node";

export const dbUrl = tessera({
  key: "DATABASE_URL",
  schema: z.string().url(),
  doc: "Primary Postgres connection string",
  // exposure defaults to "secret" — fail closed
});

export const poolSize = tessera({
  key: "DB_POOL_SIZE",
  schema: z.coerce.number().int().min(1),
  exposure: "public",
  default: 10,
});

export const dbTesserae = defineTesserae([dbUrl, poolSize]);

// the composition root — the only place sources are actually read
const cfg = bind(dbTesserae); // bare entries resolve their declared refs against
                           // this entry point's natural defaults: process.env + fs

cfg.unwrap(dbUrl);    // string — sync, typed from the schema's output
cfg.unwrap(poolSize); // number
```

There is no ambient binding to fall back to: no module-scope cache, no global registry, no value stashed on the tessera object itself. Asking a snapshot for a tessera it never bound is a named error, never a silent default:

```ts
const cfg = bind([dbUrl]);

cfg.unwrap(poolSize);
// UnboundTesseraError: DB_POOL_SIZE is not in the set bound by this snapshot.
//   Bound here: DATABASE_URL
//   Fix: add DB_POOL_SIZE to the bind() call that created this snapshot.
```

`unwrap()` has a three-way contract: a hit, `UnboundTesseraError`, or `DescriptorMismatchError` when the same key was bound under a *different* descriptor. The mismatch error carries both canonical descriptors, a field-level diff, and a paste-able `unify` snippet — see the identity page for that story.

## Call shapes: the snapshot mirrors what you bind

`bind()` accepts a single entry, an array, or a record — and the snapshot grows a value
view matching the shape you handed it. This is the answer to "what *is* `cfg`?": it is a
`Snapshot` (pinned, immutable, carrying the full `get`/`pick`/`explain` API), and its
`.value`/`.values` view mirrors your input.

```ts
// SINGLE — no brackets anywhere
const cfg = bind(dbUrl);
cfg.value;                       // : string — the one value

// ARRAY — .values is a real frozen tuple, positionally typed
const cfg = bind([dbUrl, poolSize]);
const [url, size] = cfg.values;  // : [string, number]

// RECORD — .values is a typed record under YOUR names (recommended at composition roots)
const cfg = bind({ db: dbUrl, pool: poolSize, email: emailVariants });
cfg.values.db;                   // : string
cfg.values.email;                // : the variants discriminated union — same as cfg.match(emailVariants)
const { db, pool } = cfg.values;
```

Rules that keep the sugar honest:

- **Every form returns a full `Snapshot`** — `.value`/`.values` are additive views, not a
  different kind of object. The handle API (`unwrap`, `safe`, `pick`, `explain`, `match`)
  is always there.
- **The snapshot is deliberately *not* iterable.** TypeScript cannot type positional
  destructuring from an iterable (every slot would be the union of all value types), and
  faking it with a tuple intersection would make `cfg.map(...)` typecheck and crash. The
  tuple lives at `cfg.values`, where the types are exact.
- **Lazy door:** touching a `.values` property *is* the first read — and JS destructuring
  only reads the properties you list, so `const { db } = cfg.values` validates only `db`.
  Prefer the record form when any slot may be poisoned: array destructuring cannot skip a
  throwing position.
- **Live and async bindings have no `.value`/`.values`** — destructuring would silently
  freeze liveness. Pin first: `live.snapshot().values`. This is enforced in the type
  (freshness is a type parameter; a type-widened `Tessera<string>` fails *safe*, losing the
  sugar rather than gaining a frozen value) and backstopped at runtime
  (`LivenessViolationError`).
- **The values bag is guarded.** It is a null-prototype object whose `toJSON` applies the
  same secret redaction as `Snapshot.toJSON`, with a redacting Node inspect hook — so
  `JSON.stringify(cfg.values)` and `console.log(cfg.values)` never dump secrets. Spreading
  it (`{ ...cfg.values }`) copies raw values into an unguarded plain object: treat spread
  as calling `unwrap()` on every tessera. Record keys `toJSON`, `then`, and `constructor` are
  rejected at bind (`ReservedKeyError`).
- `.values` is a **read view at the composition root** — the handoff type for libraries
  remains a `pick()`ed `Scope` (which carries the same `.values` view, so passing the
  branded scope is never worse ergonomics than passing naked strings).

## Eager by default: one aggregated failure

`bind()` validates **exactly the listed set, now** — not "everything the app ever declared". Groups are declared near their call sites, so what you bind is what that composition root actually needs. If anything is missing or invalid, you get one aggregated `MissingConfigError` listing every problem, not a whack-a-mole of one-failure-per-restart:

```ts
const cfg = bind([dbUrl, poolSize, smtpPassword]);
```

```text
MissingConfigError: 3 of 3 tesserae failed to bind

  ✖ DATABASE_URL    missing — no source produced a value
                    tried: env DATABASE_URL
  ✖ DB_POOL_SIZE    invalid — expected number, received "many"
                    from: env DB_POOL_SIZE
  ✖ SMTP_PASSWORD   invalid — 1 issue at (root)
                    from: env SMTP_PASSWORD
                    (message withheld: exposure is "secret")
```

Two details worth noticing. For secret tesserae the report shows issue counts and paths only — validator messages echo received values, so they are treated as untrusted and never printed. And `default` gets no privileged path: a defaulted value is validated at bind exactly like a sourced one, so a bad default dies at boot, not in production.

## The three doors

| Door | At bind time | At first read |
| --- | --- | --- |
| `bind(entries)` | presence **and** validation | nothing left to fail |
| `bind.lazy(entries)` | presence only | validation |
| `bind.unchecked(entries)` | nothing | presence and validation |

`bind.lazy` is the half-lazy door: raw presence is still checked eagerly, so a missing variable dies at bind — the classic 3am failure, where an unset var is only discovered hours later on the first cold code path, is structurally gone. What it defers is validator execution, for when validators are expensive or the set is large.

```ts
const cfg = bind.lazy(serverTesserae);
// missing env vars have already thrown MissingConfigError here;
// schemas run on first cfg.unwrap(...) per tessera
```

`bind.unchecked` checks nothing until first read. The name carries the warning; reach for it only when eager checks are genuinely wrong, like a multi-subcommand CLI where `--help` must not demand deploy credentials:

```ts
const cfg = bind.unchecked(deployTesserae);

if (command === "deploy") {
  cfg.unwrap(deployToken); // presence + validation happen here, per tessera
}
```

## Async entries color the binding — and the type says so

There is no separate async door: the bind result **type is computed from the entries**.
Any async-wired entry (a `provider()` whose impl declares `color: "async"`, or a `chain()`
containing one async link) makes the result an `AsyncBinding`; any live tessera makes it a
`LiveBinding`; otherwise you get a plain `Snapshot`. Sugar views (`.value`/`.values`)
exist only on snapshots, so an async entry can never hide a network hop behind a sync
read — the type will not let you.

```ts
import { chain, provider } from "tessellum";
import { bind, env } from "tessellum/async";     // async-capable bind; node combinators re-exported
import { ssm } from "./providers/ssm";       // impl: { id: "aws-ssm", color: "async", read }

const binding = bind({
  db: dbUrl,                                  // bare — process.env
  sg: chain(sgKey, provider(ssm, "/prod/sg"), env("SG_KEY")), // async link → AsyncBinding
});

const cfg = await binding.snapshot();         // resolve once — an async pin
cfg.unwrap(sgKey);                               // string — sync forever after
```

`await binding.snapshot()` is the old "resolve once, read sync forever" story; a
long-lived `AsyncBinding` uses `await binding.ready()` as the eager startup line instead.
If the store is unreachable at boot, `ready()` has no cache to fall back on, so it retries
with backoff and then fails fast past `readyTimeoutMs` ([ADR 0005](https://github.com/rejifald/tessellum/blob/main/docs/adr/0005-async-read-deadline-and-boot-outage.md)).
Provider impls must declare `{ id, color }` — the `id` is what the lockfile records, and
an impl with no declared color is a compile error, never a silently-sync guess.

The same shape of rule applies to validators: async-ness can be input-dependent, so every validation site checks for a thenable and throws `AsyncValidatorInSyncBindingError` if a schema goes async inside a sync binding.

## The Snapshot API

A `Snapshot` is always pinned and immutable — it is the only type cross-package APIs accept. Validated outputs are contractually immutable too (deep-frozen in dev mode), so nothing downstream can mutate a config value another module already read.

```ts
cfg.unwrap(dbUrl);    // string — throws the named errors above on failure
cfg.safe(dbUrl); // { ok: true, value } | { ok: false, error } — non-throwing read
cfg.explain(dbUrl); // provenance: which source won, which key resolved
cfg.toJSON();       // full dump — secrets redacted by construction
```

`explain()` answers "where did this value actually come from" without exposing it:

```json
{
  "tessera": "DATABASE_URL",
  "identity": "tesserav1:xxh128:5f21c09a…",
  "resolved": { "source": "env", "key": "DATABASE_URL" },
  "value": "[redacted: exposure is secret]"
}
```

For a provider-sourced value the record also carries **`version`** — the store's CAS token the
pin last saw (a SQL row version, an etcd revision) — which the concurrent-writer retry loop
reads back as `explain(k).version`. It is `undefined` for env, file, and defaulted values, and,
being metadata rather than the value, it is present even when the value itself is a redacted
secret.

`toJSON()` is safe to log because redaction is structural, not best-effort. And the guard goes one level deeper: `unwrap()` of a secret returns a `Secret<T>` (assignable to `T`, so no ceremony) that itself refuses serialization, so an SSR payload cannot smuggle one out by accident:

```ts
const page = { props: { db: cfg.unwrap(dbUrl) } };
JSON.stringify(page);
// ExposureError: secret value DATABASE_URL cannot be serialized
// (structuredClone/postMessage throw DataCloneError too; call .reveal() when you
//  genuinely need the raw string — see the Secrets page and ADR 0002)
```

For handing a *subset* of a snapshot to a library, `cfg.pick(dbTesserae)` returns a runtime-restricted `Scope` — reading a non-member throws `NotInScopeError`, and the root snapshot is deliberately not assignable where a `Scope` is expected. The library DI page covers this pattern in full.

## Config flows at construction time, not call time

The intended shape: one composition root binds, factories receive their slice at construction, and request handlers never touch config at all.

```ts
import type { Scope, ConfigOf } from "tessellum";

// a factory declares the exact slice it needs, in its type
export function createPool(cfg: Scope<ConfigOf<typeof dbTesserae>>) {
  return new Pool({ url: cfg.unwrap(dbUrl), max: cfg.unwrap(poolSize) });
}
```

```ts
// app entrypoint — the composition root
const root = bind([...dbTesserae, ...mailerTesserae]);

const pool = createPool(root.pick(dbTesserae));
const mailer = createMailer(root.pick(mailerTesserae));
// handlers receive pool and mailer — not root, not process.env
```

This is also what makes testing boring, in the good way — construct a literal snapshot per test and pass it in, with zero `process.env` mutation:

```ts
import { bind, literal } from "tessellum"; // core bind: literal entries only — nothing ambient
import { secretLiteral } from "tessellum/testing"; // secret-safe literal(), test-scoped

const cfg = bind([secretLiteral(dbUrl, "pg://localhost/test"), literal(poolSize, 2)]);
const pool = createPool(cfg.pick(dbTesserae)); // still validated — a bad literal fails here
```

## Why there is no ambient binding

Import-time and module-global config assumes a process-global environment that exists once and never changes. Cloudflare Workers break that assumption structurally: `env` arrives as a *parameter*, per invocation, and an import-time validator can never see it. Worse, a module-scope cache in that world is a cross-tenant bleed — the first request's env populates the cache, and every later request in that isolate reads the first request's values.

Explicit binding matches the platform's physics instead:

```ts
import { bind } from "tessellum/workers";
import { appTesserae } from "./config";

export default {
  fetch(request: Request, env: Env) {
    // bound per invocation — this request's env, no one else's.
    // The bag is a REQUIRED option: capability enters at bind, never inside wiring data.
    const cfg = bind(appTesserae, { env });
    return handle(request, cfg);
  },
};
```

Because there is no ambient store, the failure mode of a mis-wired read is a thrown `UnboundTesseraError` at the offending line — never a stale or cross-request value.

## Source refs are data; implementations are injected

A tessera's `sources` field is ordered *data* — plain JSON refs naming what to read, in precedence order. It never contains an implementation, which is why descriptors round-trip as JSON and why tools like bake can enumerate them without executing app code:

```ts
const sendgridKey = tessera({
  key: "SENDGRID_API_KEY",
  schema: z.string().min(1),
  sources: [
    { provider: "ssm", key: "/prod/sendgrid/api-key" }, // preferred
    { env: "SENDGRID_API_KEY" },                        // local-dev fallback
  ],
});
```

The *how* is supplied at the binding, per runtime, and the descriptor never changes:

```ts
// Node server: wired via provider() — declared refs resolve in order (ssm, then process.env)
import { bind, provider } from "tessellum/async";
const cfg = await bind(provider(sendgridKey, ssm)).snapshot();

// Cloudflare Worker: the per-request bag is a REQUIRED bind option — never wiring data
import { bind } from "tessellum/workers";
const cfg = bind(sendgridKey, { env });

// tests: a secret literal entry — validated like any source, zero process.env mutation
import { bind } from "tessellum";
import { secretLiteral } from "tessellum/testing";
const cfg = bind(secretLiteral(sendgridKey, "SG.test-key"));
```

This split is also the browser-safety story, stated precisely: the universal `tessellum`
entry ships no source *resolvers*, and core `bind` supplies no natural defaults — a bare
`env`/`file` ref under core bind is `NoWiringError`, and `literal()` entries are the only
thing that resolves. `env` and `file` live in `tessellum/node`, which imports `node:process`
and `node:fs`, so bundling them for a browser fails at build rather than shimming to
empty objects. Capability enters at the bind entry point — never through wiring data.

## Wiring: deployment wiring, not contract

A tessera declares *what* it needs (`{ env: "SMTP_URL" }`); a deployment decides *where that
lives on this platform*. The wiring is spelled inline, per entry — no source bag, no
separate rebind option — and it never forks the tessera's identity, because wiring is
deployment reality, not a contract change:

```ts
import { bind, env, file } from "tessellum/node";
import { chain, provider } from "tessellum";
import { smtpUrl } from "@acme/mailer/tesserae";

const cfg = bind({
  // bare: declared refs against this runtime's natural defaults
  pool: poolSize,

  // REF RENAME: patches the matching declared ref IN PLACE — the tessera's other
  // declared fallbacks survive. UnknownWiringError if no declared ref of this kind.
  smtp: env(smtpUrl, "MAILER_SMTP_URL"),
  cache: file(cacheRules, "/etc/app/cache-rules.json"),

  // FULL REPLACEMENT — chain() is the only spelling that discards declared refs,
  // and errors for this tessera list exactly what was dropped.
  sg: chain(sgKey, provider(ssm, "/prod/sg"), env("SG_KEY")),

  // groups wire as ONE entry; specialization goes inside (overrides escape the prefix)
  email: env(emailVariants, { prefix: "MAILER_", overrides: { password: "SECRET_PW" } }),
});
```

The distinction between the two override forms is load-bearing: `env(tessera, "NAME")`
*renames*, preserving the resilience chain the tessera's author declared — a deployment that
only meant to rename must not silently lose the file fallback it will need during an
outage. `chain()` *replaces*, and says so. Read provenance makes the difference
observable: every binding records which ref supplied each value, and `explain()`
and every resolution error name it.

Wiring rules, checked in the eager phase of **every** door (wiring is shape, not value
validation):

- **One entry per tessera, ever.** A tessera reachable through two entries — including a group
  member wired again as a leaf — is `DuplicateWiringError`. There is no precedence
  hierarchy to memorize; specialization lives inside the group combinator.
- **`WiringCollisionError`** — wiring *created* a convergence (two tesserae onto one
  effective name) that the declared refs did not have. Declared sharing stays legal.
- **`WiringExposureError`** — a non-secret tessera may never be wired onto any name in the
  secret-name set: routing a credential through a public-exposure tessera would launder it
  past redaction and into bake's public inlining. Backed by a well-known-secret-suffix
  denylist (`*_SECRET`, `*_TOKEN`, `*_PASSWORD`, `*_PRIVATE_KEY`, `*_API_KEY`) with an
  explicit per-name allowlist, and by the committed lockfile as the app-wide
  effective-name → exposure authority. The same denylist gates the other end of the
  pipeline: bake refuses a client-target tessera whose declared key or effective source name
  hits the set even at `exposure: "public"` (`SecretNameInClientTargetError`) — see
  [bake](/docs/bake/overview).
- **Combinators are pure data.** A `WiredEntry` is JSON-representable — tessera identity ref,
  ref kind, name/path/key strings, provider impl *ids* (impls declare `{ id, color }`;
  anonymous impls are an error on bake-covered targets). That inertness is what lets the
  bake manifest import entry modules and generate the lockfile, keeps every wiring
  attestable, and is why a per-request Workers env bag can never live inside wiring — the
  bag is a required *bind option* (`bind(entries, { env })`), and capability only ever
  enters at the bind entry point.

App-wide transforms exist, but only as a *restatement* of data the manifest declares for
the target — `bind(entries, { transform: { prefix: "MYAPP_", rename: { ... } } })` — and
the wiring digest covers them: a composition root whose wiring diverges from the
committed lockfile fails closed on client-covered targets (`WiringSkewError`). That check
is **always on** for client-covered targets and cannot be disabled — it keys off bake
coverage, never `NODE_ENV`. Server targets **opt in** with `bind(entries, { attest: true })`,
and a bake-covered server bound without it gets a dev-mode warning; the honest consequence is
that manifest-invisible wiring on a server is possible by default, which is what
`tessellum bake --check` in CI exists to catch
([ADR 0009](https://github.com/rejifald/tessellum/blob/main/docs/adr/0009-attestation-strictness.md)).
The precedence law fits in one line: per-entry and group overrides
are absolute; `rename[declared]` is terminal (it escapes prefixes); prefixes apply to
everything else.

Effective names propagate everywhere: `.env.example` and `CONFIG.md` render the names
your deployment actually reads (declared name in a comment), the lockfile records
`declared → effective` per tessera, the leak scan matches the **union** of both, and every
error speaks the effective name first —
`Missing MAILER_SMTP_URL (declared SMTP_URL; wired at src/config/root.ts:12)` — because
an error that tells a 3am operator to set a variable nothing reads is worse than no
error.

One doctrine note survives unchanged: wiring is for *deployments*. A **library** renaming
its own env var across a major does it in the contract — a declared fallback chain
`sources: [{ env: "MAILER_URL" }, { env: "SMTP_URL", deprecated: true }]` — one
intentional identity change, visible in every consumer's lockfile diff.
