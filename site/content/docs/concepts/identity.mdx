## Why identity exists

There is no runtime registry — not as a limitation, but as a rule of the design. A tessera is a frozen object that a package exports; nothing anywhere keeps a global list of them. That means the same *logical* config value routinely exists as more than one *physical* object in a real dependency tree:

- **Dual-package hazard.** A package published as both CJS and ESM can be loaded twice in one process. Each copy constructs its own descriptor object from the same source code.
- **Diamond installs.** Your app depends on `@acme/mailer@2`, and a transitive dependency still bundles `@acme/mailer@1`. Two versions of `smtpUrl` now coexist in `node_modules`.
- **Cross-package DI.** A library ships its tesserae as inert descriptors and the app binds them at its composition root. The object the library reads with and the object the app bound are frequently not reference-equal.

So `cfg.unwrap(tessera)` cannot key on object reference, and it cannot key on `key` alone — two packages may legitimately use the same key with different contracts. It keys on **content-addressed identity**: the descriptor *is* the identity.

## What counts as identity

Every field of a descriptor is in exactly one of two buckets.

| Field | Bucket | Why |
| --- | --- | --- |
| `key` | identity | The canonical name of the value. |
| `exposure` | identity | A secret/public flip must never unify silently. |
| `sources` | identity | Declared precedence is order-significant; reordering changes meaning. |
| `default` | identity | Presence **and** value both count — adding a default changes behavior. |
| `schemaId` | identity, when available | A vendor fingerprint of the schema, or your explicit `version` discriminator. |
| `freshness` | identity | Live-vs-snapshot is behaviorally load-bearing: relations reference discriminants by identity, and a `live` twin of a `snapshot` tessera must never be identity-equal. |
| `doc` | cosmetic | See below. |
| `examples`, `deprecation` | cosmetic | Documentation, not contract. |
| any unknown field | cosmetic | Forward-compatible: future metadata never breaks old identities. |

`doc` is excluded deliberately. A docs-only patch release — a typo fix, a changelog-driven wording change — must produce a descriptor that still binds against the previous version's. If prose were identity, every documentation edit would make two package versions unbindable against each other, which is exactly the failure mode identity exists to prevent.

Identity is computed over the *effective* descriptor: omitted fields normalize first (`exposure` defaults to `"secret"`, `sources` defaults to the single declared env ref named by `key`), so spelling a default out explicitly does not change the hash.

One thing that is *never* in the table: deployment wiring. The per-entry combinators you apply at `bind` — a ref rename like `env(smtpUrl, "CUSTOM_SMTP_URL")`, a full `chain(...)` replacement, a group prefix — change which effective name supplies a value, and they are covered by the wiring digest and the lockfile's declared→effective map, not by the identity hash. Wiring never forks identity: the same tessera wired three different ways in three services is still, everywhere, the same tessera. Only *declared* refs — the fallback chain the library author ships on the descriptor itself — count as identity.

The `schema` object itself is not JSON and does not round-trip; `schemaId` stands in for it. When the validator vendor provides a fingerprint, that is used. Otherwise your explicit `version` string is the discriminator — which is why the compatibility contract at the bottom of this page tells you to bump it on every schema change.

## The hash: tesserav1, RFC 8785, xxh128

The identity subset is serialized as RFC 8785 (JCS) canonical JSON, prefixed with the literal string `tesserav1|`, and hashed with xxh128:

```ts
import { tessera, identityOf } from "tessellum";
import { z } from "zod";

export const smtpUrl = tessera({
  key: "SMTP_URL",
  schema: z.string().url(),
  doc: "SMTP relay endpoint",
  version: "2",
});

identityOf(smtpUrl);
// "tesserav1:xxh128:9f2c77aa…"
```

**The hash is an index, not a security boundary.** Bindings use it for fast lookup, but whenever a read resolves through anything other than the exact bound object, structural equality of the identity subset is what decides — an accidental (or crafted) xxh128 collision cannot forge a match, because the canonical descriptors are compared field by field. Structural equality is authoritative; the hash is a shortcut to it.

The same philosophy applies to detecting tesserae at all: a descriptor is recognized by its structural `$tessera` brand, never by `instanceof`. `instanceof` fails in precisely the situations identity exists to solve — two module copies mean two class realms, and a JSON-round-tripped descriptor was never constructed by any class in the first place.

## The three-way unwrap() contract

Every read resolves to exactly one of three outcomes:

1. **Hit** — the descriptor (by reference, or by identity) is bound; you get the validated value.
2. **`UnboundTesseraError`** — no tessera with this key is bound at all. There is no ambient fallback; a read outside a binding is a named error, never a guess.
3. **`DescriptorMismatchError`** — the key *is* bound, but under a **different** descriptor.

The third case is the diamond install making contact:

```ts
import { bind } from "tessellum/node";
import { smtpUrl } from "@acme/mailer/tesserae"; // v2: schema tightened, version: "2"

const cfg = bind([smtpUrl]); // bare entry: declared refs resolve against process.env

// Meanwhile, a transitive dependency still bundles @acme/mailer@1,
// whose copy of the descriptor carries version: "1". When that copy reads:
cfg.unwrap(smtpUrlV1);
// throws DescriptorMismatchError
```

The error carries both canonical descriptors, a field-level diff, and a paste-able resolution:

```txt
DescriptorMismatchError: key "SMTP_URL" is bound, but under a different descriptor.

  requested   tesserav1:xxh128:41c09be2…
    { "key": "SMTP_URL", "exposure": "secret",
      "sources": [{ "env": "SMTP_URL" }], "schemaId": "1" }

  bound       tesserav1:xxh128:9f2c77aa…
    { "key": "SMTP_URL", "exposure": "secret",
      "sources": [{ "env": "SMTP_URL" }], "schemaId": "2" }

  diff
    schemaId   "1" -> "2"

  If these contracts are compatible, accept one descriptor and alias the
  other identity at your composition root:

    bind(entries, {
      unify: [{ accept: smtpUrl, aliases: ["tesserav1:xxh128:41c09be2…"] }],
    })
```

Descriptors contain no resolved values, so this error is safe to log even for secret tesserae — it prints contracts, never configuration.

## Resolving deliberately: unify at bind

When you have inspected the diff and decided the two contracts are compatible, you resolve the collision where all binding decisions live — the composition root:

```ts
const cfg = bind([smtpUrl], {
  unify: [
    // Reads made through the old v1 identity resolve to the v2 binding.
    { accept: smtpUrl, aliases: ["tesserav1:xxh128:41c09be2…"] },
  ],
});
```

`accept` names the descriptor you consider authoritative; `aliases` lists identity hashes that should resolve to it. This is an explicit, app-owned decision — because there is no global registry and no `provide()`, a dependency cannot redirect this from import time. Only the code that calls `bind` can.

## Validation never silently downgrades

Hash-fallback resolutions — the dual-package case, where two *identical-identity* descriptor instances exist — always **re-validate with the caller's schema** before returning:

```ts
// copyA and copyB are different objects from two module instances,
// but identityOf(copyA) === identityOf(copyB).
const cfg = bind([smtpUrlCopyA]);

cfg.unwrap(smtpUrlCopyB);
// hash-fallback hit: the raw value is re-validated with copyB's own schema
// before it is returned (memoized per schema — it runs once, not per read).
```

The same rule holds for reads through a `unify` alias: the aliased caller's value is validated by the aliased caller's schema. The point is invariant either way — a package that typed its API against its schema's output gets exactly that guarantee, no matter which physical copy of the descriptor happened to be bound first. Validation can never silently downgrade across package copies.

## Branded-type identity across installs

Everything above is the *runtime* story, and it holds no matter how many copies of tessellum are installed. The type system has a companion guarantee, and it is load-bearing for library authors who type their API `Scope<ConfigOf<typeof tesserae>>`: does that branded `Scope` still line up when a consumer's dependency graph hoists two copies of tessellum?

It does — but only because the `Scope` brand is **structural** (a well-known string key), never a `unique symbol`. A `unique symbol` brand is a *distinct type in every install*, so a `Scope` produced by one copy's `pick()` would not be assignable to a parameter typed against another copy's `Scope` — branded identity would silently evaporate in exactly the diamond-install case identity exists to survive. A string-keyed brand is the same type across copies, so cross-install assignment type-checks, while the root `Snapshot` — which lacks the brand key — stays un-assignable to `Scope` (the P12 least-privilege gate). It also governs the client path: a baked client shim is typed `Scope` through an erased `import type`, so a library's `Scope<...>` signature accepts that shim's scope even when the library carries its own copy of tessellum — the "accepts it unchanged" bake promise holds across installs only because the brand is structural.

The trade is honest: a structural brand is forgeable by writing the key by hand. That is acceptable because the brand is mistake-prevention, not a sandbox — its one job is to stop a caller *accidentally* handing you their whole configuration, and a `Snapshot` can never be passed where a `Scope` is required. The load-bearing cross-install guarantee stays the runtime content-addressed identity above; the type brand is a compile-time convenience that composes across installs *because* it is structural. This was settled, with a `tsc` spike proving the nominal brand breaks and the structural one holds, in [ADR 0007](https://github.com/rejifald/tessellum/blob/main/docs/adr/0007-cross-install-identity-and-library-consumption.md).

## Install once: the peer-dependency

Because duplication is a type-level and bundle-level cost as well as a runtime one, a library that ships tesserae declares `tessellum` a `peerDependency`, not a plain `dependency`:

```jsonc
{ "peerDependencies": { "tessellum": "^1.0.0" }, "devDependencies": { "tessellum": "^1.0.0" } }
```

A peer-dep lets the consumer's package manager dedupe to a single copy and keeps a second copy out of their bundle. It is the right default — but it is *ergonomics, not a guarantee*: peer-deps do not prevent duplication (pnpm's isolated store, incompatible version ranges, CJS/ESM dual-load, and monorepos all still produce copies). When a copy slips through, correctness is held by everything above — content-addressed identity, caller-schema revalidation, and `DescriptorMismatchError` on real drift — not by the manifest. Declare the peer-dep to make duplication *rare*; rely on identity to make it *safe*. The [library-authors guide](/docs/guides/library-authors) has the full checklist, and an escape for consumers who don't use tessellum at all.

## Collisions that never happen — and the ones that warn

Two packages using the same key are not automatically in conflict. `@acme/http` and `@acme/queue` can both ship a tessera named `TIMEOUT_MS`; bound in separate bindings, they never interact, because there is nothing global for them to collide in:

```ts
const httpCfg = bind(httpTesserae);
const queueCfg = bind(queueTesserae);
// No conflict. No registry, no ambient state, no meeting point.
```

If both land in **one** binding, each stays independently readable — `unwrap()` is identity-addressed, so each descriptor resolves its own contract — but dev mode warns, because two contracts over one raw key is usually version drift you have not noticed yet:

```ts
const cfg = bind([...httpTesserae, ...queueTesserae]);
// [tessellum] dev warning: duplicate key "TIMEOUT_MS" bound under two identities
//   tesserav1:xxh128:0d11… and tesserav1:xxh128:77aa…
```

Note the distinction: two *different* identities sharing a raw key is a warning, because it is legal and occasionally intentional. The *same* tessera reaching one `bind` through two entries — listed twice, or once bare and once inside a wired group — is a hard `DuplicateWiringError`, because one tessera must have exactly one wiring.

## The compatibility contract for library authors

If you ship tesserae, your descriptor's identity fields are API surface. This table is the changelog rule of thumb:

| Edit to a published tessera | Identity | Consequence |
| --- | --- | --- |
| Improve `doc`, `examples`, deprecation notes | preserved | Patch-safe; old and new versions still unify. |
| Add custom metadata fields | preserved | Unknown fields are cosmetic by rule. |
| Rename `key` | **broken** | It is a different tessera now; document the migration. |
| Change `exposure` | **broken** | Deliberate — a secret/public flip must force a decision downstream. |
| Add, remove, or reorder `sources` | **broken** | Declared precedence order is meaning. (This is the declared fallback chain on the descriptor — deployment wiring at `bind` is not this, and never breaks identity.) |
| Add or remove `default`, or change its value | **broken** | Presence and value both count. |
| Change `freshness` | **broken** | A live tessera and a snapshot tessera are different contracts — one can be a variants discriminant, the other cannot. |
| Bump `version` | **broken** | That is its job: bump it on every schema change. |
| Change the schema *without* bumping `version` (no vendor fingerprint) | preserved | Silent-drift hazard — identity cannot see the schema itself. This is why `version` exists, and why caller-schema re-validation is the runtime backstop. |

An identity break is not a bug; it is `DescriptorMismatchError` doing its job — surfacing, at the composition root, that two package versions disagree about a contract. The app either upgrades the lagging dependency or unifies explicitly, with the diff in front of it. What identity refuses to do is let that disagreement resolve itself quietly.
